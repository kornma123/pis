# COREONE Issue 分级与上线阻断标签规则

## 1. 目的与适用范围

本规则用于 COREONE GitHub Issue 的统一分级。每张开放 Issue 必须同时标记两个相互独立的维度：

1. 一个优先级标签：`P0`、`P1`、`P2` 或 `P3`；
2. 一个上线影响标签：`阻断上线` 或 `非阻断上线`。

默认“上线”指首次正式生产或外部试用。若某一阶段另有经过 PM 确认的 release target，以该目标为准，并在对应 Issue 中写明。

优先级表示问题本身的严重程度；上线标签表示该问题是否必须在目标上线前闭合。两者不得混为一个维度。

## 2. 优先级定义

| 标签 | 定义 | 典型情形 |
|---|---|---|
| `P0` | 已证实的最高严重度问题，必须立即进入主线处置 | 系统持续无法启动；关键账本、金额、库存或权限事实可被确定性破坏；重大安全边界失守；没有安全绕行方案 |
| `P1` | 核心业务、安全、数据完整性或关键流程缺陷 | 关账、对账、采购、库存、成本、权限、审计等核心事实可能错误；关键验收门缺失；需要优先修复但尚未达到 P0 |
| `P2` | 重要但影响有界的能力、证据或体验缺口 | 有安全降级或明确人工兜底；定向提示、证据机器化、后续能力、非关键流程准确性问题 |
| `P3` | 延期功能、清理或非紧急治理事项 | 已退役合同清理、未来 wave、当前发布范围外的 PRD 债或低风险维护项 |

约束：

- 不得为了“看起来重视”而把未验证风险升级为 `P0`。
- `unknown`、环境阻塞或缺少运行证据不得自动降级为低优先级；应先保留真实的 `BLOCKED`/`NO_GO` 状态并补证据。
- 优先级依据可复现影响、触发条件和业务后果，不依据实现工作量或排队时长。

## 3. 上线影响定义

### `阻断上线`

存在以下任一条件时使用：

- 未闭合时可能发布错误金额、库存、权限、审计或关账事实；
- 属于首次外部试用明确要求的安全硬门；
- 会导致目标环境无法启动、升级或稳定运行；
- 是目标发布所需的真实证据、财务认账、独立验收或回归门；
- 没有经过批准、可审计且不会扩大风险的临时降级方案。

### `非阻断上线`

仅在满足以下条件时使用：

- 不影响当前目标发布的核心业务正确性、安全性和数据完整性；
- 有明确、安全且不会把 unknown 伪装成 success/zero 的降级行为；
- 属于后续迭代、延期 PRD、低风险治理、已合并后的残余体验或清理事项。

`非阻断上线` 不等于“不做”，也不等于 Issue 已完成。

## 4. 双轴组合示例

- `P0 + 阻断上线`：确定性系统不可启动或关键账本可被破坏。
- `P1 + 阻断上线`：核心业务真值、安全硬门或关键回归门未闭合。
- `P1 + 非阻断上线`：需要尽快修复的活跃治理失真，但不影响当前运行时能力。
- `P2 + 阻断上线`：问题本身影响有界，但它是当前发布必须取得的外部证据或验收条件。
- `P2/P3 + 非阻断上线`：后续能力、体验优化、延期 PRD 或清理事项。

## 5. 标注与复核流程

1. 新建 Issue 时，创建者必须给出建议优先级和上线影响，并写明触发、影响及依据。
2. Issue 进入实现前，中央调度或 PM 确认最终两个标签。
3. 每张开放 Issue 必须恰好拥有一个优先级标签和一个上线影响标签；不得同时拥有两个同维度标签。
4. 出现新运行证据、范围变化、核心部分已合并、目标 release 改变或安全降级方案获批时，必须重新评估标签。
5. Issue 关闭前仍以 GitHub 实时状态为准；本文不保存动态 Issue 数量、当前分支、测试计数或完成度快照。

## 6. GitHub 写入纪律

- 标签修改是正式 GitHub 写入，只由当前唯一 GitHub 写入 owner 低频串行执行。
- 不用 workflow 或外部 AI 自动发布 review、comment、status，也不把 GitHub 当模型间实时聊天渠道。
- 标签变更不等于实现、独立复核、合并、发布或生产授权。
- 若标签与 Issue 正文的稳定风险结论冲突，先修正文或形成 PM 决策，再修改标签，不得静默掩盖冲突。
