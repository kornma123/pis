## 一、结论速览（TL;DR）

**最重要的一条：这个问题目前答不出来——而这本身就是最有价值的发现。**

专家要我们查「谁访问、点了什么、筛了什么、**导出了什么**」。核实后：**这四问，现有任何日志或埋点一条都回答不了**，因为：

1. 旧盈利视图是**纯只读页**（只发 `GET` 请求）；
2. 全站写操作审计中间件 `operation_logs` **结构性地只记写操作（POST/PUT/PATCH/DELETE），对 GET 天然不记**；
3. 前端**零浏览/点击埋点**（全量 src 约 290 个源文件 grep 遥测关键字 = 0 命中）；
4. 现有 `operation_logs` 本身是 **13 行 seed 样例**（同一时刻批量插入），根本没有真实用户流量。

因此，"有人真在用 vs 从来没人点"——**当前无法区分**。任何声称"日志显示这页被访问/导出了 N 次"的说法都是**编造**（两条承重结论已过对抗复核，见 §六）。

**据此，新视图的需求清单只能来自两个诚实来源**（本文档 §四、§五给出）：
- **能力对等**：穷举旧视图今天暴露给用户的每一个能力，作为"需评估是否保留"的候选（不是"必须全保留"）；
- **专家点的三处高风险**：① 账户明细下钻/历史 ② 导出 ③ 期间选择/跨期对比——逐一核实其在旧视图里的**真实形态**（很多和直觉不一样）。

**顺带两个关键旁证**（详见 §三、§七）：
- 有资格看这页的**活跃用户只有 4 个**（管理员 / 财务 / 2 名病理医师），占 12 名活跃账号的 1/3；其余 8 名活跃账号（6 技术员 + 采购 + 仓管）看不到。
- 新口径的后端接口（`hospital-pnl` 影子路由，P0 院级贡献毛利·标准成本）**已经建好了，但没有任何前端在用**——旧页用的是另一条 `partner-pnl`（ABC 全成本）。新视图不是从零起，是给已存在的影子接口接线。

## 三、核心发现：使用行为当前无法测量（三重结构性缺口 + 一个诚实边角）

### 3.1 缺口一——写审计对 GET 天然不记（结构性）

`operation_logs` 由全局中间件 `auditWrite` 写入，但它**只作用于写方法**：

```
middleware/audit-log.ts:37   const MUTATING = new Set(['POST','PUT','PATCH','DELETE'])
middleware/audit-log.ts:296  if (!MUTATING.has(req.method)) { next(); return }   // GET 在写审计逻辑前早返回，永不落库
```

中间件在 `app.ts:74` 全局挂载（路由之前），头注释亦明写「对读(GET)/公开接口/失败请求天然不记」。旧盈利视图的**浏览、切月份、点行下钻、看趋势、看待复核面板——全是 GET**，因此**结构性地零落库**。

### 3.2 缺口二——前端零读侧埋点

全量 grep `前端代码/src`（约 290 文件）关键字 `analytics | telemetry | trackEvent | beacon | 埋点 | gtag | mixpanel | amplitude | posthog | logEvent | pageview | sendBeacon` = **0 命中**。放宽再搜 `track( / reportUsage / usageLog / 访问日志 / 点击埋点` 的 4 个命中全是 `report()` 误匹配（`useCostAnalysisPage.ts` 的 `setXxxReport` setState），**无一为遥测**。前端 axios 拦截器（`api/request.ts`）只附 token、处理 401 刷新，**不上报任何访问事件**。

→ 客户端侧同样无法还原谁点了什么、筛了什么、导出了什么（导出即便存在也多是纯前端 `xlsx/jsPDF`，连后端请求都不打）。

### 3.3 缺口三——现有 `operation_logs` 是 seed 样例，非真迹

`sqlite3 coreone.db` 实测：

| 指标 | 值 |
|---|---|
| `operation_logs` 总行数 | **13** |
| `created_at` 范围 | 全部 = `2026-05-21T05:52:02.713Z`（同一时刻批量插入） |
| distinct `operation`（7 种，全为写动作） | 物料领用×4、采购入库×3、出库审批×2、退货处理×1、系统初始化×1、报废处理×1、库存盘点×1 |
| 含 `hospital/pnl/partner/盈利/毛利/导出/export` 的行 | **0** |

这 13 行 `operation` 是**手写中文业务标签**（如"采购入库"），而非中间件真跑产生的 `${method} ${module}` 英文格式，且 seed 库 schema 连 `outcome` 列都还没有（P-3 迁移列运行时才补）——**足证它是人工种子数据，不代表任何真实用户行为**。

### 3.4 唯一诚实边角（不改变结论）——易失的 stdout 请求日志

对抗 skeptic 逮到一处需披露的细节：`app.ts:66-69` 有一个请求日志中间件：

```
app.ts:67   console.log(`[${new Date().toISOString()}] ${req.method} ${req.path}`)
```

它运行时确实会把 `GET /api/v1/partner-pnl` 打到进程 **stdout**。但它**无法**回答四问，也不是持久日志：
- 只有 `method + req.path`，`req.path` **已剥掉 query** → 拿不到"筛了什么账期/医院"；
- **无用户身份** → 拿不到"谁访问"；
- 导出是纯前端行为、不打后端 → 拿不到"导出了什么"；
- 是**易失 stdout**，仓库内**无 morgan / access.log / createWriteStream 任何落盘或可查询存储**（已 grep 确认为 0），进程重启即丢。

→ 结论不变：**无任何持久遥测能还原该只读视图的使用**。

### 5.1 ① 账户明细下钻 / 历史——真实形态 ≠ "明细页 / 历史链接"

专家担心会漏这个。核实：旧视图的"下钻"是**纯页内形态**——
- 点某院行 → 只**重绘页内"月度趋势"折线图** + 给该行加选中高亮；
- **没有** `/hospital-pnl/:id` 明细路由、**没有**"查看历史"链接、**没有**弹窗明细（`App.tsx` 只有一条 `/hospital-pnl` 路由）。
- 所谓"历史" = 单院月度趋势折线图而已。

**关键（对新视图是机会而非风险）**：账户→**病例级**明细的数据**后端已经有**（`GET /cases` 返回逐 case 的 `caseNo/labRevenue/costTotal/grossMargin`，按 `(partnerId, caseNo)` 复合键防取他院同号成本），但**前端只消费了 `onlyFlagged=true` 的负毛利子集**喂待复核面板，**没做**"点某院 → 看该院全部 case 明细"的下钻。→ 新视图若要做真正的账户→病例下钻，是**给已有数据接线**，不是新建后端。

### 5.2 ② 导出——旧视图自己就没有（见 §六详评）

旧视图**零导出**。所以"保留导出"对能力对等是**空命题**（本来就没有可保留的）。专家真正的担忧是**反向**的：新视图若砍掉排名/自主分析，用户会**去别处导出重建旧排名**。这条路径今天走不通（§六），但设计新视图时要**主动守住这道约束**，而不是默认它安全。

### 5.3 ③ 期间选择 / 跨期对比——真实能力：只有"单月切片" + "单院时序"，**没有跨期并列**

- 期间控件只有**一个单月选择器**；无第二期间选择、无"本月 vs 上月"并列列、无环比/同比。
- 唯一含多期信息的是**月度趋势折线图**，但它是**单家医院自己的时间序列**——只能看一院跨月走势，**不能跨院在同一期间对比、也不能任意两期并列**。切换看哪家要靠点表格换行。
- → **趋势图 ≠ 跨期对比视图**。若新视图（校准视图）想提供真正的"跨期/环比"，那是**新增能力**，不是"保留旧能力"——别误以为旧视图已有跨期对比。

### 5.4 旁注：后端数据比旧视图展示的更富（新视图可直接用）

`GET /` 返回的 `PartnerPnl` 还含 **NGS 外购转销整套字段**（`ngsRevenue/ngsCost/ngsMargin/ngsOrderCount/totalMargin=技术毛利+NGS毛利`）、`avg*PerCase`、`qualityCounts` 明细——**旧视图这些一律没展示**。新视图若要展示 NGS 渠道或每例均值，数据现成。

## 六、导出治理死角评估（专家最担心的一条）

**评级：真实但当前尚未激活的潜伏风险（低—中），不应夸大成"用户已能一键导出重建旧排名"。**

三点均有代码支撑：

1. **旧盈利视图自身零导出**——全文无 `Download/downloadTextFile/XLSX`，`partner-pnl` API 也无 export 端点。今天用户在利润页**无法一键导出**院级排名。
2. **没有任何兄弟页既显示 partner/医院级利润、又能导出**：唯一消费 `partnerPnlApi` 的就是本页；全站真实可导出的盈利/成本页（`/abc/profitability`、`/abc/dashboard`、`/abc/variance`、`/abc/trend`）**一律按检测项目 / BOM / 月份聚合**，后端 `/profitability`、`/export`、`/variance` 维度只有 `project/bom/month`（`abc-v1.1.ts:2616/2660/3300`），**根本没有 partner/hospital 维度**，导出里也不带医院映射 → 用户**无法从任一兄弟页导出重建医院排名**。
3. **因此"导出到 Excel 重建旧排名"这条路当前基本走不通**。唯一残留泄漏是**肉眼抄屏** `hospital-pnl` 那张表（P-2 止血后已改为按毛利降序、中性色、不点名、不红绿，但表仍在、仍排序）——是抄屏，不是导出。

> **全站导出习惯参考**（刻画背景）：真实引擎有两个——客户端 CSV（`lib/utils.ts:47/52`）、客户端 XLSX（`await import('xlsx')` + `XLSX.writeFile`，见入库/出库/LaneC/系统日志）；另有一批**死按钮**（`report/CostAnalysis` 的导出只 `toast('报告导出成功')` 不落文件、`bom` 导出 `toast('开发中')`、`reconciliation` 导出按钮无 `onClick`）。`jsPDF/saveAs` 全仓零真实使用。

**给新视图的护栏建议**：明确保留"**院级利润不落 Excel**"这道约束。两个未来触发点会立刻把死角变成逃生口，需设防：① 给 `hospital-pnl` 补"导出"按钮；② 给 ABC `profitability` 导出加 `partner/hospital` 维度。任一发生，都要同步评估"是否等于把旧排名以更硬的形式重新发出去"。

## 七、重要旁证：新口径后端（`hospital-pnl` 影子路由）已存在、但无人消费

审计中发现后端**同时挂着两条并行路由**（ADR-003「并存不改」）：

| 路由 | 挂载 | 口径 | 前端消费者 |
|---|---|---|---|
| `/api/v1/partner-pnl` | `app.ts:134` | ABC **全成本** grossMargin（中圈）——**旧视图真正在用的那条** | ✅ `HospitalPnLDashboard`（本页） |
| `/api/v1/hospital-pnl` | `app.ts:136` | **P0 院级贡献毛利·标准成本**（内圈·**影子模式** shadowMode·四轮外审收敛两层框架） | ❌ **无任何生产前端**（唯一调用方是它自己的 vitest 测试） |

**含义**：
- 前端那个叫"医院盈利看板"的页 `/hospital-pnl`，实际打的是 `partner-pnl`（**页面路径名与后端接口名撞车、易误解**）。

## 十二、给非技术 PM 的白话汇总

**我们做了什么**：在动手换掉旧的"医院盈利看板"之前，先查了一下"这个页到底谁在用、点什么、导出什么"，好让新版按真实需求来做、别凭记忆猜。

**查出来的结果**：
1. **最重要的一条——这个问题现在根本查不出来。** 系统的操作日志**只记"改数据"的动作**（入库、领用这些），**不记"看页面"**；而这个盈利页是纯查看页，所以它被谁看过、筛过哪个月、导出过什么，**系统里一条记录都没有**，也没有任何"访问统计"埋点。现有那点日志还只是 13 条**演示用的假数据**。所以"有人天天在用"还是"根本没人点"，**目前分不出来**——谁要是说"日志显示看了多少次"，那是编的。
2. **有资格看这页的人很少**：全部 12 个活跃账号里**只有 4 个**（管理员、财务、2 个病理医生）打得开，其余 8 个活跃账号（技术员、采购、仓管）根本看不到。
3. **这个旧页其实很朴素**：能按月份筛、有一张医院排名表、点一家医院能看它的月度走势、还有一个"亏钱病例"清单。但它**没有任何"导出"按钮**——想把排名弄进 Excel 只能对着屏幕手抄。而且它的"下钻"很浅（只能看趋势图，没有真正的明细页），"跨月对比"也没有（只能看一家医院自己的走势）。
4. **一个好消息**：新口径（更准的"院级贡献毛利"）的**后台其实已经建好了**，只是还没接到任何页面上——现在这个旧页用的是另一套旧算法。所以做新版更像"给已有的新后台接个新界面"，不是从头造。

**对业务/用户的意义**：
- 换新页时，**照着旧页有哪些功能来对齐**（排名表、按月筛、单院趋势、亏损病例清单），而不是凭印象；其中要特别小心的是——旧页**本来就没有导出**，新页也建议**默认不加导出**，免得有人把"哪家医院最赚/最亏"的排名导成表格到处传（这正是需要治理的敏感点，目前这条路恰好是堵着的，别无意中打开）。
- **一个方法论提醒**：这次我们被"看不见用户怎么用只读页"卡住了。建议以后（在满足脱敏红线的前提下）给这类查看页补一个最简单的"使用统计"，否则下次换页我们还是两眼一抹黑、还得靠猜。**这条要不要做、做多细，需要您拍板。**
- 还有一个顺序问题也想请您定：新页到底是"接那套已经建好的新后台、然后把旧的退役"，我倾向这样做，但涉及退役旧接口，想先跟您确认。
