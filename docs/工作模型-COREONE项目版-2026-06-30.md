# 工作模型 · COREONE 项目版

> **版本**：2.0｜**性质**：通用 PM–AI 工作模型的 COREONE overlay
> **先读**：[工作模型 · 通用版](工作模型-通用版-PM+AI-vibe-coding-2026-06-30.md)。本文只增加 COREONE 的事实源、升档条件、验收门和项目教训，不复制通用流程。
> **policy owner**：PM（产品目标与业务政策）｜**maintenance owner**：每次修订只能有一个 owner，具体身份记在 PR/handoff｜**reviewer**：未参与同一实现的独立视角
> **last-reviewed**：2026-07-12｜**next-review-trigger**：项目权威索引、成本/收入口径、权限/隐私政策或发布治理发生变化时

## 0. 继承与覆盖关系

```text
通用版（PM–AI 默认协作）
  └─ COREONE 项目版（只加项目约束）
       ├─ 业务权威：成本域索引 / 收入口径 / ADR
       ├─ 可机验事实：golden registry
       ├─ 开放决策：PM 待拍板队列
       └─ 动态执行：PR / 每任务 handoff / GitHub / Git
```

规则：

1. 本项目版**默认**只收紧通用版；确需例外时，必须记录证据、有权角色批准和重审触发，且不得绕过法律、组织或平台正式门。
2. 本文不是业务口径正文。涉及收入、成本、权限或数据时，必须继续读下列权威文件，不能靠本文摘要实现。
3. Git/worktree/PR/跨工具运行纪律由 [agent operating contract](agent-operating-contract.md) 管；本文不复制其规则。
4. 对应业务域有权威索引时，先按“该域权威索引 → 当前 ADR/spec → golden registry → 历史材料”的顺序裁决；没有索引时由任务 owner 找到该域明确入口并记录，不能拿成本域索引裁决 RBAC/前端问题，也不从文件日期或标题猜。

## 2. COREONE 风险升档

通用版 R0–R3 在本项目按以下触发器执行：

### 至少 R2

- 收入、成本、院级贡献毛利、收费拆分、对账、关账/反关账、golden；
- 跨表 join、数据导入、迁移、派生列、历史重算；
- RBAC、认证、账号状态、审计、患者/医院财务数据；
- 新增或改变关键用户流程、财务决策看板；
- 任何会把 Candidate Anchor 误当 Locked Golden、或把估计值展示成事实的改动。

### 必须 R3

- 生产部署、生产数据库写入或迁移；
- 密钥/令牌/账号轮换，厂商侧吊销，以及影响生产、业务/个人数据或外部利益相关者的高风险仓外写操作；
- 删除业务数据、清 Git 历史、force-push、不可逆关账动作；
- 对外财务更正、正式披露或会影响医院关系的动作；
- 修改 merge/release authority 或平台治理设置。

普通错字、小样式和明确的内部文档修订仍可 R0；一般 CRUD 可 R1。任务已授权的常规分支 push、开/更新 PR 或评论按 operating contract 执行，不自动升事故级。不要因为项目处理过高风险事故，就让每个小改动都走重流程。

COREONE 的正式合并门、批准条件与例外一律以 operating contract 为准。通用版 R0/R1 的轻量化只减少 PM–AI 方法论产物，不改变该正式门。暂时没有 reviewer 时，只能把工作做到“已实现/已验证、待审批”；若要增加低风险文档例外，必须在 operating contract 层显式决定，本 overlay 不越权改门。

## 3. COREONE 的完成证据

### 3.1 金额、口径与 golden

1. 先从权威索引确定当前 spec/ADR，写清范围内、范围外、事实源、降级和重审触发。
2. PM 先确认大白话规则和验收例子；真数据已就绪时再确认真实结果。未就绪则明确 Candidate Anchor，可以开发但不能宣称产品验收完成。
3. 至少要有正例、会失败的反例和守恒/对账关系；实现自己产出的结果不能独立证明自己。
4. 凡涉及收入/成本口径：**有变化必须**更新对应 ADR/golden；确实无变化时，才可在任务证据包中声明“无口径变化”并说明理由。
5. `golden registry` 的 `✅` 加上目标版本的 live CI 阻断证据，才能说“已锁”；任一缺失即降为 Candidate Anchor。`⬜` 必须说“待校准/未进 CI”。不得在本文件复制当前金额、测试数量或 CI 名称。

### 3.2 权限、账号与敏感数据

权限验收至少包含：有权角色能完成、无权角色 API 被拒、前端入口与后端权限一致、禁用/删除账号不能靠旧 token 继续、审计记录不泄露敏感值。只看菜单或按钮不是权限验收。

真实患者和医院财务资料不得以个人 `Downloads` 路径作为权威。每个 R2/R3 数据任务应有 source manifest；已有有效 manifest 时直接引用，不重复填表：

```text
source id / 业务期间 / 保管人
来源系统与导出方式 / 文件 hash 或稳定版本
用于计算的最小字段 / PII 等级与脱敏方式
受控存储位置（不写个人绝对路径）
缺失时降级 / 何时重取或重审
```

未经明确必要性与脱敏审查，不把原始 PII、密钥、生产库或个人下载文件提交到 Git、日志、测试样本、AI 对话/上传、外部 connector 或浏览器工具。真数据尚未就绪时可用 Candidate Anchor 开发，但不得宣称产品验收完成。

### 3.3 UI、导入与业务闭环

- 新的关键流程或财务决策界面：实现前先让 PM 看 mockup；实现后用真页面、真实权限、错误态和代表性数据走查。
- 数据导入：除解析正确外，还要验幂等、失败不留半写、回滚、重复数据、边界值和下游可消费。
- 当声称“端到端用户业务功能完成”时，“页面有按钮”或“E2E mock 了一个响应”不够；验收必须穿过真实前端调用、后端路由、权限、适用的持久化/数据结果，再回到用户可见结果。纯展示、局部组件或独立后端能力按其声明边界验收，不强造全链路。

### 3.4 安全、生产与仓外动作

安全任务必须把两层状态分开：

- **代码/仓库已止血**：漏洞代码、默认值、扫描器或发布包已修，且有测试/检查证据。
- **外部项终态**：逐项分开写——`已执行并技术验证` 才允许说该项“已止血”；`NOT_APPLICABLE` 说明为何无需执行；`风险已接受、动作未执行` 必须明写仍未技术处置，并记录 owner、期限和重开触发。

第一层绿不等于外部项已执行。高风险仓外写在执行前要有明确授权、目标和范围；dry-run/备份/回滚在适用时必须做，不可回滚时先披露并给止损/前滚方案。未取得上述三种诚实终态之一时，只能写“待 operator 执行/验证”。

## 4. 院级贡献毛利：从口径到就绪门的五层状态

这是 COREONE 最能说明“为什么不能只说测试绿”的例子。相关任务必须逐层报告：

| 层 | 什么才算到位 | 权威证据 |
|---|---|---|
| **口径已决定** | 收入分子、可避免材料、月轴、工序范围、数据就绪门和权限已被 PM/ADR 明确 | 成本域索引指向的 P0 spec + ADR |
| **数据已就绪** | 指定医院/月的对账单、工作量和抗体明细可 join，期间/病例号/脱敏均已核 | source manifest + 数据就绪检查 |
| **引擎已实现** | 计算、降级、状态词、权限和失败路径已编码 | 代码 + BDD/单测/集成测试 |
| **产品已验收** | 已关账的脱敏真实医院月端到端手核，包含要求的守卫场景，并形成成本侧 Locked Golden；财务界面已走真人验收 | [ADR-007 真医院月验收闸](COREONE-ADR-007-P0验收闸-真医院月端到端手核成本golden-2026-07-04.md) + golden registry + PM 验收 |
| **已发布** | 目标环境部署、数据迁移/权限/健康检查完成，有回滚点与目标系统证据 | operator 回执 + 上线后业务检查 |

“引擎能跑”不等于“P0 完成”；“fixture 单测绿”不等于真数据接通；“PR 合并”不等于发布。

## 5. 用真实案例校准本模型

以下写的是稳定教训，不是当前进度快照；现状仍回到 GitHub、权威文档和目标系统查询。

1. **纯实验室收入**：真实对账单与 LIS 逐病例核对后，曾把一个总账锚误当成纯实验室收入锚；错误在标签/用途，不代表原金额计算本身必然错误。有效口径与锚状态始终以 G1 + golden registry 的现场证据为准。教训是先锁范围与事实源，再锁数字。
2. **贡献毛利**：口径、ADR、阈值讨论和引擎实现只是不同就绪层；真正完成还需要已关账真实院月、三件套 join、手核、守卫场景和成本侧 Locked Golden。教训是把“定义 ready、数据 ready、实现 ready、验收 ready、发布 ready”分开。
3. **安全事故**：公开仓库代码止血后，厂商吊销、生产密钥与账号轮换仍是仓外事项。教训是仓库证据不能冒充目标系统技术验证；外部项必须有技术验证、证据化不适用或具名风险接受之一，高风险动作仍须授权。
4. **假功能/假绿**：项目曾出现真按钮和被 mock 的 E2E，但后端入口不存在；也出现测试吞掉自己的失败。教训是验收要穿透真实链路，并用会失败的反例证明测试有牙。
5. **栈式 PR 与 AI 复核**：PR body 里的“多轮复核”与 co-author 记录不等于 GitHub 正式 review/check；栈上游变化后，旧结果也不能自动覆盖新基线。教训是咨询性复核、平台审批、PM 验收和 merge authority 四者分别取证。
6. **多会话与状态漂移**：把测试数、下一步和进行中状态写进长期模型，或让多个会话共同追加大日志，会快速过期并制造冲突。教训是一文件一 owner，动态状态只进 PR/每任务 handoff，稳定教训才进本文件。

## 6. 多 AI / 多会话在 COREONE 的任务卡

通用版 handoff 字段之外，R2/R3、多会话或跨 owner 的 COREONE 任务还要写；R0 不建，R1 只填适用项：

```text
涉及的业务域与权威入口
是否命中收入/成本/golden/RBAC/PII/生产升档
数据 source manifest（如涉及真数据）
golden：无影响 / Locked Golden / Candidate Anchor
口径：引用 ADR / 新待拍 / 明确无变化
owned files / excluded files / 上下游 PR
仓外动作：已授权 / 待授权 / 不涉及
```

共享事实链（例如同一收入分子同时被 API、报表和 golden 消费）要在任务卡列出所有消费者。ownership claim 与冲突停手规则沿用通用版 §6；reviewer 若要接修先完成 handoff。栈式 PR 必须标真实依赖和“能否单独落地”。

## 7. 现在就对本用户启用

Codex 与 Claude 默认使用通用版 §0 的开场和 §8 的进度/决策/验收/收尾模板，PM 不需要重复要求。本项目只追加四行，避免复制模板后再次漂移：

```text
COREONE 权威入口：本任务跟哪份索引 / ADR / spec。
升档命中：收入/成本/golden/RBAC/PII/生产中的哪一项，或“未命中”。
数据与 golden：source manifest；Locked / Candidate / 无影响；当前就绪层。
仓外动作：不涉及 / 待授权 / 已授权待执行 / 已执行待验证 / 已执行并技术验证 / NOT_APPLICABLE（证据化不适用）/ 风险已接受但动作未执行。
```

## 8. 演进与防漂移

1. 普适教训改通用版；COREONE 专属触发器/权威路由/稳定案例改本项目版。
2. 新业务决定进 ADR；可机验锚进 golden registry；开放问题进 PM 队列；当前状态进 PR/handoff。
3. 不引用个人目录、私有 memory key 或某个模型专属技能作为项目权威；稳定规则必须能从 fresh clone 的仓库入口找到。
4. 修改工作模型时，先查询 live ownership claim；每个实际触碰文件在任务卡/PR 声明唯一 owner 和影响范围，reviewer 按风险与正式 gate 决定，不在别人的开放 PR 上代写。
5. 历史版本由 Git 保留，不靠正文堆积当前 PR/SHA/测试数来“留证”。
