# COREONE 真跑验收质量 Loop（薄入口）

> **状态**：v1.0（2026-07-12）· 待 PM 定稿拍板
> **从属**：本文是《`docs/COREONE-质量Loop契约-2026-07-12.md`》+ PM 验收铁律「真跑真实数据 / 审全部功能点 / 往前推前先交讨论项」在**验收阶段**的薄入口——只写本阶段增量，其余全在契约。⚠️ **验收 ≠ 跑测试**：单测/CI 绿只是写码 loop 的出口，本 loop 是人视角真跑（上次真跑当场逮出单测全绿却存在的多个 bug，这就是它存在的理由）。冲突以契约与其上游为准。
> **输入** = 定稿 PRD 的验收例 + 定稿 mockup + 已完成实现；bug 修复验收例 = 原复现路径 + 受影响相邻功能点。

## 0. 位置

写码 loop 完成（PR 合并）→【真跑验收（本文）】→ PM 验收拍板 → 才算「完成」（= 消费者被服务；发布/上线是独立状态，见契约 §4 / 总览从属头）。

## 1. 本阶段增量（契约之外，验收特有）

**① 起真环境**：前后端真起、页面真登录真点（不用「代码看起来对」替代）、真实业务形态数据（真项目名/真金额量级/真单据结构）。**验收可能创建或改写未跟踪的本地 dev 库与临时运行产物；这些文件均不得提交**，提交前核对 working tree 与 staged delta（细则见 guardrails）。
**② 逐条跑 PRD 验收例**：全部功能点不抽查，逐条记「过/不过」+ 证据（截图/录屏/接口返回）；验收例之外的主流程也整条走一遍。禁「跑了几个都对」= 全对的抽查心态。
**③ 换角色再跑**：每个相关角色用**运行库真实权限**登录跑一遍（该看见的看见 / 不该看的看不见 / 不该点的点了被稳妥拒绝、提示说人话）。
**④ 边界与误操作 + 与定稿 mockup 逐屏比对**：空态（新库第一次进来）/ 极端数据（0 条 / 999+ / 超长名 / 大金额）/ 手滑（双击/回退/中途刷新）；长得不一样的地方要么改实现、要么回 Mockup 重拍，不「差不多就行」。
**⑤ 问题三归类**：实现 bug → 回写码 loop（附复现路径）/ 需求·设计问题 → 回 PRD 或 Mockup 重拍（**定稿是唯一基准；基准错了改基准、不许现场绕过**）/ 新想法 → 登记待办、**本次不扩范围**。
**⑥ 交验收包**（验收特有）：验收例勾选表（逐条 过/不过 + 证据）/ 角色视角勾选表 / 问题清单（已三类归好）/ 没覆盖到的边界（诚实列）/ 结尾大白话。

## 2. 验收特有退出追加项（叠加契约「通用定稿退出模板」，不重写那四条）

- 验收例 100% 跑过且绿（证据齐）；相关角色视角全跑过；mockup 比对无未拍板差异；
- **口径台账的等级/源未被偷改**（本阶段只复核「没被偷改」；真值验证走队列、不塞进验收快门——规则见契约 §5）；
- 问题清零，或 PM 逐条明确「接受带病上线」并已登记。

（「PM 明确说验收通过·沉默不算」等四条通用退出条件在契约「通用定稿退出模板」，本入口不复述、只加上面这些验收特有项。）

## 3. 碰钱/口径功能加固（叠加契约）

① 黄金锚逐位不变；② **独立重算**：抽至少一条真实样本、脱离系统手工重算一遍核对（验「算法对不对」，不只看系统自洽）；③ 审计留痕抽查：该留痕的写真的落了审计、operator 记对了人。修完 bug 至少重跑受影响功能点；碰钱改动全量重跑。

## 5. 给 PM 的大白话

最后一关：代码合并了不等于做完。AI 把系统真的跑起来、用真实样子的数据、照你拍板过的需求清单一条一条点过去、每条留截图证据；还换成每种角色各跑一遍，确认该看的能看、不该看的看不见。发现的问题分三堆：程序错了回去改、需求本身有问题回来找你重拍、临时新想法记下以后再说不夹带。涉及钱的还要抽一条真实样本手工重算一遍核对。你亲口说「验收通过」，这件事才算完。
