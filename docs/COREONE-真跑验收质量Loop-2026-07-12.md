# COREONE 真跑验收质量 Loop

> **状态**：v1.0 草稿（2026-07-12）· 按「产出必过质疑关」规矩，PM 质疑/拍板后生效
> **从属关系**：PM 已立验收铁律「验收要真跑真实数据、审全部功能点、往前推之前先交讨论项」的流程化落地。输入 = 定稿 PRD 的验收例 + 定稿 mockup + 已完成实现；bug 修复的验收例 = 原复现路径 + 受影响的相邻功能点。
> **适用**：功能合并后（或候选版本）的验收。⚠️ **验收 ≠ 跑测试**：单测 / CI 绿只是写码 Loop 的出口条件，本 loop 是人视角真跑——上次真跑验收当场逮出单测全绿却存在的多个 bug，这就是它存在的理由。

## 0. 在工作流里的位置

```
写码 Loop 完成（PR 合并）→ 【真跑验收 Loop（本文）】→ PM 验收拍板 → 才算「完成」
```

使命：把「完成」的定义钉死在**消费者被服务**——功能在真环境、真数据上按 PRD 的每条验收例跑通，而不是「代码合了、测试绿了」。

## 1. Loop 全景

```
┌─> ① 起真环境（前后端真起、真登录、真实形态数据）
│   ② 逐条跑 PRD 验收例（全部功能点，不抽查）
│   ③ 换角色再跑（相关角色的真实权限视角逐个过）
│   ④ 边界与误操作 + 与定稿 mockup 逐屏比对
│   ⑤ 问题三归类：实现bug / 需求设计问题 / 新想法
│   ⑥ 交 PM 验收包
└── 有问题 → 修完回 ①（改过的功能点重跑；碰钱的全量重跑）
    验收例全绿 + 问题清零（或 PM 明确接受）+ PM 说「验收通过」→ 完成
```

## 2. 每步做什么

### ① 起真环境
- 前后端真起、页面真登录真点，不用「代码看起来对」替代。
- 数据用真实业务形态（真项目名、真金额量级、真单据结构），不用「测试1」。
- 记住：验收会跑脏 git 跟踪的 dev 库——**跑完复原**（细则以 CLAUDE.md 提交陷阱条款为准）。

### ② 逐条跑验收例
- 以定稿 PRD 的验收例为清单，**逐条**跑，逐条记「过 / 不过」+ 证据（截图 / 录屏 / 接口返回）。
- 验收例之外的主流程也整条走一遍——验收例是底线，不是天花板。
- 禁抽查心态：「跑了几个都对」不等于全对。

### ③ 换角色再跑
- 每个相关角色用**运行库真实权限**登录跑一遍：该看见的看见、不该看见的看不见、不该点的点了被稳妥拒绝（提示说人话）。

### ④ 边界与误操作 + mockup 比对
- 空态（新库第一次进来）、极端数据（0 条 / 999+ 条 / 超长名称 / 大金额）、手滑路径（双击 / 回退 / 中途刷新）。
- 与定稿 mockup 逐屏比对：长得不一样的地方，要么改实现，要么回 Mockup Loop 重拍——不许「差不多就行」。

### ⑤ 问题三归类
- **实现 bug** → 回写码 Loop（附复现路径）。
- **需求 / 设计问题** → 回 PRD 或 Mockup Loop 重新拍板（定稿是唯一基准；基准错了改基准，不许现场绕过）。
- **新想法** → 登记进待办，**本次验收不扩范围**。

### ⑥ 验收包（固定五件套）
1. 验收例勾选表（逐条：过 / 不过 + 证据）
2. 角色视角勾选表
3. 问题清单（已按三类归好）
4. 没覆盖到的边界（诚实列出，不许静默）
5. 结尾大白话：用户现在能干什么了、哪里还有已知瑕疵

## 3. 退出条件（全部满足才算「验收通过」）

1. 验收例 100% 跑过且绿（证据齐）
2. 相关角色视角全跑过
3. 问题清零，或 PM **逐条**明确「接受带病上线」并已登记
4. mockup 比对无未拍板差异
5. PM 明确说「验收通过」——**沉默不算**

## 4. Loop 护栏

| 护栏 | 规则 |
|---|---|
| 防假验收 | 「页面渲染出来了」≠ 验收；每条结论背后必须是真操作证据 |
| 防 CI 幻觉 | 别把 CI e2e 绿当全量回归网（PR 门只跑极小子集、夜间全量长期飘红）；关键流程本地真跑相关 spec |
| 防扩范围 | 验收中冒出的新想法一律登记，不现场加需求 |
| 防脏库 | 验收跑脏的 dev 库结束后复原，别把脏库提交 |
| 重跑纪律 | 修完 bug 至少重跑受影响功能点；碰钱改动全量重跑 |

## 5. 碰钱 / 口径功能的加固

1. **黄金锚比对**：相关黄金数字逐位不变。
2. **独立重算**：抽至少一条真实样本，脱离系统手工重算一遍核对——验「算法对不对」，不只看系统自洽。
3. **审计留痕抽查**：该留痕的写操作真的落了审计记录、operator 记对了人。

## 8. 给 PM 的大白话

这份文档管最后一关：代码合并了不等于做完了。AI 必须把系统真的跑起来、用真实样子的数据、照着你拍板过的需求清单一条一条点过去，每条都留下截图证据；还要换成每种角色的账号各跑一遍，确认该看的能看、不该看的看不见。发现的问题分三堆处理：程序写错了回去改代码，需求本身有问题回来找你重拍，临时冒出的新想法记下来以后再说、不许趁机夹带。你亲口说「验收通过」，这件事才算完。
