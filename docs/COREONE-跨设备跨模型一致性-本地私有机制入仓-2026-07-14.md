## 1. 元规则：规则落仓库，单机记忆只是缓存

> 法源已在仓，此处不复述：工作模型 COREONE 项目版 §8.3（不引私有 memory 为项目权威，稳定规则必须能从 fresh clone 的仓库入口找到）、契约 §2（稳定规则 vs 动态事实）、报告结论 Loop §②（「文档/清单/记忆说」≠「现状是」，核实先行）。本节只补一条执行细化：

- **入仓时限**：新立的工作规则（PM 拍板、事故教训、复核范式）在**当次任务内**落进上述权威文档或本文，不允许「先记单机记忆、日后再说」。既往机制审查已把「PM 新规只在 AI 记忆、未进仓库正文」定性为跨设备漂移病灶，本文即该病的清欠产物。

## 2. 对抗面板（L1 等价兜底）的执行纪律

> L1 复核的合法终态与等价兜底引擎的定义在质量 Loop 契约 §6（唯一源，本文不复述、不改写）。以下只补 §6 未覆盖的**面板内部、发现粒度**执行细化。

- **发现级判定必须三态：confirmed / refuted / UNVERIFIED。** 某条发现的反驳者全部因基础设施失败（会话限额、断流）而无有效票时，它是「未验证」不是「被反驳」——绝不许静默归入 killed。UNVERIFIED 必须单列，并在 PR body 诚实声明，由事实 owner 读码自核后按 surfaced-items 流程（`docs/github-issue-pr-management-loop.md`）处置。粒度澄清：这是**完成态面板产物内单条发现**的判定标签，与契约 §6 取缔的「断流→跳过→已复核」**面板终态**第三态是两个层面、互不替代——UNVERIFIED 恰是把「没验」明示出来，防它伪装成「已复核」。
  出处：2026-07-14 对账关账竞态修复（PR #189）面板，14 个 verify agent 撞会话限额，两个镜头共 6 条发现险被误判为「已被反驳」，靠 run 日志（journal.jsonl）捞回逐条自核。
- **先看失败计数再采信结果**：拿到面板产出先查失败 / 出错的 agent 数，非零就去 run 日志捞原始 findings，不直接消费聚合结论。
- **子代理必须沙箱**：凡对抗子代理带 shell / git 复现，任务 prompt 顶部写死——只使用当前会话已提供、且经核实与真仓及所有 worktree 隔离的沙箱能力创建一次性目录 / 仓库；禁止触碰真仓与任何 worktree 路径；禁止进入项目目录。当前会话 / 仓库唯一执行入口未提供该能力时，不绕过 hook、不另建 helper，须把该 shell 复现标为 UNVERIFIED 并交回入口 owner / 操作者。跑完由派发者**独立核实**（真仓 status、有无异常新分支），不信子代理自述完好。
  出处：2026-07-08 worktree-GC 对抗面板，一个子代理在真 worktree 建了桩分支并短暂删除未跟踪脚本。
- **认知护栏**（正向法源：报告结论 Loop §②「把『名字出现过』当『可用/在用』＝失真，以实测运行为准」）：**反方向同样成立**——子代理拿训练截止前的旧知识断言「某工具 / 模型**不存在**、必失败」也不可信；判定可用性只认实测运行证据（CI run、实际调用回包）。
  出处：AI 审查门联调期，一个 verify agent 断言「DeepSeek V4 Pro 非真实模型、该门永不出绿」，与 CI 实测出绿相反。

## 3. Git / Shell 操作纪律（两起真实事故的固化）

- **绝不把会改远程 / 工作树的命令当「验证」执行。**（高风险动作 dry-run / 备份 / 回滚的总则在工作模型通用版 §3 风险分档，此处不复述。）本条的操作增量：确认 CLI flag 语义用 `--help`、官方文档或 dry-run；为测试检测器 / 门禁而构造的危险命令字符串只作为**数据**喂给检测器，绝不真的交给 shell。
  出处：2026-07-10，为核实一个批量推分支 flag 的语义直接执行了它，把主仓 27 个本地分支误推到公开远端（已按 PM 拍板完整复原；远程跟踪分支的 reflog 前值是区分「可删」与「须复位」的取证关键）。
- **git 写操作必须定位到经核实的当前 task worktree**：`push` / `add` / `commit` / `reset` / `worktree` 等写操作前先核对目标 worktree 与 scope。使用仓库唯一执行入口执行**其 allowlist 已允许的** Git 写操作时，由入口在执行前验证并锁定 cwd；这不授权 allowlist 外的动词。不经该入口、且操作已由当前任务另行授权时，才使用 `git -C <绝对路径>` 显式定位。不同宿主对 shell cwd 的持久性行为不同，禁止依赖跨调用保留的 `cd` 状态。
- **破坏性动作前，status 出现非预期改动 = 停下核实来源。** 强制删 worktree、丢弃改动、hard reset 之前，若 `git status` 见到改动文件，先查是否有别的 agent 正在该处活着工作（worktree 独占底线见契约 §3，此处不复述），也不轻信另一个 agent 的「这里是干净的」转述——它看不见同一 worktree 里的第二个后台进程。
  出处：2026-07-13，凭另一 agent 的「零提交、干净」报告强制删除 worktree，销毁了第二个后台 agent 未提交的完整实现。
- **核分支 diff 纯净必用 three-dot + merge-base 交叉**：two-dot（`git diff origin/master origin/<branch>`）会把 master 在 fork 之后的新增显示成分支的「幻影改动」，误触「发现越界改动就停」的闸。分支真实贡献看 three-dot（`git diff origin/master...origin/<branch>`）；master 侧另用 merge-base 交叉验证；分支落后就先 `git merge origin/master` 再开 PR。

## 4. 在 worktree 里跑测试的正确姿势

- 当前任务合同与执行入口同时允许安装依赖、执行目标测试时，新建 worktree 没有 `node_modules` → 默认在对应 lockfile 所在目录按仓库工具链运行 `npm ci`；Windows PowerShell 若因执行策略拦截 `npm.ps1`，使用受支持的 `npm.cmd ci`。只有源 / 目标 lockfile 逐字节一致、Node / npm 与平台架构兼容，且当前会话已提供安全的目录链接能力时，才可复用同 tip sibling worktree 的依赖；没有该能力就放弃复用并回到允许的 `npm ci`，不绕过 hook、不另建 helper。独立 review 或其他缺少安装 / 测试权限的会话只能复核固定 head 的 CI 或操作者授权运行所产出的可核验证据；证据缺失或不足时，把相应运行验证标为 UNVERIFIED 并交入口 owner / 操作者，不得伪造 implementation state。
- worktree 没有设备私有 `.env` 时，fail-closed 的 `JWT_SECRET` 缺失可能连锁误红（如 `JWT_SECRET environment variable is required`、`initializeDatabase is not a function`）。先按 CI 等价环境复现再归因：一次性随机 secret 由操作者在启动会话前或 CI 安全预注入，不写 `.env`、不硬编码 secret。当前会话 / 仓库唯一执行入口没有安全环境注入能力时，停止该验证并交回入口 owner / 操作者；不得用 inline eval、命令串联或绕过 hook 临时注入。
- 若经上述校验复用依赖并创建目录链接，链接仍可能以未跟踪状态出现在 `git status`。提交前必须核对状态并只显式暂存 owned files，不能因它名为 `node_modules` 就假定已被 ignore。

## 5. 前端写权限判据（与后端同源）

> 「按 `capabilities` 能力门控（nav + 页内）」的总则已在前端标准 DoD 清单（唯一源，此处不复述）。本节补**写级（W）判据**与失效判据的甄别：

- 后端路由使用 `requirePermission('<module>', 'W')` 时，写按钮 / 写表单显隐使用 `canAccess('<module>', 'W')`（`前端代码/src/lib/permissions.ts`），两端读取同一份能力矩阵。
- 后端路由使用 `requireAnyRole(...)` 或其他具名授权组合子时，新增 / 修改的前端写判据必须在 `permissions.ts` 的集中式 helper 中精确镜像同一守卫语义；不得把 `canAccess(..., 'W')` 当成这类路由的通用判据，也不得在页面组件内另写一套角色例外。存量组件内角色判断是待现场检索和迁移的违规项，不是可沿用范式，本文不维护易漂移清单；完成迁移前后端守卫始终是最终安全边界。
- `getUserPermissions()` / `user.permissions` 是**恒空返回的失效判据**：登录 / 刷新响应**不下发 `permissions` 数组**（user 对象含 id / username / realName / role / primaryRole / roles / capabilities / canSeeCost），该函数恒返回空数组，但仍有残余调用点待迁移。依赖它、或在页面组件内临时硬编码角色集合（如 `role ∈ {admin, technician}`）的判据都会与后端持 W 人群漂移（前端把有权用户的按钮藏了、后端却放行）。
  出处：#138 / PR #162。存量违规以现场 `grep -rn getUserPermissions 前端代码/src` 为准（本文不维护清单）。另注意：`docs/FRS/FRS-01-认证与登录.md` §4.1.5 仍写「返回的 `user.permissions` 为固定数组」，与现行登录响应不符，属陈旧描述——以现行代码为准（该失真已作为浮现项交 PM 处置）。
- `canAccess` 的边界语义：`capabilities` 缺失时放行（仅无 capabilities 的 legacy user 对象会触发，真实边界由后端守卫兜底）；正常登录态 `capabilities` 必为对象（最坏是空对象）。

## 6. 并行分派铁律（一次开多条工作线时）

> 拆线按「文件所有权互斥 + handoff 自包含」的底线在契约 §4 / §8 与根 `CLAUDE.md`；「清单常失真、核实先行」的总则在报告结论 Loop §②（均为唯一源，此处不复述）。本节只补分派动作的增量：

- **共享文件保持单一实现 owner**：同一时刻只有一条线可修改一个文件；确需两条线先后触碰时，用串行 handoff 交接 owner。若必须并行，先把内容拆成不同物理文件并分别认领，不能靠分区约定或后合者 merge 消解同文件并行写入。
- **核实先行在分派场景的实证**：分派前逐项对当前 master 核实待办清单再定线——曾把 9 个候选模块核成 4 条真线，5 项「清单说缺」实为已实现。
- （Claude Code 特定）用户启动的 spawned task 默认带 ultracode 头（多代理编排 + 对抗验证）——PM 的常设偏好（2026-07-06 确认）。

## 8. 固定 SHA 复核会话在任务合同闸下的执行纪律（2026-07-23）

> 本地任务合同闸（PreToolUse hooks → `scripts/claude-task.cjs` 的 shell-guard / write-guard 等）的能力、白名单与豁免以该脚本及 hooks 配置**现场**为唯一源；本节不复制清单、不记录其当前状态，只收闸下做 fixed-SHA 复核的**执行纪律**增量。复核总合同（review preflight、固定对象、默认不改写）在契约 §3 / §6，GitHub 复核交流合同在 `.claude/rules/pr-governance.md` §8，均不复述。

- **闸的拦截文案与真因解耦；被拦先查命令形状，再查白名单，最后才疑合同。** shell-guard 拒绝时恒报「no active task contract」，与真实原因无关。排查顺序：①命令含重定向 / 串联 / 替换（`2>&1`、`>`、`&&`、`;`、`$()`、反引号、shell 循环）→ 拆成单命令或单竖线只读管道（stderr 随工具回包返回，无需 `2>&1`）；②可执行名是否在当前白名单（`cd`、`date`、`ln`、`cp`、`tee`、`env` 类不在）→ 换允许等价物；③才怀疑任务合同状态。工具 `--help` 也可能被拦，查用法直接读脚本源码。
  出处：2026-07-23 PR #59 复核，同一报错连挡 4 次调用（fetch 串联、gh / preflight 带 `2>&1`、claude-task `--help`），真因全是形状而非权限。
- **detached 复核 = 仓内硬只读。** 固定 SHA 复核进 detached worktree；R0 合同要求命名分支，而闸下复核会话取不到命名分支 → 建不了任务合同 → 仓内写入全拦（改不了 fixture、写不了 scratch 脚本、**重放不了 mutation**）。mutation 类声称只能验「辨识性测试存在、断言有判别力、且通过」，并在复核报告如实声明该边界，不写成「已亲手重放」。
- **复核会话注不进 JWT_SECRET：环境性失败按签名归因 + CI 同 SHA 证据覆盖。** §4 的注入路径在复核会话同样关闭（无合同 → 写不了 `.env`；env 前缀与 `env` 命令不在白名单）。凡带 `JWT_SECRET environment variable is required` 签名的文件级 / 套件级失败属环境缺省，**不归 PR**：先按签名归因、对账 CI 与本地测试总数缺口，该批文件以同 SHA 的 CI 原始日志为覆盖证据，剩余差异才谈缺陷；自设密钥（harness 模式）的测试文件本地照常跑。
  出处：PR #59 复核，本地较 CI 多出 20 套件 / 67 测试失败，全部带该签名；对账后本地与 CI 结论一致。
- **e2e-required 红先读 plan 步日志再归因。** e2e 诚实口径（唯一源：`docs/COREONE-PRD-整体产品-2026-07-14.md` E2E 诚实口径条、`scripts/e2e/plan-pr-e2e.cjs`、`前端代码/e2e/impact-map.json`；legacy 域未补可信 critical spec 前 fail-closed）下，命中 legacy 域的 PR 在 plan 步即被拒、**任何 spec 都不会执行**——该 PR 是「零 e2e 实证」而非「e2e 业务回归」。复核报告须分开写清「机制拦截（待 PM 裁）」与「无 e2e 覆盖」两件事，不把 job 红叉等同于业务回归。
  出处：PR #56 / PR #59 同类红，plan 步约 10 秒即拒。
- **复核操作序列（顺序即纪律）**：fetch → `gh` 读 PR / checks / 关联 Issue → 建并进入 detached worktree → 核对固定对象（head / tree / parent）+ merge-base + master 侧重叠面 → review preflight → 装依赖跑全量（失败按本节归因）→ 复跑 PR 自有 focused 测试 → 精读 diff 与服务层语义 → 前端构建 → 报告落指定位置。每步的允许形态以闸脚本现场为准。

## 9. 诚实边界（原 §8）

- 本文不实现「另一台设备 / 另一个模型如何发现这里」的路由层（Skill / hooks / 入口接线），只在 guardrails 保留内容指针，不复制执行入口。入口能力与状态从实际 Skill / hooks / script 及 GitHub 现场读取；历史实施出处见 #192 / PR #197。
- 本文收编截至 2026-07-14 的单机记忆存量；此后的新教训按 §1 元规则当场入仓，不再累积到单机。
- 出处里的 PR / Issue 编号是**历史事故定位符**（已冻结的过去事实），不是实时状态；现行状态一律 `gh` 现场查询。

## PM 大白话

**做了什么**：把只存在这台电脑上 AI 私有记忆里的一批工作规矩——怎么派机器互相挑刺才不误杀结论、哪类命令绝不能「跑跑看」、换个工作目录怎么正确跑测试、前端按钮显隐该看哪份权限数据、一次开多条线怎么分工——整理成仓库里这份正式文档，并在安全规范文件里加了一节指路；已经有正式出处的规矩（比如「讨论要循环到没新问题才动手」）只列了一张指路表，不再抄第二份。
**结果是什么**：以后换一台电脑、换一个 AI（Codex 或新的 Claude），只要拉下 GitHub 仓库就能拿到同一套规矩；其中两条来自真实事故（误推 27 个分支、误删同事 AI 的成品），固化后不会再靠「这台机器的记忆」防复发。
**对业务或用户意味着什么**：多台设备、多个 AI 并行干活时行为一致、少踩重复的坑，你不需要再给每个新会话口头重讲一遍规矩；仓库唯一执行入口管发现与执行，本文管稳定规则内容（历史实施出处见 #192 / PR #197）。
