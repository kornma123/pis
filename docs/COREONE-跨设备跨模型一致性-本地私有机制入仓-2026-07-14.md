## 1. 元规则：规则落仓库，单机记忆只是缓存

> 法源已在仓，此处不复述：工作模型 COREONE 项目版 §8.3（不引私有 memory 为项目权威，稳定规则必须能从 fresh clone 的仓库入口找到）、契约 §2（稳定规则 vs 动态事实）、报告结论 Loop §②（「文档/清单/记忆说」≠「现状是」，核实先行）。本节只补一条执行细化：

- **入仓时限**：新立的工作规则（PM 拍板、事故教训、复核范式）在**当次任务内**落进上述权威文档或本文，不允许「先记单机记忆、日后再说」。既往机制审查已把「PM 新规只在 AI 记忆、未进仓库正文」定性为跨设备漂移病灶，本文即该病的清欠产物。

## 2. 对抗面板（L1 等价兜底）的执行纪律

> L1 复核的合法终态与等价兜底引擎的定义在质量 Loop 契约 §6（唯一源，本文不复述、不改写）。以下只补 §6 未覆盖的**面板内部、发现粒度**执行细化。

- **发现级判定必须三态：confirmed / refuted / UNVERIFIED。** 某条发现的反驳者全部因基础设施失败（会话限额、断流）而无有效票时，它是「未验证」不是「被反驳」——绝不许静默归入 killed。UNVERIFIED 必须单列，并在 PR body 诚实声明，由事实 owner 读码自核后按 surfaced-items 流程（`docs/github-issue-pr-management-loop.md`）处置。粒度澄清：这是**完成态面板产物内单条发现**的判定标签，与契约 §6 取缔的「断流→跳过→已复核」**面板终态**第三态是两个层面、互不替代——UNVERIFIED 恰是把「没验」明示出来，防它伪装成「已复核」。
  出处：2026-07-14 对账关账竞态修复（PR #189）面板，14 个 verify agent 撞会话限额，两个镜头共 6 条发现险被误判为「已被反驳」，靠 run 日志（journal.jsonl）捞回逐条自核。
- **先看失败计数再采信结果**：拿到面板产出先查失败 / 出错的 agent 数，非零就去 run 日志捞原始 findings，不直接消费聚合结论。
- **子代理必须沙箱**：凡对抗子代理带 shell / git 复现，任务 prompt 顶部写死——复现只在 `mktemp -d` 的一次性仓库里做；禁止触碰真仓与任何 worktree 路径；禁止 `cd` 进项目。跑完由派发者**独立核实**（真仓 status、有无异常新分支），不信子代理自述完好。
  出处：2026-07-08 worktree-GC 对抗面板，一个子代理在真 worktree 建了桩分支并短暂删除未跟踪脚本。
- **认知护栏**（正向法源：报告结论 Loop §②「把『名字出现过』当『可用/在用』＝失真，以实测运行为准」）：**反方向同样成立**——子代理拿训练截止前的旧知识断言「某工具 / 模型**不存在**、必失败」也不可信；判定可用性只认实测运行证据（CI run、实际调用回包）。
  出处：AI 审查门联调期，一个 verify agent 断言「DeepSeek V4 Pro 非真实模型、该门永不出绿」，与 CI 实测出绿相反。

## 3. Git / Shell 操作纪律（两起真实事故的固化）

- **绝不把会改远程 / 工作树的命令当「验证」执行。**（高风险动作 dry-run / 备份 / 回滚的总则在工作模型通用版 §3 风险分档，此处不复述。）本条的操作增量：确认 CLI flag 语义用 `--help`、官方文档或 dry-run；为测试检测器 / 门禁而构造的危险命令字符串只作为**数据**喂给检测器，绝不真的交给 shell。
  出处：2026-07-10，为核实一个批量推分支 flag 的语义直接执行了它，把主仓 27 个本地分支误推到公开远端（已按 PM 拍板完整复原；远程跟踪分支的 reflog 前值是区分「可删」与「须复位」的取证关键）。
- **git 写操作一律显式定位**：`push` / `add` / `commit` / `reset` / `worktree` 等写操作一律 `git -C <绝对路径>`。不同宿主对 shell cwd 的持久性行为不同（有的宿主每条命令重置回主仓根、`cd` 不跨调用保留），多 worktree 环境下不显式定位的写命令默认打在主仓上。
- **破坏性动作前，status 出现非预期改动 = 停下核实来源。** 强制删 worktree、丢弃改动、hard reset 之前，若 `git status` 见到改动文件，先查是否有别的 agent 正在该处活着工作（worktree 独占底线见契约 §3，此处不复述），也不轻信另一个 agent 的「这里是干净的」转述——它看不见同一 worktree 里的第二个后台进程。
  出处：2026-07-13，凭另一 agent 的「零提交、干净」报告强制删除 worktree，销毁了第二个后台 agent 未提交的完整实现。
- **核分支 diff 纯净必用 three-dot + merge-base 交叉**：two-dot（`git diff origin/master origin/<branch>`）会把 master 在 fork 之后的新增显示成分支的「幻影改动」，误触「发现越界改动就停」的闸。分支真实贡献看 three-dot（`git diff origin/master...origin/<branch>`）；master 侧另用 merge-base 交叉验证；分支落后就先 `git merge origin/master` 再开 PR。

## 4. 在 worktree 里跑测试的正确姿势

- 新建 worktree 没有 `node_modules` → 先从主仓（或同 tip 的 sibling worktree）symlink：后端在 worktree 的 `后端代码/server` 内 `ln -sfn <主仓>/后端代码/server/node_modules node_modules`；前端同理指向主仓 `前端代码/node_modules`。
- worktree 没有主仓那份未跟踪的 `.env` → 直接跑全量后端测试会因 fail-closed 的 `JWT_SECRET` 要求连锁误红一批文件（报 `JWT_SECRET environment variable is required`、`initializeDatabase is not a function`）——**不是你的改动引起，别白查**。按 CI 同款姿势跑：`export JWT_SECRET="$(openssl rand -base64 48)" && npm run test:node`。
- `.gitignore` 里 `后端代码/server/node_modules/` 与 `前端代码/node_modules/` 都是**带尾斜杠的目录专属模式**——只匹配真实目录、不匹配 symlink。所以上一条 symlink 出来的 node_modules 会以未跟踪状态出现在 `git status`；guardrails 既有的「只显式暂存目标文件」纪律在 worktree 下同样适用，且多了这一个诱因。

## 5. 前端写权限判据（与后端同源）

> 「按 `capabilities` 能力门控（nav + 页内）」的总则已在前端标准 DoD 清单（唯一源，此处不复述）。本节补**写级（W）判据**与失效判据的甄别：

- 写按钮 / 写表单显隐的**唯一判据** = `canAccess('<module>', 'W')`（`前端代码/src/lib/permissions.ts`，读登录响应下发的 `user.capabilities`），与后端 `requirePermission('<module>','W')` 读同一份能力矩阵，天然对齐。
- `getUserPermissions()` / `user.permissions` 是**恒空返回的失效判据**：登录 / 刷新响应**不下发 `permissions` 数组**（user 对象含 id / username / realName / role / primaryRole / roles / capabilities / canSeeCost），该函数恒返回空数组，但仍有残余调用点待迁移。依赖它、或退化成硬编码角色集合（如 `role ∈ {admin, technician}`）的判据都会与后端持 W 人群漂移（前端把有权用户的按钮藏了、后端却放行）。
  出处：#138 / PR #162。存量违规以现场 `grep -rn getUserPermissions 前端代码/src` 为准（本文不维护清单）。另注意：`docs/FRS/FRS-01-认证与登录.md` §4.1.5 仍写「返回的 `user.permissions` 为固定数组」，与现行登录响应不符，属陈旧描述——以现行代码为准（该失真已作为浮现项交 PM 处置）。
- `canAccess` 的边界语义：`capabilities` 缺失时放行（仅无 capabilities 的 legacy user 对象会触发，真实边界由后端守卫兜底）；正常登录态 `capabilities` 必为对象（最坏是空对象）。

## 6. 并行分派铁律（一次开多条工作线时）

> 拆线按「文件所有权互斥 + handoff 自包含」的底线在契约 §4 / §8 与根 `CLAUDE.md`；「清单常失真、核实先行」的总则在报告结论 Loop §②（均为唯一源，此处不复述）。本节只补分派动作的增量：

- **共享文件显式分工**：确需两条线碰同一文件时，卡上写明「各占哪块、撞车后如何消解」（如各占不同区、后合者以 merge 消解），不留默契。
- **核实先行在分派场景的实证**：分派前逐项对当前 master 核实待办清单再定线——曾把 9 个候选模块核成 4 条真线，5 项「清单说缺」实为已实现。
- （Claude Code 特定）用户启动的 spawned task 默认带 ultracode 头（多代理编排 + 对抗验证）——PM 的常设偏好（2026-07-06 确认）。

## PM 大白话

**做了什么**：把只存在这台电脑上 AI 私有记忆里的一批工作规矩——怎么派机器互相挑刺才不误杀结论、哪类命令绝不能「跑跑看」、换个工作目录怎么正确跑测试、前端按钮显隐该看哪份权限数据、一次开多条线怎么分工——整理成仓库里这份正式文档，并在安全规范文件里加了一节指路；已经有正式出处的规矩（比如「讨论要循环到没新问题才动手」）只列了一张指路表，不再抄第二份。
**结果是什么**：以后换一台电脑、换一个 AI（Codex 或新的 Claude），只要拉下 GitHub 仓库就能拿到同一套规矩；其中两条来自真实事故（误推 27 个分支、误删同事 AI 的成品），固化后不会再靠「这台机器的记忆」防复发。
**对业务或用户意味着什么**：多台设备、多个 AI 并行干活时行为一致、少踩重复的坑，你不需要再给每个新会话口头重讲一遍规矩；这和 #192（让新设备自动发现这些规则的工具层）配套，一个管内容、一个管入口。
