## 一、两系统定位对比

| 维度 | LIS 自动计费系统（v5.2） | COREONE 按医院收入分析（PR8） |
|---|---|---|
| 所属系统 | **病理信息系统(LIS) 内的"收费系统"改造**，**不在 COREONE 进销存项目中**（沟通记录 §1.1：「收费系统不在当前项目中，需要基于病理信息系统的数据字段进行设计」） | COREONE 进销存项目内，收入侧分析（`后端代码/server/src/utils/` + `routes/case-revenue-v1.1.ts`） |
| 计算方向 | **正向**：按 81 项新收费代码 + case 数量 → **生成真实账单**（交易型，产生应收/实收金额） | **反向**：按同一套规则 + LIS 数量列 → **估算实验室技术收入占比**（分析型，不产生金额）<br/>`实验室收入 = 财务实收 × 技术占比`（`partner-pnl.ts:6-9, 50-80`） |
| 触发方式 | 用户在计费页**手动点"更新计费信息"**，与诊断流程完全剥离（沟通记录 §2.1） | 财务对账单导入（`POST /import`，`case-revenue-v1.1.ts:21`），与 LIS 计费机制无关 |
| 金额真值来源 | 自身引擎按规则算出 = 账单金额本身 | **外部对账单实收**（开单金额=计费金额×扣率，`billing-revenue.ts:25-33, 142-143`），引擎只算占比不算金额真值 |
| 收费码语义 | 写入账单的旧码/院定制码（一期切换到新码） | **导入时 code-agnostic**：账单码不解析语义，仅取金额 + 病理号 join（`billing-revenue.ts:10-11, 117-122`） |
| 共享目录 | 上海·申康《20260605 病理类项目收费代码-YZ.xlsx》新增 81 码 → 33 服务项目 | 同源 81 码目录的「常见档」子集，价格逐条照抄（**`charge-catalog.ts:5-9` 红线注释**） |
| 适用范围 | 启用新码的医院 | **任何医院**（含旧码/私立），因引擎只需 LIS 数量列（`charge-engine.ts:7-8`） |

**红线（`charge-catalog.ts:7-9` 原文引用）**：
> ⛔ 红线：本目录服务【收入侧】(实验室收入 = 财务实收 × 引擎技术占比)，与【成本侧】(`fee_standards` / `cost-calculator.ts` / ABC 引擎) **完全独立**，互不读写。

两系统是同一套申康 81 码目录的**镜像应用**：LIS 用它产生金额，PR8 用它解释金额结构。目录是唯一的耦合点 —— 也是后续治理(GOV-1)的着力点。

## 二、为何自动计费不阻断 PR8 合并（BLOCK-0）

**结论（已对抗验证，summary §B/§C BLOCK-0）：PR8 可独立合并，不依赖 LIS 自动计费上线。**

理由（逐条用源码作证）：

1. **PR8 的输入不是 LIS 账单，是两份与计费机制无关的原始数据**：
   - LIS **原始数量列**（HE 切片/蜡块/免疫组化/特染/EBER/PD-L1），`case-charge-mapping.ts:18-28` 的 `LisCaseQty`。这些数量是病理实验产生的客观计数，无论 LIS 计费系统是否改造都存在。
   - 财务**对账单实收**，`billing-revenue.ts` 以 **code-agnostic** 方式只取「开单金额」聚合（`billing-revenue.ts:10-11, 117-122`），明确**不解析收费码语义**。账单上写旧码还是新码、是否由自动计费生成，对 PR8 都透明。

2. **引擎是纯函数，目录由调用方传入**（`charge-engine.ts:11-12, 84`），与 LIS 运行时零依赖。占比计算只看 PR8 自己的 `charge_codes` 目录 + LIS 数量列。

3. **导入幂等、与触发模型无关**（summary §C COMPAT-1）：`case_revenue UNIQUE(case_no, service_month)` upsert + 明细先删后插（`case-revenue-v1.1.ts:5-6, 38-43, 87-88`），重复导入安全，与 LIS 何时触发计费无关。

因此：**LIS 自动计费是上游正向系统的演进，PR8 是下游分析系统**，两者通过「目录 + 数量 + 实收」三类数据松耦合，无运行时调用关系。PR8 当前缺口（第七节）是**引擎自身准确性**问题，与 LIS 是否上线正交，不构成合并阻断。

## 三、v2 高价值升级路径

### 3.1 OPP-1（核心）：从「数量列估算占比」升级为「按真实账单码精确分类占比」

**现状（v1 估算路径）**：PR8 没有逐项账单码，只有 LIS 数量列。它用 `mapCaseToCharges` 把数量列映射成收费项，再用引擎算**理论金额**求占比（`case-charge-mapping.ts:63-89` → `charge-engine.ts:84-108`），最后 `实验室收入 = 实收 × 占比`。占比是「基于数量列重建的金额结构」，对旧码/私立医院通用，但**对单 case 是近似**。

**v2 机会（钩子已铺好）**：账单上线新 81 码后，对账单导入时**已经把逐行 `charge_code` 原样存进 `case_revenue_lines`**（`billing-revenue.ts:10-11` 注释「原始明细行原样存(备 v2 逐码核对)」；落库见 `case-revenue-v1.1.ts:44-46, 90-91` 的 `insLine`）。届时可直接用 `charge_codes` 目录的 `category` 字段对**真实账单金额**按码分类求占比，无需重建：

```
真实占比 = Σ(技术类账单行实收) / Σ(全部账单行实收)
```

这是从「重建近似」到「按真实金额结构精确分类」的升级，钩子（存 charge_code）已在 v1 内置，零返工。

### 3.2 DATA-1：诊断占比从「HE 切片估算」变精确，救回 HE=0 的 case

v1 诊断组分靠 HE 切片数驱动（`case-charge-mapping.ts:66-67`），HE=0 时诊断占比缺失、`partner-pnl.ts:67, 70` 标 `partial_quantities`。v2 用真实账单的诊断行金额求占比即可救回（需 v2 补**远程诊断 020000** 入目录，summary B3 MISS-3）。

### 3.3 REUSE-1：样本「常规/复杂」由 LIS 权威输入替代推断

v1 用关键词 `detectSpecimenType` 推断组织/细胞学（`case-charge-mapping.ts:53-57`），复杂组织 `tissue_complex` 需人工指定。v5.2 在登记时让登记人员**手动选「常规/复杂」**（沟通记录 §2.3-2.4，常规=030000/复杂=040000），其语义与 PR8 的 `tissue`/`tissue_complex`（`case-charge-mapping.ts:15-16`）**一一对应**。v2 接入该字段即可把推断升级为权威输入。

### 3.4 ADOPT-1：双路径 + 可信度升级，渐进不推翻估算

升级策略（Adoption-First）：
- **精确路径**：医院已启用新码 且 账单码命中目录 → 按真实码分类，`quality` 升至更高档。
- **估算路径**：其余（旧码/私立/未启用）→ 继续 v1 数量列估算（`partner-pnl.ts` 现有逻辑）。

估算路径是**广度优势**（覆盖任何医院），精确路径是**深度优势**。二者并存、按可信度分档标注，**绝不用精确路径推翻估算路径**，避免「上线即断档」。

### 3.5 数据流对比图（v1 估算 vs v2 精确）

详见下方 widget 渲染的数据流对比图。文字版：

```
v1 估算路径（当前，覆盖任何医院）:
  LIS 数量列(LisCaseQty)
      │  mapCaseToCharges  (case-charge-mapping.ts:63)
      ▼
  收费项组合 CaseChargeItem[]
      │  computeCaseSplit  (charge-engine.ts:84) —— 按规则重建"理论金额"
      ▼
  各类占比(techRatio/diagnosisRatio)
      │  × 财务实收(对账单 netAmount, code-agnostic)
      ▼
  实验室收入(labRevenue)   ← 占比为"重建近似"

v2 精确路径（账单上线新81码后，估算路径继续保留）:
  对账单逐行(charge_code 已存 case_revenue_lines)
      │  按 charge_codes.category 直接分类真实金额
      ▼
  真实占比 = Σ技术行实收 / Σ全部行实收
      ▼
  实验室收入   ← 占比为"真实账单金额结构"，quality 升档
```

## 四、治理机制

### 4.1 GOV-1 [MEDIUM]：目录唯一权威源 + 价目漂移纠错报表

**风险**：LIS 与 COREONE 各自内嵌同一套申康规则，价目调整需双系统同改 → **配置漂移**（一边改了一边没改，占比/账单不一致）。

**机制**：
1. **以 PR8 `charge_codes` 表为 COREONE 侧唯一权威源**（当前种子 `charge-catalog.ts:17-55`，DB 落库读取见 `charge-catalog.ts:79-93`）。所有占比计算只读这一张表，禁止散落硬编码。
2. **为 `charge_codes` 表增治理字段**：目录版本号、生效日期、变更人（与已有 `status='active'` 配合，`charge-catalog.ts:81`）。每次申康调价 = 新版本行，旧版本停用，可追溯。
3. **价目漂移纠错报表**：用真实账单金额定期跑「**账单实收 vs 引擎理论金额**」差异报表 —— 引擎理论金额由 `computeCharge`（`charge-engine.ts:36`）按目录规则算出，与账单逐行实收对比。差异 = 漂移信号（目录过期 / LIS 未同步 / 院定制折扣）。这与**增量纠错架构**（原始不可变 / 派生可重算 / 对账差异=纠错信号）完全一致：差异不是错误而是下一轮目录校正的输入。

### 4.2 GOV-2 [LOW]：方案文档增「与 COREONE 收入分析的接口」一节

自动计费上线后，对账从「金额核对」升级为「逐码核对」（账单现在带语义化新码）。需在 **LIS v5.2 方案文档**中增补一节「与 COREONE 收入分析的接口」，明确：
- 目录治理责任方（谁维护申康码 → COREONE `charge_codes` 表为权威）；
- 账单 `charge_code` 字段须随账单导出（PR8 v2 OPP-1 逐码分类依赖它）；
- 码书写规范唯一化（见第六节 FMT-1）。

## 六、收费码格式规范化（FMT-1）

**问题**：v5.2 附录收费码**前导零位数混用 14/15 位**（如 `12100000010000` 14 位 vs `012100000080000` 15 位，见 `v5.2-design-facts.md` 附录表），且含后缀字母（`…120000c` / `…110000b`）。PR8 内部**统一 15 位**（`charge-catalog.ts:17-55` 全部 `012…` 15 位），自洽无 bug。

**影响**：
- **v1 无影响**：PR8 是 code-agnostic 估算路径，账单码不参与 join（`billing-revenue.ts:10-11`），引擎只用 PR8 自己的 15 位种子，内部一致。
- **v2 必做**：OPP-1 精确路径要拿账单 `charge_code` 去 join `charge_codes` 目录。**逐码 join 前必须先建「码规范化函数」**：统一前导零位数 + 规范后缀字母(a/b/c)，否则 `12100000010000` 与 `012100000010000` join 不上 → 真实账单行被判 unmatched，占比被低估（同 `charge-engine.ts:69-70` unmatchedCount 守卫语义，但守卫只能事后报、不能事前对齐）。

**行动**：
1. v2 实现 `normalizeChargeCode(code): string`，作为 join 前的唯一入口；
2. **上线前与 LIS 团队对齐唯一书写规范**（统一 15 位 + 后缀字母大小写），写入 GOV-2「接口」一节，作为账单导出与目录维护的共同契约。

## 七、PR8 准确性缺口清单（B3 全部，三档分类）

## 八、结论

1. **可合并**：PR8 与 LIS 自动计费松耦合（目录+数量+实收三类数据），无运行时依赖，BLOCK-0 已验证 → **PR8 独立合并不受阻**。
