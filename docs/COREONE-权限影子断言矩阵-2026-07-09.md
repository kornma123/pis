# 权限影子断言矩阵（Phase-2 权限翻转的门 · 本轮只断言不接管）

> 创建 2026-07-09 · 迁移序第 2 步（前置=路由注册表 Phase1 #107 + 授权具名组合子 #109）
> 实现：`后端代码/server/src/shadow-matrix/` · 门+埋雷：`后端代码/server/tests/shadow-permission-matrix.test.ts`

## 一句话

建一套「穷举比对**派生权限** vs **现行权限**」的影子矩阵，作为将来「权限从路由注册表派生」翻转的**机器门**：
可见性、守卫（模块/读写级别/条件）**双断言**，放宽必拦，无守卫单列，端点取运行时真相，生产角色单独出报告，
再用**埋雷自测**让机器证明这张网不漏——**本轮只断言、不接管**（绝不改任何权限判定）。

## 为什么

- 路由注册表 Phase1（#107）让「新页默认有归宿」，但**权限判定**仍由 `permissions.ts` 手工映射（`NAV_PATH_MODULE`）
  与后端 `app.ts` 挂载守卫各自决定；注册表的 `permModule` 只是**声明**、尚未接管。
- Phase 2 的计划是让注册表成为单一事实源、权限从它**派生**。**翻转前必须先有一张机器可证的门**：
  「派生出来的权限」与「现行手工权限」在每一格上一致，且任何**放宽（提权）**被无条件拦下。
- 这张门就是本矩阵。它只**读** `route-registry.ts` / `permissions.ts` / `authz-combinators.ts` 表达的守卫并比对，
  **绝不改判定**（never flip）。

## 两个独立断言（维度不同·并列跑）

> 专家终裁：上一版把可见性维度丢了（`role` 变量未用=化石）。必须**两个都在**。

### 断言 A · 可见性（角色相关 · per `role × route`）

| 侧 | 来源 |
|---|---|
| 现行 `legacyVisible(role,route)` | `permissions.ts`：`NAV_PATH_MODULE`(path→module) + `getAccessiblePaths` 角色特例（`/` 恒可见 · 导入器三页+LIS 限财务/管理员） |
| 派生 `registryVisible(role,route)` | `route-registry.ts`：active 路由的 `permModule`（`null` → 同一角色特例分类器） |

能力引擎两侧**共享**（`rbac-matrix.ts` 的 `SEED_MATRIX`/`hasLevel`），故 diff 只在「同一路由被两侧映到不同模块」时出现——正是要抓的翻转错。

| 情形 | 判定 | 理由 |
|---|---|---|
| 不可见 → 可见 | **BLOCK** | 放宽/提权，无条件拦 |
| 可见 → 不可见 | **review** | 收窄，须确认不破坏流程 |
| 可见 → 不可见 **且命中活跃写端点**（`operation_logs`） | **escalated** | 会掐死正在跑的流程，提级人裁 |

### 断言 B · 守卫（角色无关 · per `endpoint × method`）

`guardId = { module, level(R|W), conditions:Set<具名条件> }`

| 侧 | 来源 |
|---|---|
| 现行 `actual` | **运行时端点集**（`app._router` 自省·地面真相） + **源码守卫静态抽取**（挂载层 `requirePermission` + 路由中间件 + `const` 别名 + handler 内 `assertNotSelfReview`/`assertCaliberChangeAllowed`） |
| 期望 `expected` | **已批准的守卫快照** `expected-guards.snapshot.json`（ratified·committed） |

| 情形 | 判定 |
|---|---|
| 模块不同 | **BLOCK** |
| level 降级 W→R | **BLOCK** |
| conditions 少任一条（非超集） | **BLOCK** |
| 任一侧无守卫（module 空且无条件） | **UNGUARDED**（独立类别·须进 public 白名单否则红·SEC-1） |
| 守卫被拿掉（期望有→现在无） | **BLOCK**（弱化到零） |
| 收紧（升级 R→W / 多条件 / 期望无→现在有） | **review** |
| 一致 | equal |

> 别给守卫排「强弱」造语义（会漏）——就用「等式或阻塞」。

## 两道门（不许静默过）

- **CI 门** = `BLOCK==0` 且 `escalated==0`。
- **UNGUARDED** 全部须在 `public-endpoints.allowlist.json` 登记（否则红）。
- **review** 全部须在 `review-decisions.json` 裁决清单登记（否则红·逼显式裁决）。
- **端点集地面真相=运行时自省**：静态守卫与快照都对齐运行时端点集——防「端点从源码与快照同时丢 → 无 diff 假绿」。
- **路由集相等前置**：遍历前先断言 `legacyRouteSet == registryRouteSet`（双向差集空）。

BLOCK/escalated **永不能**靠登记洗白（那是放宽·无条件红）；能登记的只有 review（非放宽的收紧/drift/新端点/移除端点）。

## 埋雷自测（SEC-2 · 机器证明这张网不漏）

四维各**故意植一个 diff**，断言矩阵逐个抓到且 verdict 正确：

1. 改一处可见性（不可见→可见）→ **BLOCK**（外加收窄→review、收窄+活跃写→escalated）
2. 降一级 W→R → **BLOCK**
3. 删一个条件（conditions 非超集）→ **BLOCK**
4. 换一个模块 → **BLOCK**

加反向/边界雷：无变化→equal（无假阳性）· 收紧→review 非 BLOCK · 两侧无守卫→UNGUARDED 非静默 ·
守卫被拿掉→BLOCK · 新端点/移除端点不静默 · 解析器单测锁守卫抽取（别名/W升级/SoD/口径/挂载）。
四雷全中 = 矩阵完备性机器可证。

**端到端雷（穿过装配好的门·非只打纯函数）**：门闸装配（`evaluateGate`）——未登记白名单的 UNGUARDED /
未裁决的 review → `clean=false`，BLOCK/escalated 永不能靠登记洗白，路由集不等硬红；escalated **经 `runMatrix`
端到端产出**（收窄命中活跃写端点·非绕过纯函数）；**parser→assembly 端到端**——真守卫在临时文件里写坏经
`buildActualGuards→diffGuard` 被抓 BLOCK。**词法自测**——每个被建模源文件 `blankComments` 后 EOF 回 state 0
（正则/字符串失步=响亮红非潜伏）。

## 独立复核（机制 5·对抗面板）与已知边界

本矩阵建成后过 **6 视角对抗面板**（Workflow·守卫解析正确性 / 判定方向 / 边界安全 / 埋雷是否有牙 /
静默放行洞 / 运行时覆盖），逮到 **4 条真缺陷，均已修 + 变异证有牙**：

| # | 严重度 | 缺陷 | 修法 | 证据 |
|---|---|---|---|---|
| 1 | **HIGH** | 解析器把守卫**判别性选项**抹平成同一 condition：`requireAdmin({primaryRoleOnly})` 与 `requireAdmin()` 都记 `'admin'`、`assertNotSelfReview({failClosedOnMissing})` 都记 `'sod:self-review'` → 删掉选项=真放宽却被判 equal 静默过门 | 编进 condition：`admin:primaryRole`/`admin:rolesAware`、`sod:self-review:failClosed`/`sod:self-review` | **变异实证**：删真实 alerts 的 `primaryRoleOnly` → 门产 `PUT /alerts/rules/:id [BLOCK] 条件缺失：少了 [admin:primaryRole]`（修前静默、修后硬红） |
| 2 | MED | `escalated`（收窄命中活跃写端点）在真实路径恒死代码（`isActiveWriteRoute` 未接线） | 接 `operation_logs` 真实数据源（`activeWriteRoutesFromLogs`·活体门+生产报告都接·CI :memory 无写历史故 0·诚实） | 端到端雷经 `runMatrix` 产 escalated |
| 3 | MED | 埋雷只打纯比较函数、不穿 parser→assembly→门闸 → 欠建模 + 门闸突变可双双存活 | 抽 `evaluateGate` 纯函数独立埋雷 + parser→assembly 端到端雷 | 门闸/装配雷（见上）|
| 4 | LOW | `blankComments` 无正则词法态 → `/[",\n\r]/` 内引号误当定界符、状态机失步（abc-v1.1.ts 实测） | 加正则字面量词法态 + 词法自测（EOF state 必回 0） | 277 端点解析零变化（正则修不回归）+ 词法自测覆盖全部被建模源 |

**翻转前提（completeness·Phase-2 翻转真开门前必须先解决·非本轮 bug）**：

1. **CI 的可见性断言 A 只跑 fixture（SEED_MATRIX）**——生产角色能力可经「角色权限」页漂移，只由**手动、非-required**的
   `production-report.ts` 检查。**含义**：翻转前真正拦「生产角色放宽」的那半张门是**要记得跑**的手动步骤。
   → 翻转 checklist 必须把「跑生产报告且就绪」列为 required 前置件（无报告/未就绪=门不开）。
2. **快照批准治理**：一个 PR 若同时弱化守卫 + `SHADOW_MATRIX_UPDATE=1` 重批快照，门直接绿——防线是人工复核
   `expected-guards.snapshot.json` 的 git diff。#1 修复后**弱化会在快照 diff 里显形**（condition 字符串变化），
   但仍建议给快照文件加 CODEOWNERS 强制评审 + PR 模板问「是否重批了快照、复核了哪些端点」。
3. **页级 vs 方法级粒度**：断言 A 是 `role×route` 页级（单个 `permModule`），真实鉴权是 `endpoint×method` 级、
   且一个页可跨多模块（abc 页读 `abc_dashboard`、写 `abc_config`）。若 Phase 2 让单个 `permModule` 同时接管
   「页可见」与「端点守卫」，多模块-每页的现实会被压平、写端点的挂载层 R 前置会丢 → **这是翻转设计的前提**，
   须在翻转方案里显式解决（本矩阵已把 endpoint×method 守卫独立成断言 B、不与页级 A 混同，为此留了空间）。

## 生产角色 = 独立翻转门产物（终裁 7）

CI 跑 **fixture**（`SEED_MATRIX`）。**翻转前**用 `production-report.ts` 对**真实库角色**跑断言 A，产出带日期报告
（`docs/shadow-matrix-reports/`）；**无报告或报告未就绪 = 翻转门不开**（fail-closed）。
实测：真实库 5 角色（finance/pathologist/procurement/technician/warehouse_manager·无 lab_director/admin 行）
同样 0 可见性 diff → 与 fixture 一致。（证明报告读的是真实角色、非 fixture。）

## 给 PM 的白话汇总

**做了什么**：给系统装了一张「权限体检网」。将来我们打算让「谁能看哪个页、谁能调哪个接口」都由**一张总表**
（路由注册表）自动决定，而不是像现在这样在好几个地方手工维护。但**直接切换有风险**——万一切换后某个角色
突然能看见/操作了本不该碰的东西（提权），就麻烦了。所以**这轮先不切换**，只先建好这张体检网：它把
「按新总表算出来的权限」和「现在实际的权限」逐一比对，任何**放宽（多给了权限）都会让 CI 变红、挡住合并**。

**结果**：对当前系统跑了一遍——**完全干净**。277 个后端接口全都有守卫、前端 43 个页面的可见性新旧两套算法
**逐格一致**（说明之前的注册表迁移做得很干净）；只有 6 个**本来就该公开**的接口（登录/登出/健康检查/读自己的
权限）被单独标出来、并明确登记为「有意公开」。我们还**故意在测试里埋了 4 个假错误**（偷偷给某角色多开一个页、
把一个写权限降成只读、删掉一道审批门、换错一个权限模块），验证这张网**每个都能抓到**——证明它不是摆设。

**独立复核**：建完后专门派了一支「找茬小队」（6 个互不通气的审查视角）拼命挑毛病，**真挑出 4 处问题并当场修好**——
最重要的一处是：原来把两种严格程度不同的「管理员校验」当成一样，等于体检网有个盲点、真放宽也看不出来；
现已修好，还用「故意改坏真实代码再看网能不能抓到」的方式**验证过确实能抓**（改坏后 CI 立刻变红）。

**对业务/用户的意义**：这是给未来那次「权限体系升级」上的**安全保险**。有了它，那次升级就能有机器兜底、
不会悄悄把权限放开出漏洞；升级前还会拿**真实的角色配置**再体检一次、出一份带日期的报告，报告不过关就不许升级。
对现在的用户**零影响**（这轮一行权限判定都没改，纯加了体检和测试，所有既有功能和财务口径原样通过 1067 项测试）。
