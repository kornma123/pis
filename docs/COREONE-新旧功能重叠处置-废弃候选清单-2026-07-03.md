### ① 老 `Reconciliation`（消耗对账） vs 新 `account-reconcile`（账实核对）

**是什么**

| | 老 消耗对账 | 新 账实核对 |
|---|---|---|
| 前端页 | `pages/reconciliation/Reconciliation.tsx` | `pages/account-reconcile/AccountReconcilePage.tsx` |
| 路由 | `App.tsx:77` `/reconciliation` | `App.tsx:94` `/account-reconcile` |
| 侧栏 | `AppSidebar.tsx:59` **消耗对账** | `AppSidebar.tsx:66` **账实核对** |
| 后端 | `reconciliation-v1.1.ts`，挂载 `app.ts:111`（`reconciliation:R`） | `account-reconcile-v1.1.ts`，挂载 `app.ts:124`（`account_reconcile:R`） |
| **算什么** | **物料消耗方差**：实际出库 `SUM(outbound_items.quantity)` vs BOM 标准用量 `bom_items.usage_per_sample × 病例数`，输出 theory/actual/diff/diffRate（`reconciliation-v1.1.ts:218-296`）；含 **BOM 修正提案审批**（`/logs/:id/approve|reject`，限成本核准角色） | **账单收入核对**：差异 = 账单片数（`case_revenue_lines`）vs LIS 物理片数（`lis_cases.ihc_count/special_stain_count`），6 认定原因 + 补收 gate + 关账状态机（`utils/reconcile-account.ts`、`reconcile-compute.ts`），**不依赖成本价** |

**真实关系**：**不同域各司其职，非同域重复。** 一个是"物料到底用了多少 vs BOM 该用多少"（成本/耗用方差），一个是"医院账单片数 vs 我们实际做的物理片数"（收入/账实差异）。唯一交叠点 = 共用 `lis_cases` 病例底表，但**消费列/口径完全不同**（老用 `COUNT(*)` 当消耗理论量分母；新用 `ihc_count/special_stain_count` 当物理片数比对量）——同表不同列、正交复用，不构成功能重复。`pages/account-reconcile/` 内 grep `/reconciliation/` 零命中，无隐藏交叉依赖。

**两条 API 是否都仍在用**：都在用。
- 老：`useReconciliationPage.ts` **11 处**真实调用 `/reconciliation/*`（`:122` summary、`:130` projects、`:138` materials、`:148` cases、`:167` logs、`:199` projects/:id/materials、`:213` cases/import、`:257` logs、`:290` logs/:id/approve、`:305` logs/:id/reject、`:316` cases/:id PUT）。
- 新：`api/account-reconcile.ts` **12 处**调用 `/account-reconcile/*`（overview / compute / workbench / diffs/:id/verdict / hospital-months/* / close / supplements/* …）。

**三档建议**：
- ✅ **保留双方并明确分工**（推荐）——两者是不同对账域，各有独立页/侧栏/权限模块/活跃消费方，均**不可退役**。
- ⚠️ 加下钻导航——**不建议强行加**：两者无下钻关系（一个走物料成本方差、一个走收入账实差异），硬连反而误导口径。
- ❌ 合并 UI / 合并路由——**不建议**：会把两个不同域的对账挤进一个信息架构，口径混淆。若产品侧希望统一"对账"入口，**建议在信息架构上分组为「消耗对账」「账实核对」两子项**，而非合并路由。

**退役前置核实**：老 `/reconciliation` 有 11 处活跃前端调用 → **不能退役**（inline + codex 一致）。

**PM 待拍（本对专属边界）**：老 Reconciliation 的病例导入（`/reconciliation/cases/import`）与 `lis-cases` 导入是否去重？当前两处各有 `MAX_LIS_IMPORT_ROWS` 上限、代码注明"与 lis-cases /import 一致"，属**已知并存边界**，非本线退役范围。

## 3.5 · 补充发现（讨论中·2026-07-03 追加）：抗体每片成本 vs 物料/进销存 数据层重叠

> **来源**：PM 追问"抗体不就是一种物料吗？不该直接算库存吗？改改字段不就行？"——核实后这是一对**线-3 首轮只审页面/路由层、未覆盖到的数据层重叠**。以下为核实结论 + 方案 + 待讨论口径。**本节讨论中，随讨论更新。**

### 核实到的事实（均带 file:line）

1. **抗体的价确实存了两处、维度不同**：`materials` 表（`DatabaseManager.ts:96`）里有 **98 行**名字带"抗体"的物料（demo seed `seed-pathology-data.ts:586+`），存**元/瓶**（¥280–2800）；`antibodies` 台账（`DatabaseManager.ts:1171`）存**元/片**（Ki-67 ¥4.26/片）。同名、不同单位、**当前 demo 数据层重复**（成本引擎不读 materials 里的抗体，故暂未"双算"，属埋着的隐患）。
2. **"物料单价 × BOM每样本用量 = 材料成本"这条链已存在**：`bom-v1.1.ts:58` 已算 `Σ(price × usagePerSample)`；更完整的 `cost-calculator.ts:580 calculateMaterialCost` 已实现 `价 × 用量 × 片数`，且**取价优先用入库批次加权价 `batches.inbound_price`**（即"跟真实采购走"），服务 ABC / 出库成本快照。
3. **抗体没走这条链**：`antibodies` 表与 `materials`/`bom_items` **零 JOIN**，抗体每片成本走独立台账（`antibody-cost.ts`）；且 `computeFullSlideCost` **不 join 任何收入/毛利/P&L**——逐抗体成本当前是**纯孤岛，喂不到医院盈亏看板**。
4. **`materials` 装不下抗体成本口径**：缺 3 字段——换算率（一瓶做几片）、剂型（原液/即用，价差约 6 倍，需并入唯一键）、每片用量/每片价；且抗体真价差约 **344 倍**（¥0.29–99.82），必须逐抗体取价、不能用单一 `materials.price`。
5. **台账真正独有、库存表达不了的加工层有 5 块**：二抗/显色（`detection_systems`）、工时/设备（`ihc_cost_params`，**G2 估·待校准**）、剂型双价、LIS 抗体名↔台账名映射（`antibody_aliases`，且被**对账域复用**做返工/多病灶初判）、G2 校准状态机。真正"纯重复"仅 4 列基础主数据（供应商/瓶价/规格/分类）。

### 三档方案

| 方案 | 做法 | 好 | 坏 |
|---|---|---|---|
| **A 抗体并入物料** | 抗体录进 `materials` + 加换算率/剂型/每片用量字段，每片成本走 库存×BOM | 价单一来源、自动跟采购 | 每个 IHC 项目都要建 BOM 挂 191 种抗体（维护极重）、剂型拆双物料丢 `UNIQUE(name,form)` 语义、5 块加工层一件不省还要重新接线、牵动对账线——独立复核判为**"净负收益的伪合并"** |
| **B 维持台账 + 补前端** | 就给现有独立子系统做页面（后端零改动） | 最快、后端零改 | 价仍手工、仍孤岛、双份价隐患未解 |
| **C 台账留、价改为自动派生**（初步倾向·讨论中） | 台账保留（换算/剂型/二抗/工时/LIS 映射有价值），但把每片价改为**引用物料/采购真实价 ÷ 换算率自动算**，并把 `computeFullSlideCost` 接进盈亏看板 | 满足"价来自真实采购"的正确直觉、单一价源、消孤岛；且"采购价→每片成本"引擎（`calculateMaterialCost`）**已存在**，非从零建 | 要做一次"台账↔物料/采购"对接接线；需先想清标准成本 vs 实际成本口径（见下） |

### 待讨论/待 PM 拍的**业务意图**（技术路取决于此）

- **① 你要的是"标准成本"还是"实际成本"？** 走库存批次价=**实际成本**（跟真实采购、会随每批采购价跳动）；台账手工值=**标准成本**（稳定、可控）。这决定"两处存价"到底是 bug 还是刻意的标准/实际双轨。
- **② 每片成本要不要进医院盈亏看板？** 现在 `computeFullSlideCost` 不喂任何 P&L——逐抗体成本的初衷若是"把准成本喂进盈亏"，那**接通（接到任意盈亏口径）才是第一优先**，比台账-vs-物料之争更急。
- **③ 谁维护价、多久一次？** 财务定期重定标准价 vs 仓库采购自动流入——组织/流程口径决定架构。
- **④ 精度优先级摆对了吗？** 一张 Ki-67 片 ≈ 一抗¥4.26 + 二抗¥15 + 工时¥8 + 设备¥3；**工时/设备这 ¥11 目前是 G2 粗估**。对便宜抗体，粗估的工时/设备远大于一抗价——**先校准工时/设备**可能比"把抗体价接到批次价"对准成本更值。
