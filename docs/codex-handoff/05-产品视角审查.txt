# 审查框架（逐条，结合真实素材回答）
1. **JTBD / 问题契合**：财务每月要完成的真实工作是什么？这个导入器是否真的减负、可信、可被采纳？
   （adoption-first：再正确若难用/看不懂/不敢信，就会被弃用）
2. **与真实素材对齐**：7 模板 / 19 院的真实多样性，现实现覆盖到哪、漏在哪？逐项明细 / 类别汇总 / 会诊远程 /
   诊断服务费 / 服务费混合 / 外送明细 / 共建——逐个判断哪些走得通、哪些会卡、哪些口径会错。
   （重点：宽表的远程/诊断服务费现"待人工归类"够不够用？类别汇总只有数字怎么进盈亏？纯外送院/纯远程院的页面体验？）
3. **财务可信度**：实验室收入=Σ(IN结算) 的口径在真实数据上站得住吗？体检卡四项（识别率/对账闭合/病例匹配/黄金值）
   足以让财务"敢据此入库、敢汇报院级盈亏"吗？哪里会让财务**不信任**或**误判**？
4. **系统边界正确性**：转销(NGS/FISH)/远程/共建移出、实物工序计入——边界划得对吗？有无现实场景被划错或划漏
   （如同一院同类服务有时自己做有时外送、共建院里的线下会诊、月度 retainer）？
5. **工作流与采纳**：建档(配置页) → 测试台 → 月度向导 的流程，对真实财务是否顺手？
   未匹配内联归类 / 设导入基线 / 409 缺口确认 / 跨月切换 / 改了规则要不要重算历史——这些闭环真闭合吗？哪步会卡死或被绕开？
6. **完整度诚实呈现**：估算 vs 已对账、待人工归类、待核算成本——这些"诚实信号"是产品优势，还是会被财务当成"这系统做不出来"？
   呈现方式（看板"估算 N 例"、测试台体检卡）够不够让人安心？
7. **缺什么才能真上线 / 被采纳**：产品上还缺哪些关键能力（批量导入、审计留痕、对账差异报表、多院区、
   追溯重算入口、权限/治理、与 LIS/财务系统的衔接、异常与回滚体验…）？按对采纳的影响排序。
