# 配置驱动对账单导入器 — 产品目的与真实素材说明（给 codex 产品视角审查）

> 这份文档是「产品视角审查」（05 提示词）的必读上下文。目的是让你不只盯代码，而能判断
> **这个功能是否真能解决它要解决的业务问题、跟真实素材对不对得上、财务用户会不会用起来**。

---

## 1. 公司与系统在做什么

- **COREONE** = 面向**病理免疫组化/特殊染色**领域的**进销存 + 单张切片成本控制系统（PSI）**。
- 业务实体是一家**第三方病理实验室**：给**多家医院**（19+ 家，分布各地）做病理检测的「实物工序」——
  取材、制片、切片、染色（HE / 免疫组化 IHC / 特殊染色）、出报告等。
- 每月，实验室与各医院之间按**对账单**结算。实验室想知道：**每一家合作医院到底赚不赚钱（院级盈亏）。**

## 2. 这个功能要解决的核心问题

财务每月收到**各家医院格式各异的对账单**，需要把它们变成「每家医院的实验室收入」，再结合
**ABC 成本**（作业成本法核算的单 case 成本）算出**院级毛利**，发现亏损医院/亏损病例。

难点（也是这个功能存在的理由）：
1. **对账单格式杂**：真实是 **7 个模板家族 / 19 家医院**，列名、结构、口径各不相同（见下「真实素材」）。
2. **不是所有收入都算「实验室收入」**：实验室只核算**自己做了实物工序的那部分**；外送转销、远程诊断、
   共建分成等要**移出**（否则"越做越大"，混入不属于实验室工序的钱）。哪些计入/移出，**逐院可能不同**。
3. **收入口径**：实验室收入 = **Σ(计入实验室的服务项「结算金额」)**；结算金额 = **医院收费(开单) × 扣率**。
4. **数据不完整 + 判断逻辑要持续纠正**：账单到得有早有晚、有的院只给汇总数字、有的服务边界要反复校正。

## 3. 用户与权限

- 主用户 = **财务 / 管理员**（配置页、导入测试台、月度向导都标「仅财务/管理员」）。
- 财务的真实月度工作：拿到各院对账单 → 导入 → 核对 → 入库 → 看院级盈亏。**这是高频、重复、要"敢信"的工作。**

## 4. 业务口径与系统边界（务必据此判对错）

- **实验室收入 = Σ(计入实验室的服务项结算金额)**；结算 = 开单 × 扣率（扣率即"结算/开单"，0–1）。
- **计入实验室（IN）**：组织学、细胞/宫颈 TCT、院内冰冻、线下外院会诊（送蜡块/切片来、我们加做 IHC/分子）。
- **移出（OUT，交"公司级运营系统"，不进实验室盈亏）**：外送分子（NGS/FISH/HPV-E6E7）、远程诊断
  （含月度固定保底费 retainer）、共建分成净额、科研。
- **判据 = 有没有实物工序**（切片/白片/染色/制片）。远程诊断（数字云端、无实物）≠ 线下会诊（实物加做=IN）。
- **逐院配置驱动**：每家医院一份版本化配置（业务线 + 识别规则 + 计入/移出 + 三级扣率 + 模板/列映射 +
  计税口径 + 特殊结算）。同一类服务在不同院可能计入/移出不同，所以"逐院"是核心，不能写死全局。

## 5. 增量纠错架构（这个产品的世界观，重要）

因为"数据不完整 + 周期长 + 判断逻辑需持续纠正"，系统刻意采用：
- **原始不可变 / 派生可重算 / 推断可覆盖且留痕**；
- **真实 case 黄金集**做锚（和睦家 2026 第 4 周 25 个 case 全组织学 → 实验室收入 **¥13,152** @扣率 0.8303，
  全程口径不能算错这个数）；
- **完整度显式**：算不出来的就显式标「估算 N 例 / 待人工归类 / 待核算成本」，**不假装精确、不静默**；
- **版本化追溯**：改了规则能判影响面、能回滚、能按版本重算；
- **对账差异 = 纠错信号**：逐行结算 Σ vs 对账单独立声明的合计行，差了就说明漏读/口径错。

> ⚠️ 这意味着「诚实标注估算/待归类」是**设计哲学**，不是 bug。但产品上要判断：财务会把它当"靠谱的诚实"，
> 还是当"这系统做不出来"？这正是要你审的。

## 6. adoption-first（采纳优先）原则

本项目反复强调：**再正确的功能，如果财务日常用起来负担过重、看不懂、不敢信，就会被绕开或弃用**。
所以产品审查的最高标准不是"技术正确"，而是"**财务会不会真的每月用它**"。

## 7. 真实素材（请实际打开看）

- **真实对账单样本（PII 已脱敏）**：`后端代码/server/tests/fixtures/statements/*.json`（**7 模板家族各一**）：
  - `out_line_item__hemujia`（逐项明细，和睦家；最规整，黄金锚来源）
  - `out_category_summary__dongan`（类别汇总，东安；只有类别小计，无逐 case）
  - `out_consult_remote__pingquan`（会诊远程，平泉；**宽表**：远程会诊结算/免组结算等列语义在表头）
  - `out_diagnostic_fee__ningbo`（诊断服务费，宁波；宽表）
  - `out_service_fee_mixed__wenzhou`（服务费混合，温州中心；含分配率）
  - `out_outsourced_detail__ganzhou`（外送明细，赣州；纯外送=实验室收入可能为 0）
  - `out_joint_venture__shimen`（科室共建，石门；共建利润分成，结构异类）
- **收费目录**：`后端代码/server/src/utils/charge-catalog.ts`（收费码）、`ngs-catalog.ts`（NGS 产品参考价）。
- 现实规模参照（背景，不在仓库）：某月 LIS 导出约 **1454 case / 22 院**（LIS 给的是数量列、不是项目名）；
  对账单一批 **19 院 60 文件**；收费目录 **81 码 → 33 项**。
- **真实数据洞察**：约一半结算金额其实在"实验室工序"之外（外送分子/共建/远程）——印证"移出"的必要；
  各院 IN 占比差异极大（和睦家 ~98% vs 温州中心 ~12% 几乎全分子 vs 赣州 0% 纯外送 vs 平泉纯远程）。

## 8. 这个功能 = 三件套（产品形态）

1. **后端分类引擎**：对账单网格 → 按逐院配置解析 → 分类（计入/移出/未匹配/歧义）→ 逐行结算 →
   Σ(IN结算)=实验室收入 → 落库 case_revenue → 院级盈亏看板。
2. **导入测试台（admin/财务）**：选院 → 上传样表 → **体检卡四项**（识别率 / 对账闭合 / 病例匹配 / 黄金值）
   → 逐线拆分 → 未匹配行**内联归类**（写回该院配置、立即生效）→ 设为月度导入基线。
3. **财务月度导入向导**：三步（上传 → 预览核对 → 入库）；未匹配/不平 → 409 门禁要财务确认才落库；入库后跳看板。

## 9. 已知的诚实边界 / 约束（你可质疑这些够不够）

- **宽表模板（远程会诊/诊断服务费）**：业务语义在列表头、逐行项目名为空；当前**保守落「待人工归类」**+
  显式告警（不自动按列展开，避免把"远程会诊=移出"误判成"线下会诊=计入"而虚增实验室收入）。全自动列展开=后续功能。
- **院内冰冻 / 多重染色**：真实生效需 LIS 补列或对账单 v2 账单码。
- **NGS**：外购转销，独立渠道（不在对账单），需独立订单样例才能跑真实数据。
- **类别汇总 / 共建利润表**：无逐 case，走专用解析，不做逐 case 落库。
