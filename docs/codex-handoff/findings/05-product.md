# 段⑤ 产品视角审查

产品判断：现在已经证明“规整逐项明细”可以跑通，但离财务每月真正用起来还差一个核心闭环：非逐 case / 纯外送 / 共建等真实模板要能入库并诚实进看板，宽表要有列级归类上下文，月结要有批量工作台、重算追溯和审计。

## 【阻断采纳】

1. 多个真实模板只能预览，不能完成“入库 → 院级盈亏”的主任务。

   为什么是产品问题：真实素材不是 7 个里只有和睦家逐项明细重要。`out_category_summary__dongan_2601.json` 只有类别小计，合计结算额 ¥121,016.90，里面同时有常规病理/IHC/EBER/TCT 这类 IN，也有 HPV/基因/FISH 这类 OUT；`out_joint_venture__shimen_2603.json` 是科室共建利润表，医院收入总额和利润按科室给；`out_outsourced_detail__ganzhou.json` 是纯外送，合计结算额 ¥40,219.20，没有病理号。现实现对 category_summary / joint_venture 只返回 `note`，月度向导直接显示“暂不支持逐 case 入库”；纯外送因为没有病理号，commit 会走到“无可逐 case 落库的明细行”。这意味着一批真实医院会从院级盈亏里消失，财务必须线下维护另一套表，系统就不会成为月结主流程。

   建议方向：增加“月度对账事实”层，不强行要求每张账单都逐 case。对 category_summary 支持类别→业务线(IN/OUT)映射并写入 partner-month aggregate；对 pure OUT 账单允许入库为 `labRevenue=0 / outSettle=全额 / 无 case 成本`，用于月结完整性和公司级转销交接；对 joint_venture 建专用共建分成事实表，进入看板时明确“不参与实验室工序毛利”或单列展示。

2. 宽表“待人工归类”诚实但不可操作，财务看不到该按什么归。

   为什么是产品问题：平泉样本的关键业务语义在列头：`远程会诊结算`、`免组结算金额`、`结算合计`，行内项目名为空，病理号还是纯数字；宁波样本也是 `基础诊断费用`、`免疫组化结算`、`病理医生进修学习` 这类列语义。当前测试台的待归类卡只展示“（无项目名）/ 病理号 / 金额”，不带来源列、诊断方式、候选业务线或原始行片段。对平泉这种纯远程院，用户既不能一键把“远程会诊结算”归到 OUT，也不能确认“免组结算”是否 IN；最后很可能只能 `confirm:true` 入库为 0 实验室收入，或乱加一个前缀规则。

   建议方向：解析宽表时把每个金额来源列展开为带 `sourceColumn/sourceLabel` 的明细，测试台按“列”归类而不是只按“行”归类；提供候选规则如“远程会诊结算 → 远程诊断 OUT”“免组结算金额 → 线下会诊/IHC IN”，由财务确认后写入该院配置。保守不自动入库是对的，但必须给财务足够上下文完成判断。

3. 新医院建档的第一条内联归类会卡死，破坏 adoption-first 的首用体验。

   为什么是产品问题：真实上线时 19 家医院不会一次性配置好。财务的自然动作是“选一家新医院 → 上传样表 → 对未匹配行归类 → 保存为该院规则”。但 preview 对无配置医院返回 v0，前端 classify-rule 带 `expectedVersion: 0`，后端先 seed v1 再保存会 409。用户看到“配置已被更新，请基于最新预览重试”，但这不是他人并发修改，而是首用路径自撞。第一张样表归类失败，会让财务认为这个建档流程不可靠。

   建议方向：把“首张样表建档”做成显式流程：preview v0 后允许第一条规则直接创建 v1/v2；或者在用户选择新医院时先创建默认配置，再预览。错误文案也要区分“首次建档初始化”与“他人并发修改”。

4. 外购转销和独立渠道还没形成一个财务可用的闭环。

   为什么是产品问题：产品边界明确说 NGS/FISH/HPV-E6E7、远程、共建等 OUT 不进实验室工序收入，但它们仍是财务月结事实。当前对账单里的 OUT 行有的不能入库（赣州纯外送），独立 NGS 后端有订单 API，但前端没有 NGS 导入/预览入口，月度向导也不提示“这张账单应转到外购转销台账”。看板类型里有 NGS 毛利字段，但财务没有完整路径把真实订单、安全成本、缺价缺成本确认和月结状态串起来。

   建议方向：把 OUT 做成一等公民：月度工作台里分“实验室收入”“外购转销/远程/共建移出”“未归属”，外购转销有独立导入入口、缺成本治理、和公司级运营系统交接状态。实验室 P&L 可以排除 OUT，但月结不能丢 OUT。

## 【重要缺口】

1. 缺少“月结工作台”，单院单文件流程撑不起 19 院 60 文件。

   为什么是产品问题：真实背景是一批 19 院 60 文件，而现在产品形态是选一家医院、上传一个文件、预览、入库。财务每月需要知道哪些医院已收到文件、哪些已预览、哪些有未匹配、哪些已确认入库、哪些成本未接通、哪些可以关账。没有批量状态页，财务只能靠记忆和文件夹管理月结进度。

   建议方向：增加 `YYYY-MM` 月结工作台：医院列表按状态分组（未收单、待归类、待确认、已入库、成本待核、已关账），支持拖入多文件自动识别医院/模板，批量预览但逐院确认，输出月结差异报表。

2. 规则版本和导入基线有了，但缺“影响范围 + 历史重算”。

   为什么是产品问题：逐院配置是会持续纠错的。财务改了“HPV-E6E7 属于 OUT”或把某宽表列从未匹配改成远程 OUT 后，最关心的是“哪些已入库月份受影响、差额多少、是否要重算看板”。当前配置页有版本、回滚、设基线，case_revenue 也存 `config_version`，但没有影响预览、重算入口、关账月调整策略。没有这一步，财务要么不敢改规则，要么改完也不知道历史报表是否已经同步。

   建议方向：保存配置前后生成影响评估：按历史 import_batch 重新 dry-run，列出受影响月份/医院/金额差；未关账月可一键重算，已关账月生成调整单。基线应绑定“从哪个账期开始生效”，而不是只标一个当前版本。

3. 体检卡里的“黄金值”没有产品化，信任锚无法被普通财务使用。

   为什么是产品问题：黄金值是让财务敢信的关键，但前端没有输入或维护 `goldenExpected` 的位置，测试台和月度向导也不会把外部期望实收带给 preview。结果体检卡大多数时候只显示“未录入期望实收”，黄金锚停留在测试代码里，没进入真实工作流。

   建议方向：在导入测试台/月度向导增加“财务期望实收/账单声明实验室收入”字段，支持从文件合计自动带入或人工录入；按医院+账期保存期望值和来源，体检卡显示差异、阈值、确认人。

4. 病例匹配依赖 LIS，但缺少“LIS 未导入/晚到/错院”的解决路径。

   为什么是产品问题：病例匹配是四项体检之一，但真实世界 LIS 约 1454 case / 22 院，可能晚于对账单、没有登记月、医院名别名不同。当前导入页只显示命中/缺失计数，用户看不到缺哪些 case、应该去哪里补 LIS、是否只是“本月 LIS 未导入”。当体检卡因为 LIS 缺失变成 todo 时，财务会不清楚是账单问题、LIS 问题还是系统配置问题。

   建议方向：把病例匹配做成可恢复状态：`无本期 LIS 数据`、`部分缺失`、`错院/别名疑似` 分开；提供缺失 case 列表导出、跳转 LIS 导入、医院别名匹配建议；月结工作台允许“先按收入入库、标记成本/病例待补”。

5. 默认业务线目录漏掉真实 OUT 项，导致常见外送项反复未匹配。

   为什么是产品问题：温州样本里 HPV-E6E7 多行，产品说明也明确 HPV-E6E7 属于 OUT；默认配置有 NGS、FISH、远程、共建，但没有 HPV/E6E7 关键词。上线初期如果每家医院都要手工补这些显然常见的 OUT 词，财务会觉得系统“不懂业务”。更糟的是这些未匹配金额会阻塞入库或被确认后丢进未识别。

   建议方向：把默认业务线从“8 条示例线”升级为可维护的服务目录：NGS/FISH/HPV/STR/远程/共建等 OUT 项预置同义词和前缀，允许逐院启停/覆盖；测试台从未匹配项中推荐“加入全局目录”或“仅该院生效”。

## 【体验问题】

1. `confirm:true` 是高风险动作，但产品没有要求原因、影响摘要和后续待办。

   为什么是产品问题：月度向导会告诉财务“确认后照常入库，未识别金额不计入实验室收入”。这确实能避免卡死，但它也把“未识别多少金额、为什么允许入库、谁确认”变成一次按钮点击。财务审计时需要解释为什么某院某月少计了实验室收入，当前没有确认理由、附件、待办或二次复核。

   建议方向：确认入库弹层必须展示未匹配/歧义金额、对账差额、无合计行原因，并要求选择原因（待补规则/纯外送/暂按 0/其他）和备注；入库后在看板/工作台保留“待复核”状态，不能只显示已入库。

2. “诚实信号”还不够可行动。

   为什么是产品问题：看板显示“估算 N 例”“未接通”是正确方向，但财务下一步不知道该做什么。估算来自未对账、缺 LIS 数量、成本未回填还是 NGS 缺成本？不同原因对应不同人处理。只有标签没有 drill-down 和责任人，会被理解成“系统做不出来”。

   建议方向：每个诚实信号都加下一步：估算 N 例可点开 case 清单，未接通成本跳成本回填/重算，待归类跳测试台对应批次，NGS 未核跳订单补成本。月结工作台按责任角色分配待办。

3. 建档流程对财务来说仍偏“配置表”，缺少样表驱动的引导。

   为什么是产品问题：合作医院配置页有 6 个 tab 和大量字段，但财务最自然的建档方式是上传该院样表，让系统识别模板、列映射、业务线候选和扣率，再逐步确认。现在测试台和配置页是两个页面，规则能写回，但列映射、计税口径、特殊结算、基线生效期等仍需要用户理解系统模型。

   建议方向：做“新医院建档向导”：上传样表 → 自动识别模板/列映射 → 显示 IN/OUT 候选 → 设置扣率/特殊结算 → 跑黄金/体检卡 → 设首个基线。配置页保留为高级维护，不作为首用入口。

4. 类别汇总和共建的预览结果对用户不可读。

   为什么是产品问题：后端已经能返回 `parsed.categories` 或 `parsed.depts`，但前端遇到 `preview.note` 只显示一行“该模板暂不支持逐 case 入库”。财务看不到东安每个类别、石门每个科室、金额是否对平、哪些应计入/移出，也无法把它作为核对工具。

   建议方向：即使暂不支持入库，也要展示专用预览表：类别/科室、收费、结算、建议 IN/OUT、声明合计、差额、不可入库原因和下一步入口。否则“预览”对这些模板没有实际价值。

## 【增强建议】

1. 建立模板库和医院样本库。

   每个模板家族保留真实样本、识别规则、常见列映射、已知风险、上次通过日期；新文件识别后展示“匹配模板：温州服务费混合，置信度 92%，上次成功入库 2026-04”。这会显著提升财务信任。

2. 输出月结对账差异包。

   每月关账前生成 Excel/PDF：各院账单合计、实验室收入、移出金额、未匹配金额、成本接通率、毛利、确认人、待办。财务汇报和复核需要这个产物，而不只是看网页。

3. 把配置和导入纳入 maker-checker。

   大额规则变更、把未识别金额确认入库、把 OUT 改 IN，应由财务提交、管理员或另一财务复核；否则系统虽然有版本历史，但治理强度不够支撑真实财务口径。

4. 把“公司级 OUT 去向”作为产品接口定义清楚。

   实验室 P&L 排除 OUT 是正确的，但 OUT 不能在系统里消失。建议定义 OUT 台账 API/导出：外购转销、远程、共建、科研分别给公司运营系统、财务总账或手工复核，形成闭环。

5. 对外购转销建立成本完整度评分。

   NGS/FISH/HPV 等 OUT 项如果未来进入总毛利，需要像体检卡一样有订单号、售价、外包成本、月份、医院匹配、缺成本确认。没有完整度评分时，NGS 毛利会比实验室收入更容易被误信。
