# 路线图 v2 启发与优化建议

基于 `docs/COREONE-配置驱动导入器-产品路线图-2026-06-29.md` 合并终稿 v2。

## 1. 有没有启发

有，而且比较关键。v2 最大的价值不是补了更多功能项，而是把之前“产品想做什么”和“工程该先修什么”之间的摇摆收束成了可排期的决策。

### 1.1 从功能清单变成了可执行路线

原路线图 v1 更像问题地图：列出 A-H Epic、阻断点、体验问题和技术债。v2 进一步把北极星、成功标准、数据模型、Phase 0/1/2/3 和验收包合并到一张执行图里，已经可以拿来拆任务。

尤其是“Phase 0 正确性修复不依赖产品 Epic”这个切法有启发。它避免了一个常见陷阱：因为未来要重构月结事实层，所以现在污染数据的串账、缺成本写 0、配置读回滚不归一先不管。v2 的判断更稳：这些是当前就会污染可信度的问题，应该先修。

### 1.2 四个用户决策让方案从讨论态进入建设态

v2 锁定了四个关键口径：

- 病例身份采用全链路复合键 `(partner_id, case_no)`。
- 先修正确性子集，而不是等所有产品 Epic 落地。
- 联合运营按逐院 `special.joint` 配置决定是否进 P&L 主表。
- 已关账月份只出调整单，未关账月份可重算。

这四点把几个最容易反复争论的口径压住了。没有这些决策，后续会在数据模型、迁移、报表口径、测试用例上持续返工。

### 1.3 “缺失不等于 0”被提升成产品原则

v2 明确把 NGS 缺成本、缺价、缺 LIS、缺规则统一为质量标记，而不是按 0 静默入账。这一点很重要，因为它不是技术细节，而是财务可信度原则。

这个原则还能外推到更多地方：

- 未匹配 LIS 不等于没有病例。
- 缺成本不等于毛利为收入。
- 纯 OUT 不等于金额消失。
- 未归类不等于应收为 0。

后续所有看板、导出、P&L 都应该沿用这个表达方式。

### 1.4 OUT 被正式定义为月结事实

v2 把 OUT 从“实验室收入之外的东西”提升为“一等公民台账”。这能解决真实月结最大的断点：财务不是只关心实验室劳务收入，还要知道外送、远程、共建、转销是否已登记、是否有成本、是否已交接。

这个判断对产品形态有直接启发：COREONE 的价值不是一个导入器，而是病理实验室月结 sub-ledger。导入只是入口，真正的产品是“收入、外送、成本、调整、关账”的闭环。

### 1.5 3 样本验收包是正确的 MVP 抓手

v2 选择东安科目汇总、赣州纯外送、平泉宽表远程会诊作为 Phase 1 验收包，是合理的。它们分别覆盖：

- 非逐病例聚合事实。
- 无病例号纯 OUT。
- 宽表列级语义。

这三个样本能快速验证新架构是否真的脱离了“所有东西必须逐 case”的旧模型。

## 2. 有没有更优的方案

有。v2 已经是可执行版本，但我建议进一步优化两个层面：交付切法和数据模型形态。

我的结论是：不要把 Phase 1 做成“多个 Epic 并行建设”，而应该做成“一个最小可关账月”的纵向切片；数据层不要一上来过早分裂成多张业务事实表，而应先建立统一的对账单规范行，再派生各类账本。

## 3. 更优方案一：以“最小可关账月”替代“功能 Epic 并行”

v2 的 A-H Epic 是完整的，但如果直接按 Epic 并行拆，会有两个风险：

- 每个 Epic 都能单独完成一些功能，但财务仍跑不完一个月结闭环。
- 月结工作台、OUT 台账、宽表归类、聚合事实层互相依赖，拆太散会导致验收口径漂移。

更优切法是按一个完整月结闭环交付。

### 3.1 推荐 Phase 1A：最小可关账月

目标：用 3 个样本跑通一个最小月结闭环，而不是先做完所有模板能力。

范围：

- 创建月份工作台基础壳：月份、医院、文件、状态、金额汇总。
- 支持 3 个验收样本：东安、赣州、平泉。
- 每个文件都进入统一批次表，有原始文件哈希、模板版本、规则版本。
- 东安进入聚合收入事实。
- 赣州进入 OUT 台账，`labRevenue = 0`，`outSettle = 全额`。
- 平泉保留来源列，能按列确认规则。
- 所有缺成本、缺 LIS、缺规则进入质量标记。
- 输出一份月结差异包：IN、OUT、未归类、成本未就绪、人工确认。

验收方式：

- 财务能从月结工作台看到 3 个文件各自状态。
- 能解释每一笔金额为什么进 IN、OUT 或待处理。
- 能导出一份可对账的月结包。
- 不要求覆盖全部 7 模板，也不要求复杂 maker-checker。

### 3.2 推荐 Phase 1B：模板和规则扩展

目标：在最小月结闭环成立后，扩展覆盖其他模板。

范围：

- 温州混合服务费。
- 宁波宽表诊断费。
- 石门联合运营 `special.joint`。
- 默认服务目录补齐 HPV-E6E7/FISH/NGS/远程。
- 模板化预览组件。
- confirm 原因、影响摘要、审计。

这个顺序比“先做完整工作台 + 完整事实层 + 所有模板”更稳。它先证明系统能关一个小账，再扩面。

## 4. 更优方案二：先建统一规范行，再派生业务账本

v2 的数据模型提出了 `statement_case_revenue`、`statement_aggregate_revenue`、`statement_out_ledger`、`statement_quality_flags`。方向是对的，但建议不要一开始就把业务事实完全分裂。

更优的数据形态是两层：

### 4.1 第一层：不可变导入与规范行

建立统一的规范化对账单行：

- `statement_import_batches`
- `statement_raw_rows`
- `statement_normalized_lines`

`statement_normalized_lines` 是核心。它不直接代表最终 P&L，而是代表“系统理解后的对账单金额行”。

建议字段包括：

- `batch_id`
- `partner_id`
- `service_month`
- `case_no`
- `source_sheet`
- `source_row`
- `source_column`
- `source_label`
- `template_family`
- `line_grain`: `case` / `aggregate` / `out` / `joint` / `adjustment`
- `business_line`: `IN` / `OUT` / `UNKNOWN`
- `service_code`
- `amount_role`: `gross` / `settlement` / `discount` / `cost` / `adjustment`
- `amount`
- `classification_status`
- `rule_version`
- `quality_state`

好处：

- 宽表列语义、科目汇总、纯外送都能用同一层承载。
- 后续派生多个账本时不会丢原始上下文。
- 规则重算可以重新派生账本，而不是改原始理解层。
- 新模板接入时先落规范行，再决定是否能入病例收入、聚合收入或 OUT 台账。

### 4.2 第二层：派生账本与快照

从规范行派生财务可用账本：

- `case_revenue_ledger`
- `partner_month_revenue_ledger`
- `out_settlement_ledger`
- `joint_operation_ledger`
- `quality_flags`
- `pnl_snapshots`
- `adjustment_entries`

这层才服务 P&L、月结工作台和导出。

这样做比直接把导入结果写进多张最终事实表更稳，原因是：真实对账单经常先处于“部分理解、部分待分类”的状态。统一规范行能承接中间态，而最终账本只承接已确认或可解释的结果。

## 5. 更优方案三：把月结状态机前置

v2 已经说每月每院每文件要有状态，但还可以进一步产品化为状态机。

建议状态：

- `uploaded`: 已上传。
- `parsed`: 已解析。
- `classified`: 已归类。
- `review_required`: 待复核。
- `posted`: 已入账。
- `cost_pending`: 成本待补。
- `ready_to_close`: 可关账。
- `closed`: 已关账。
- `adjusted`: 已出调整。

状态机要绑定两个维度：

- 文件/批次状态：这份文件走到哪一步。
- 月份/医院状态：这个医院这个月是否可关账。

这比单纯做多个列表更有利于财务使用，因为财务月结最关心的是“还差什么不能关账”。

## 6. 更优方案四：把质量标记做成“阻断等级 + 责任人”

v2 的 `statement_quality_flags` 很重要，但如果只是一张 flags 表，容易变成没人处理的告警池。

建议每个质量标记至少有：

- `severity`: blocking / warning / info。
- `owner_role`: 财务 / 实施 / 成本 / 系统管理员。
- `resolution_action`: 补 LIS / 补成本 / 建规则 / 人工确认 / 忽略并说明。
- `blocks_posting`: 是否阻断入账。
- `blocks_closing`: 是否阻断关账。
- `resolved_by`
- `resolved_at`
- `resolution_note`

这样月结工作台就能回答三个关键问题：

- 这条问题是否影响入账？
- 是否影响关账？
- 谁该处理？

## 7. 更优方案五：补一棵指标树，避免路线图只看交付

v2 现在已经有成功标准，但还可以补成可量化指标。

建议核心指标：

- 月结完成时长：从首个文件上传到月份关账的自然日。
- 文件闭环率：已上传文件中进入 posted/closed/adjusted 的比例。
- 金额解释率：可归因到 IN/OUT/调整/未匹配/成本待补的金额占总金额比例。
- 自动分类率：无需人工规则新增即可完成分类的金额比例。
- 人工触达率：每万元结算金额需要人工处理的次数。
- 成本就绪率：已入账收入中成本已接通金额比例。
- 规则变更影响可解释率：规则变更后能给出影响批次和金额的比例。

有了这些指标，路线图才能从“做了哪些功能”转成“月结是否真的更快、更可信”。

## 8. 我建议的最终路线

### Phase 0：可信度止血

维持 v2 不变：

- 全链路 `(partner_id, case_no)`。
- 配置读/回滚归一。
- NGS 缺成本/售价质量标记。
- 黄金值不回退。

### Phase 1A：最小可关账月

新增重点：

- 统一规范行 `statement_normalized_lines`。
- 月结状态机基础版。
- 东安、赣州、平泉 3 样本端到端。
- 最小月结差异包。

### Phase 1B：模板扩展与确认体验

扩展：

- 温州、宁波、石门。
- 默认服务目录。
- 模板化预览。
- confirm 原因、审计、影响摘要。

### Phase 2：治理与关账制度

补齐：

- 规则影响预估。
- 未关账重算、已关账调整单。
- maker-checker。
- LIS 匹配恢复。
- 黄金值产品化。

### Phase 3：规模化与自动化

扩展：

- 批量识别。
- 样本库。
- NGS 前端入口。
- 公司级 OUT 接口。
- 风险评分和任务分派。

## 9. 需要确认的问题

我目前不需要阻塞性澄清，但有 4 个问题建议在进入 Phase 1A 前确认：

1. Phase 1 的验收目标是“选 3 个样本跑通一个可关账月”，还是“7 类模板都能有初步入口”？我建议前者。
2. OUT 台账在 Phase 1 是否只做 COREONE 内部 sub-ledger，不对接外部系统？我建议先内部闭环。
3. `statement_normalized_lines` 这种统一规范行是否可以作为事实层前置设计？我建议加，否则后续模板接入会不断分叉。
4. 月结工作台是否以“能不能关账”为主，而不是以“导入器任务列表”为主？我建议直接围绕关账状态设计。

## 10. 结论

v2 是一个明显更成熟的终稿：它把方向、原则、决策、数据模型和排期连起来了，可以作为工程排期基线。

我的更优建议不是推翻 v2，而是把它进一步收敛为“一个最小可关账月”的纵向切片，并在数据层先建立统一规范行，再派生病例收入、聚合收入、OUT 台账和 P&L 快照。这样能更快验证真实财务月结是否跑得通，也能降低后续模板扩展和规则重算的返工。
