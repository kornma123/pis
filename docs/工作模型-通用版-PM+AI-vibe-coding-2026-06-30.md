# 工作模型 · 通用版 · 非技术 PM + AI 做 vibe coding

> **版本**: 1.0（2026-06-30）| **性质**: 项目无关，新项目可直接套用 | **配套**: 项目版（各项目自己的实证+专项决策）
> **维护**: 活文档，随实践演进。见文末「演进机制」+「变更记录」。

---

## 0. 一句话

**意图无法靠文档（PRD/规格/代码）验证，对非技术 PM 只能靠"你判断得了的具体产物"验证。整个工作流 = 尽早把意图变成可判断的产物 + 逼 AI 摊开隐藏假设 + 用真实数据产出能对的答案 + 全程独立复核。而这一切的引擎是「讨论」。**

---

## 1. 模型（四段）

```
前置会  →  讨论 + 逼AI摊假设(preflight) + 拿真数据产出手核答案   ── 管「对不对」
实现    →  BDD/TDD 守着这些手核答案                             ── 讨论延伸进来，设计选择照样摊出来
建完    →  mockup / preview → 真人反应                          ── 管「喜不喜欢」(测试替不了)
全程    →  一道独立复核（不共享盲点的第二视角）
```

**核心认知**：讨论不止于前置会。实现段那些看着像"技术活"的选择，很多是产品/领域决策的伪装——照样讨论，不闷头 bake 进代码。

---

## 2. 为什么：防三层「我不知道我不知道」

| 层 | 是什么 | 谁看不见 | 哪段防它 |
|---|---|---|---|
| **UU-1 代码债** | 代码在悄悄烂/埋雷 | PM 看不见代码 | 独立复核 + 结果验证（最难解，只能缩小盲区） |
| **UU-2 AI 误读意图** | AI 自信地建了个连贯但错的东西 | AI 看不见自己错了 | **前置会：逼摊假设 + 拿真数据** |
| **UU-3 UX≠预期** | 界面/交互不是用户要的 | PM 也说不清、看到错的才知道对的 | 建完 → mockup/preview → 真人反应 |
| **UU-4 数据/权威错位** | 哪张表/哪个系统才是"事实源"没被制度化——对账单、LIS、国标、成本台账、配置各是**局部真相**，谁也没声明降级策略和重审触发 | 谁都看不见（每人都以为自己那张表是权威） | **显式声明**：每个口径写清 事实源 + 降级策略 + 重审触发条件（比"讨论更多"更底层） |

**真正的敌人不是"AI 撒谎或搞坏"，是"所有人自信地在错的东西上达成一致"。** 更工程化地说：危险来自**未声明的事实源、未声明的降级策略、未声明的重审触发**（UU-4）。四段模型不断引入"不共享盲点的视角"（真实数据、真实用户、独立复核）来破它。（来源：codex 09 复核——三层 UU 补第四层。）

## 3. 核心机制（可复用动作）

1. **逼 AI 摊假设 → 人确认「结果」不是「文字」**：动手前让 AI 说"我假设了什么/我不得不猜了什么/我可能在哪误解你"；AI 把候选规则**在真实数据上跑出结果**，PM 确认那个**结果**（"会把这几行标成X、数从A变B，对吗"），不是确认复述。
2. **读真数据，不猜**：边界/口径有歧义 → 读**行级真数据**裁决，不用 AI 常识猜。
3. **拿真数据产「手核答案」+ 守恒校验**：正确答案**独立**核出来（不能拿 AI 自己算的对自己），用一个**守恒等式**验证。防自证的假绿。
4. **验证靠结果不靠 AI 自述**：信确定性输出（测试 pass/fail、守恒差 0、你能常识核对的真数据），不信"我做好了"。
5. **独立复核贯穿**：每个发现/规则/修复，用**独立**视角（第二个 AI 换框架、对抗验证、或人）再核。
6. **诚实分级**：每个结论标 硬（真实数据）/ 估（多源交叉）/ 弱锚（必须实测校准），并明说**做不到**的。
7. **说人话**：面向 PM 一切用大白话，不用黑话；出稿前当 lint 自查。
8. **⭐ 讨论结论必须落成"会失败的断言"，否则等于没锁**：手核出的黄金锚，要变成 **CI 里会红的自动断言**（不是 `it.todo`、不是只打印的脚本、不是散落文档里的数字）。**最大盲点不是"讨论不够"，是"讨论后的结论没被机器约束"**——讨论把答案找出来了，但测试是 todo、旧口径仍全绿，实现者可以完全不实现也不被卡。配套 **golden registry**：每个 golden 一条登记（数据源 / 复现脚本 / 断言 / 是否 CI 红绿），见项目版 `docs/golden-registry.md`。（来源：codex 09——¥27,870 被讨论出来但红测试是 todo。）

## 4. 操作铁律

1. **每步产出立即落文档**——环境会崩、分支/worktree 会被清，只有 git 里的文档留得住。
2. **别漂移到 logistics**——"分支怎么合、用 A 还是 B"不是模型步骤；回到"拿真数据产出手核答案"。
3. **手核答案的正确性来自独立现实，不能自证**。
4. **小步**——同一场景别堆超过 3 个动作；一次锁一个口径。
5. **PM 是终审**——"看到错的才知道对的"，UX 那半边归真人，测试替不了。
6. **不确定就摊出来问，别自信地填空**——尤其在钱/关键逻辑的边界上。

## 5. 每段产出物 + PM 触发话术

| 段 | 产出物 | PM 话术 |
|---|---|---|
| 前置会·摊假设 | 假设清单 + 待确认口径 | "先别写码。把假设、你猜的地方、可能误解我的列出来，在真数据上跑个结果给我看。" |
| 前置会·手核答案 | 黄金锚 + 口径文档 | "用真实数据把正确答案核出来，加一个守恒校验，落成文档。" |
| 实现 | 会跑的测试（守 golden） | "按这个口径写测试再实现，跑绿给我看。" + 遇设计岔路"这个选择摊出来讨论。" |
| 建完 | mockup/真页面 + 真人走查 | "摆到真实用户面前，给他 3 个任务看卡哪。" |
| 全程 | 独立复核 verdict | "换个独立视角把这个再核一遍，挑算术和口径。" |

## 6. 适用场景矩阵（别让所有项目都被逼走重流程）

本模型的重心（真数据 / 守恒 / golden / 事实源声明）对**财务/数据/口径类**系统极强；对纯 UI/增长/内容/低风险内部工具则**过重**。按"钱味/数据味"选强度：

| 场景 | 前置会摊假设 | 手核黄金 + 守恒 | golden 落 CI 断言 | UU-4 事实源声明 | mockup 真人 | 独立复核 |
|---|---|---|---|---|---|---|
| **钱/口径/数据模型/关键逻辑**（如本项目收入口径） | 必 | 必 | 必 | 必 | 视有无 UI | 必 |
| **一般 CRUD / 内部工具** | 建议 | 有数值口径才要 | 有黄金才要 | 建议 | 建议 | 抽查 |
| **纯 UI / 样式 / 增长文案** | 轻 | 免 | 免 | 免 | **必**（这类靠真人不靠守恒） | 轻 |
| **琐碎修改 / 已验证重复动作** | 免 | 免 | 免 | 免 | 免 | 免 |

一句话：**越碰钱/数据越往上一行走（重证据链），越偏观感/低风险越往下走（重真人反应）**。别把守恒/golden 套到纯 UI，也别把纯样式流程套到收入口径。（来源：codex 09——通用版加"适用场景矩阵"。）

## 7. 演进机制（这是活文档，怎么让下个会话接住）

1. **入口固定**：项目 `CLAUDE.md` / `session-log` 指向本文档为"方法论权威"，每会话启动读到。
2. **谁改**：任何一次实践如果**修正/新增**了机制或铁律 → 在下方「变更记录」追一条 + 更新对应正文。
3. **通用 vs 项目分流**：普适的改**通用版**；项目专属的实证/决策改**项目版**（避免污染通用版的可复用性）。
4. **交接**：会话结束在 `session-log` 留指针（"本会话对工作模型的修正见变更记录 vN"），下会话据此接住。

**⚠️ 硬门槛（只靠指针会漂——codex 09：活文档不能只靠 session-log 指针，要有机器/流程强制）**：

- **PR checklist**：改口径/数据模型/权限时，PR 必须"更新对应 golden registry / ADR"或显式写"无口径变化"，否则不合。
- **CI 阻断**：golden registry 里每个 golden 的断言必须在 CI 跑、失败即挡合并（本项目=vitest golden 测试，非可选摆设）。
- **每个活文档带元信息**：owner + last-reviewed + next-review-trigger（触发重审的条件，如"新增一家医院模板时重审 §2 拆分"）。
- **session-log 只做入口不做权威**：权威在 ADR / golden registry / 正文；session-log 是找到它们的指针。
