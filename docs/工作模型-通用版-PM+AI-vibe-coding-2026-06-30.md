# 工作模型 · 通用版 · 非技术 PM + AI 共建软件

> **版本**：2.0｜**性质**：稳定方法论，项目无关｜**配套**：每个项目一份 overlay
> **policy owner**：PM（产品目标与业务政策）｜**maintenance owner**：每次修订只能有一个 owner，具体身份记在 PR/handoff｜**reviewer**：未参与同一实现的独立视角
> **last-reviewed**：2026-07-12｜**next-review-trigger**：出现一类重复失败，或新增/推翻一条稳定机制时
> **边界**：本文件管“PM 与 AI 怎样把产品做对”；Git/worktree/PR 等运行纪律由项目的 operating contract 管。当前 PR、SHA、测试数量、下一步等动态事实不得写进本文件正文。

---

## 0. 先这样开始：PM 只需给 6 件事

> **PM 日常只需看本节和 §8 模板**；其余由 AI 执行，遇到需要拍板的地方再翻成大白话。

PM 不需要先写完整 PRD。能给多少给多少，缺的由 AI 用少量关键问题补齐：

```text
目标：希望谁的什么问题得到改善？
优先级：为什么现在做？和什么相比更重要？
必须有：这次不可缺的结果。
不做：明确不在本次范围内的事。
不可接受：即使功能能跑，也绝不能出现的结果。
验收：看到什么、算出什么或完成什么任务，才算满意？
```

AI 的默认开场必须先给结论，不先倾倒技术过程：

```text
我理解的目标：……
对业务/用户的影响：……
优先级/期望时点：……
本次边界：做……；不做……
不可接受结果：……
风险档位：低 / 常规 / 高影响 / 需明确授权（R0 / R1 / R2 / R3），原因是……
我会自主推进：……
需要 PM 决定：……（没有就写“当前没有”）
验收证据：……
```

只问**答案会改变范围、口径、风险或验收方式**的问题。不会改变方案的问题，AI 采用最安全、最可逆的默认值，写明假设并继续。

## 1. 一句话模型

**PM 决定为什么做、什么不能错、何时值得上线；AI 负责把目标翻译成可判断的结果，先暴露关键假设，再在授权边界内自主实现，并用独立证据证明。**

我们防的不是单纯“AI 会不会撒谎”，而是所有人一起在错误前提上达成一致。为此，每个重要结论都要引入至少一种不共享盲点的证据：真实用户、真实业务数据、可失败的测试、独立 reviewer 或目标系统回执。

## 2. 角色、权力与不能混淆的身份

| 角色 | 有权决定 | 必须交付 | 无权冒充 |
|---|---|---|---|
| **PM / 产品 owner** | 目标、优先级、业务边界、用户体验、业务口径、产品验收，以及本人权限内的业务风险接受；决定是否**请求进入**合并/发布 | 清楚的拍板或“暂不决定”；对可见结果做验收 | 平台审批、安全/法务结论、merge/release 权限或 operator 执行 |
| **实现 owner（AI 或人）** | 已批准边界内的可逆技术选择、实现顺序和局部细节 | 假设、方案、实现、验证证据、残余风险和回滚说明 | PM 的业务拍板、独立 reviewer、生产 operator |
| **reviewer** | 对正确性、安全性、证据充分性给复核结论 PASS / BLOCK / 有条件通过；只有被平台或项目规则正式指定时才有阻断权 | 由 reviewer 本人产生、锚定具体制品/版本的记录、证据和未覆盖边界 | GitHub 正式 APPROVE、PM 验收或发布授权，除非本人确实拥有该权限并执行了对应动作 |
| **merge authority** | 按仓库规则决定/执行合并 | required gates、正式批准和合并结果 | PM 业务验收或生产发布 |
| **production / external operator** | 按授权执行发布、密钥轮换、数据迁移、厂商操作等 | 目标环境证据、执行时间、结果，以及适用的回滚/止损点 | “代码已合并”等于“生产已完成” |

稳定规则：

1. **产品目标与业务政策的 owner 是 PM**；“当值 AI”不是共同 policy owner。安全、隐私、法务、平台或 operator 的正式权限仍归对应有权角色，PM 不能单独替代。
2. 同一文件同一时刻只有一个实现 owner；reviewer 在正式 handoff 前不代写该 owner 的文件。
3. “第二个 AI 看过”是咨询性复核，不自动等于 GitHub 正式审批；“PR body 写已复核”也不是审批证据。
4. 单人仓库暂时找不到独立 reviewer 时：有组织/平台正式门就不得静默豁免，只能做到“已实现/已验证、待审批”；没有正式门时，可披露独立性缺口、加强替代证据，并由对应的风险 owner 明确接受残余风险后继续。R2/R3 仍不得称为“正式 APPROVE”，安全/隐私/法务等风险只能由真正有权的 owner 接受。

## 3. 风险分档：轻任务轻走，高风险才走重流程

命中多项时取最高档。PM 可随时升档；AI 只有拿出证据并记录理由才可降档。

| 档位 | 典型任务 | 最小流程 |
|---|---|---|
| **R0 琐碎、可逆** | 错字、明确的小样式、已验证的重复动作 | 直接做 → 目标检查 → 简短收尾。免 ADR、免独立复核、免长期文档。 |
| **R1 常规** | 一般文档、常规增删改查（CRUD）、内部工具、普通页面改动 | 只填相关字段的短任务卡 + 大白话验收例子 + 适用的检查；改变交互形态时先看原型，完成后走真页面。独立复核可抽查。 |
| **R2 高影响** | 金额/口径/公式、跨表派生、数据库结构（schema）/迁移、权限/认证、医疗或财务数据、关键业务流程 | 完成本文件 §4.1–4.3 的开工检查；事实源/降级/重审触发；必要 ADR；大白话验收场景；可失败证据；独立复核；PM 对业务取舍和残余风险拍板。项目另有机器 preflight 时再追加运行。 |
| **R3 高影响外部或不可逆** | 生产写入/发布、密钥吊销/轮换、删除数据或历史、force-push、不可逆迁移、对外财务更正、影响外部利益相关者的通知 | R2 全部 + **执行前明确授权** + 确认目标/范围/不可逆性 + 适用时 dry-run/备份/回滚 + operator 执行证据 + 上线后验证。若当前指令已明确授权该具体动作，并明确目标和范围，则无需重复询问；不可逆性仍须在执行前披露。不可回滚时必须先给止损/前滚方案。AI 不得自授权。 |

升档触发器采用业务语言：会不会错钱、错权限、泄露敏感信息、破坏历史、影响生产或让用户无法恢复？只要答案可能为“会”，至少 R2；若是影响生产、密钥、业务/个人数据、外部利益相关者或难以恢复的外部写操作，升 R3。已在任务中授权的常规可逆协作动作（如任务分支 push、开/更新 PR）按项目运行契约执行，不因“发生在 GitHub”自动升 R3。

## 4. 最小闭环：从目标到上线

### 4.1 分档与澄清

R0 只需一句目标、一次目标检查和简短收尾，不建任务卡。R1 只填与本任务相关的字段；没有适用自动测试时做目标检查并写“不适用”，不为纯文档硬造测试。R2/R3 才使用完整任务卡：目标、用户、范围、不做、不可接受、验收例子、风险档位、owner/reviewer、外部动作边界。

澄清不是无限追问。把未知分两类：

- **阻塞项**：不决定就会做成不同产品、算错钱、越权、泄密或无法验收。必须停下来让 PM 拍板。
- **非阻塞项**：可以用安全且可逆的默认值继续。记录默认值、owner 和何时重审，不阻塞开工。

停止条件：目标、范围、不做、验收已明确；R2/R3 还需事实源和所有高风险阻塞项明确。再检查一轮没有新增真正阻塞项，即可开工。

### 4.2 先让 PM 看得到，再锁设计

- **界面/交互**：改变页面结构、主流程或文案策略时，写真代码前先给 mockup/prototype，让 PM 判断方向；实现完成后再走真实页面，检查性能、权限、错误态和真实数据。前后两次验证不是一回事。
- **难以反转的决定**：用一页架构/决策记录（ADR）写背景、选项、决定、理由、后果和重审触发；小而可逆的选择不写 ADR。
- **验收例子**：把规则写成“在什么前提 / 做什么 / 应看到什么”（Given / When / Then），先确认句子，再翻成测试。PM 不需要审代码。

### 4.3 证据契约先于实现

R2/R3 在动代码前说明：

1. 哪个系统/文件/表是事实源，为什么不是另一个；
2. 事实源缺失时如何降级，降级结果要标什么；
3. 什么变化会使结论过期并触发重审；
4. 哪个正例、反例和守恒关系能证明实现不是自证；
5. 是否涉及个人敏感信息（PII）、密钥、生产数据，以及数据可否进入仓库、日志、测试样本、AI 对话/上传、外部 connector 或浏览器工具。

金额或数据结论分两类，名称不可混用：

- **Locked Golden**：来源、复现、断言和 CI 阻断均已存在；破坏它会真实失败。
- **Candidate Anchor**：仍是估计、待校准或未进 CI；可以辅助判断，但不得说“已锁住”。

### 4.4 实现

AI 在已批准边界内自主推进，小步提交、只改 owned files，并持续检查新事实是否推翻任务卡。出现下列情况必须暂停：

- 需要扩大范围或改变已拍板口径；
- 发现不可接受结果无法避免；
- 需要 R3 外部写操作或新的敏感数据权限；
- 证据表明原方案会错，且安全替代方案会改变产品取舍；
- ownership 冲突或上游依赖已改变。

发现 PM 的判断与证据冲突时，AI 不盲从也不争辩：先给可复查证据，再解释业务影响、可选方案和推荐项，由有权角色决定。

### 4.5 验证与独立复核

验证要回答“用户要的结果真的发生了吗”，不是只报命令绿了：

- 可执行行为的测试要覆盖关键业务例子和失败路径；R2/R3 至少保留一次实际的修复前失败、负向控制或变异证明，证实证据真的会变红。无法安全变异时给等价、可复查的反例证据并说明限制。纯文档用链接、矛盾/漂移和 diff 检查，不伪造业务测试。
- CI 绿只证明它运行过的检查；不证明 PM 喜欢界面、不证明生产已部署，也不证明仓外操作已执行。
- 测试数量、agent 数量和 PR body 自述只能作背景，不能独立证明完成。
- R2/R3 reviewer 要写**独立性声明**：未参与哪部分实现、换了哪根轴（模型/证据源/反例集/执行环境）、审了哪个具体版本、未覆盖什么。记录必须由 reviewer 本人产生；目标版本或相关依赖变化后，旧验证/review/批准默认失效，除非证据明确覆盖新状态。

### 4.6 PM 验收、发布与复盘

PM 用业务任务验收：用户能否完成目标、结果是否可理解、不可接受结果是否被挡住。隐藏的后端能力也要用业务场景或可读样例演示。

发布是独立状态：合并前后都不能自动宣称上线。R3 需要 operator 证据、健康检查、关键业务检查和适用的回滚/止损点。复盘只沉淀重复出现且可复用的教训；当前进度、PR 和测试数量留在任务/PR/handoff，不进入长期方法论。

## 5. “完成”必须拆成状态台账

| 状态 | 允许的说法 | 最低证据 |
|---|---|---|
| **已实现** | 代码/文档已经写好 | 具体 diff/制品及对应版本存在 |
| **已验证** | 某项行为已被检查 | 具体命令、场景、结果、时间和对应版本；失败路径也有证据 |
| **已评审** | 某 reviewer 看过 | reviewer 身份、具体版本、结论、残余风险；咨询性与正式 review 分开写 |
| **PM 业务验收/批准** | 产品结果或业务取舍获 PM 同意 | 决策/验收记录及范围；不能替代正式技术审批 |
| **正式技术审批** | 仓库/组织要求的 review 已满足 | reviewer 本人产生的平台记录、目标版本和结论 |
| **合并授权/结果** | 有权角色允许合并，或已合并 | required gates + merge authority 记录；与发布分开 |
| **发布/外部执行授权** | R3 动作获有权角色授权 | 目标、范围、operator、不可逆性和授权记录 |
| **已执行并技术验证** | 目标系统确实发生变化 | 技术回执与上线后检查；只有这一行允许说该项“已上线/已止血” |
| **NOT_APPLICABLE** | 有证据证明该环境/动作不适用 | 证据、范围和重开触发；不得说成“已执行” |
| **风险已接受、动作未执行** | 有权风险 owner 接受暂不处置 | owner、理由、范围、期限/重开触发；必须明写“未技术处置” |

R0 默认只报“结果 + 目标检查”；只有任务曾讨论仓外或高风险动作时，才补充“未执行”。R1 只列适用状态；R2/R3 必须逐项写完整台账。没有证据的项写“未验证/未批准/未执行/不适用”，不能省略成一句“完成”。

## 6. 多 AI / 多会话协作

R2/R3、多会话、跨 owner 或需要外部交接的任务保留一个动态事实包，优先放 PR body；没有 PR 时放每任务 handoff。R0 不建 handoff，R1 仅在有 PR/交接时使用相关字段。完整字段：

```text
task id / 业务目标 / 风险档位
owner / reviewer / PM 决策人 / operator（如有）
base 与依赖 / owned files / excluded files
已拍决定与待拍决定
验收场景 / Locked Golden 或 Candidate Anchor 影响
已实现 / 已验证 / 已评审 / PM 业务验收 / 正式审批 / 合并 / 发布或外部终态
外部依赖 / 残余风险 / 回滚 / 下一位 owner
```

并行防冲突：

1. 编辑前查询 GitHub/任务 handoff 中可发现的 live ownership claim；claim 写明路径与 owner。发现重复 claim 立即停，原 owner 释放或正式 handoff 后才能转移；ownership 不授予审批或合并权。
2. 一文件一 owner；共享文件拆不开就串行，不靠“大家小心”并行写。
3. reviewer 先评审，不在同一轮边审边代写；需要接修时先正式 handoff owner。
4. 栈式任务写清真实上游、下游和不能单独落地的范围；每次交接重新核依赖，不信旧快照。
5. PM 决策队列只保存唯一 decision ID、稳定问题、决策 owner、拍板结果/重审触发和关联任务指针；“开放/已拍”只随具名决定更新，不承载实现、PR/check 等执行进度。变化中的当前影响留在关联 PR/handoff/GitHub，不复制另一份决定正文。
6. 长期文档只放稳定结论；动态状态不共同追加到一份大日志，避免并发冲突和旧状态冒充当前事实。

## 7. 给 Codex / Claude 的默认沟通准则

无论使用哪种模型，都遵守同一套 PM 接口：

1. 先给结论和业务影响，再给必要技术证据。
2. 术语第一次出现就翻成大白话；能用业务结果表达，就不让 PM 读命令和堆栈。
3. 只问会改变方案的关键问题；非阻塞问题用安全默认继续。
4. 发现错误时给证据、风险和建议，不因 PM 或前一个 AI 已说过就照抄。
5. R3 高影响外部写操作必须在执行前确认；用户在当前任务中已明确授权的具体动作无需重复问，但不得扩大授权。日常可逆协作动作按项目运行契约执行。
6. 每次交付清楚区分：已实现、已验证、已评审、待 PM 决策、外部依赖、未执行动作和下一步。
7. 不把 agent 数、测试数、PR 描述或“我已检查”当作结论本身。

## 8. 四个可直接复制的模板

### 8.1 进度汇报

```text
结论：当前离目标还有什么。
业务影响：用户现在能/不能做什么。
已实现：……
已验证：……
未验证/未执行：……
正在做：……
风险/阻塞：……
需要 PM 决定：……（没有则写无）
下一步与下次汇报点：……
```

### 8.2 决策请求

```text
要决定：一句话问题。
为什么现在必须决定：不决定会怎样。
选项 A / B / C：各自业务影响、风险、可逆性。
建议：选……，因为……
默认处理：若暂不决定，我能否用什么安全方式继续；不能则明确停在哪里。
```

### 8.3 验收请求

```text
请按这几个业务任务验收：……
预期看到：……
不可接受结果已如何被挡住：……
我已验证：……
我尚未验证：……
PM 可回复：通过 / 有条件通过（条件：……）/ 不通过（差距：……）
```

### 8.4 一页式收尾

```text
目标与最终业务结果
范围内 / 范围外
状态台账：实现 / 验证 / 评审 / PM 业务验收 / 正式审批 / 合并 / 上线或外部终态
关键证据与残余风险
PM 待拍、外部依赖、回滚方式
下一步（owner + 触发条件）
PM 大白话：这轮对用户/业务到底意味着什么
```

## 9. 两层文档怎样继承

1. 通用版只放任何项目都成立的 PM–AI 协作规则。
2. 项目 overlay **默认**只增加项目事实源、风险触发器、验收门和更严格约束；不得复制通用正文。
3. 确需例外时，必须写明证据、有权角色批准和重审触发；例外不得绕过法律、组织或平台正式门，也不能静默覆盖。
4. ADR 保存难反转的业务决定；golden registry 保存可机验锚；决策队列保存开放问题；PR/handoff 保存动态状态。
5. 个人目录、私有 memory key、当前 PR/SHA/check 名/测试数量不得成为稳定规则或事实源。
