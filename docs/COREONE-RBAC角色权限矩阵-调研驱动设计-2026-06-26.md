# COREONE RBAC 角色—权限矩阵（调研驱动设计）

> **生成**: 2026-06-26 · **依据**: 深度调研（22 源 / 57 claim / 25 条 3 票对抗验证 → 22 确认 3 否决）
> **替代**: 旧 `FRS-03-角色管理.md`（仅描述现状、与代码多套口径矛盾，已弃用）
> **定位**: 面向独立/第三方病理诊断实验室（中国，免疫组化 IHC + 特殊染色）的耗材进销存 + 单片 ABC 成本系统

---

## 0. 一句话结论

病理实验室在标准层面被切成**两条平行线**——**医疗诊断线**（病理医师：取材/镜检/诊断/报告签发，受资质门禁）与**技术处理线**（技术员：制片/染色/设备，**明确不诊断**）；**耗材/库存/采购/财务/成本**这条「运营线」在病理标准里没有岗位定义，须按**最小必要(least privilege)+ 职责分离(SoD)+ 角色围绕岗位职能**三原则推断。由此：**成本/利润可见性归财务与运营管理者，不归诊断医师**；**技术员归「技术+消耗」线（出库/消耗对账录入），不碰成本/利润**。

## 1. 真实岗位与职责（有据）

| 线 | 角色 | 核心职责 | 依据强度 |
|---|---|---|---|
| **诊断线** | 病理诊断医师 pathologist | 取材、镜检、病理诊断、报告**审核/签发**（签发权=中级+且≥3年）。**不碰运营/成本** | 高（CNAS-GL048、T/NAHIEM 98-2023） |
| **技术线** | 病理技术员 technician | 标本处理/脱水/包埋/制片/染色、设备使用维护、阳性对照选择(QC)。**明确不诊断** | 高（CNAS-GL048 附录A） |
| 诊断+技术 | 二者共担 QC | 室内质控 + 室间质评是医师与技术员各自必备能力（非独立成本岗） | 高（CNAS-GL048） |
| 运营线 | 标本接收登记 | 标本-申请单双核对、分配病理号（独立职能；本系统无此模块→N/A） | 高 |
| 运营线 | 库房管理 warehouse | 出入库/库位/盘点（**保管 custody** 须与**台账/对账 record-keeping** 分离） | 高（内控 SoD / COSO） |
| 运营线 | 采购 procurement | 采购订单/供应商（**需求提出**须与**供应商选择/定价**分离） | 高（《企业内部控制应用指引第7号》） |
| 运营线 | 财务/成本会计 finance | 成本核算/单片成本/盈利分析/BOM 标准成本（**成本/利润可见性归此线**） | 中（least-privilege/SoD 推断，见 §6 caveat） |
| 运营线 | **实验室主任/运营管理者 lab_director** | 全局总览（含成本/盈利）+ 配置/审批；标准要求设技术负责人，区别于一线 bench | 高（角色存在）/ 中（成本可见性归属为推断） |

> ⚠️ 运营线（库房/采购/财务/主任）在病理标准里**无岗位定义**，是按通用内控 + RBAC 原则的**合理推断**，非病理文献直接给定。

## 2. 耗材/成本场景下「谁负责什么」（有据）

| 环节 | 负责角色 | 说明/依据 |
|---|---|---|
| ① 库存/库房（入出库/库位/盘点/调拨） | 库房管理（执行）；技术员（盘点执行/出库领用） | custody 与 record/对账分离 |
| ② 工作台试剂实际领用消耗（出库） | **技术员** | 技术线职责；非诊断医师 |
| ③ 消耗对账（实际 vs BOM 标准用量） | 技术员**录入实际** + 运营/管理层**核准**（**非诊断医师独立**） | 中（SPD 相邻先例 + SoD 推断） |
| ④ 采购与供应商 | 采购 | 需求/选择/定价三岗位分离 |
| ⑤ 成本核算/单片成本/盈利分析 | **财务/成本会计**（维护）+ 主任（查看） | BOM 标准用量×采购价→标准成本；盘点取实际→差异对账→周期调 BOM |
| ⑥ 报废/质控(QC) | 技术员（+医师 QC） | QC 归技术线+诊断线 |
| ⑦ 检测项目(panel) + BOM 配方 | 财务/运营维护标准用量；技术员/医师只读 | 两级 BOM 是标准成本基础 |

**与现状的核心纠正（粗体处）**：
- **病理医生**：成本/盈利/ABC/单片成本/物料成本 一律 **—**（现状错误地全开）。病理在「耗材+成本」系统里角色极小：只读检测项目/库存/BOM/预警 + QC。
- **技术员**：归「库存+消耗对账+QC」线（出库/盘点/消耗对账录入/报废 W），成本/盈利 **—**（我之前错给了成本 KPI）。
- **财务**：成本/盈利的**唯一 W 拥有者**（这点我之前对了）。
- **运营总览（含成本）**：归 `lab_director`（或 admin 暂代），不下放诊断/技术线。

## 5. 多角色账户处理（设计建议）

**现状**：`users.role` 单值、JWT 单 role、`ROLE_PERMISSIONS` 按单角色——**系统当前不支持多角色**。

**调研结论**：多角色在小型实验室确有发生，但属 **open question**；通用 SoD 要求**不相容职务必须分离**（采购需求 vs 选择定价；保管 vs 对账核准），CNAS 亦有「技术员与医师不相互兼职」信号。

**推荐设计（待你定）**：
1. **数据模型**：`users` 增 `roles`（多值 / 或 `user_roles` 关联表），保留一个 `primary_role` 作身份展示。
2. **能力合并（关键）**：前端 nav + 仪表盘 + 后端鉴权一律**按「该用户所有角色权限的并集」**驱动（capability-driven），而非单角色 switch——**天然兼容单/多角色**，且顺手根除 §7 的 403 toast。
3. **SoD 守卫**：对**不相容组合**（如 采购选择 + 财务核准、库房保管 + 对账核准）在分配时**告警/需审批**，小实验室可走「SoD 豁免审批 + 双人复核」。
4. **成本可见性**：作为**可配置默认**（默认仅财务+主任可见成本/利润），而非硬编码——因该结论是原则推断（§6）。

## 6. Caveats（诚实）

1. **「成本/利润可见性归财务而非诊断医师」缺中国病理实验室直接文献**——是 least-privilege(NIST AC-6/HIPAA) + SoD(COSO) + billing-vs-clinical 类比的**合理推断**。两条试图直接锚定的声明在对抗验证中被**否决(0-3)**。**建议落地为可配置默认，而非硬断言。**
2. **来源分层**：高=CNAS-GL048 / NHC 2009 指南 / 《企业内部控制应用指引第7号》；中=T/NAHIEM 98-2023（团体标准，最弱层级，且为建设标准非角色标准）；弱但有 NIST/COSO/HIPAA 背书=reanda/Censinet 等。
3. **场景外推**：消耗对账(SPD/定数包)与成本核算(检验科 BOM)实证均非「独立第三方病理 IHC/特染」专属；per-slide 抗体用量为合理外推，落地须用真实工作台数据校准。
4. **取材归属**：取材(grossing)按中国标准由**医师**承担（非技术员）。

## 7. 当前实现的偏差（待修，→ 实施清单）

1. **403 toast 根因**：仪表盘 hook 无差别给所有角色拉 `getStats/inbound/outbound`，axios 拦截器逢错弹 toast → 财务进首页弹 3 个、病理/技术/采购各 1 个。**修法：按角色能力只调有权接口。**
2. **前端 nav ↔ 后端 requireRole ↔ ROLE_PERMISSIONS ↔ 本矩阵 四套口径打架**——以本矩阵为单一事实源，三处对齐。
3. **病理/技术员去成本**：前端 nav 去 ABC/成本入口、后端 `/abc`·`/reports` 角色表去 pathologist/technician（除非定为 lab_director 可见）、仪表盘改非成本 KPI。
4. **财务补库存只读**：财务 `/inventory` 当前后端 403，但财务需 R（成本看板要库存上下文）→ 评估开放只读或仪表盘不拉库存。

## 8. 用户确认的落地决策（2026-06-26，逐行评审后锁定）

> **§8 为用户确认的最终「初始种子值」，覆盖 §3 推荐处。** 因决定做**数据驱动 RBAC**（矩阵存 DB、管理员可在「角色权限」页改），本表只是初始种子，后续随业务自由调，无需改代码。

### 8.1 三项架构决策
1. **多角色：一步到位**——`user_roles` 关联表（多值）+ `primary_role`（身份展示）+ 鉴权/nav/仪表盘按**所有角色权限并集**驱动（capability-driven）+ 不相容组合 SoD 告警。
2. **成本可见性：可配置默认**——默认仅 `finance`+`lab_director`(+admin) 可见成本/利润，做成开关（非硬编码），因该结论是内控原则推断（§6 caveat）。
3. **数据驱动 RBAC（用户强调"后续配置灵活性"）**——矩阵存 DB（复用/扩展 `roles.permissions`），前端 nav + 后端鉴权 + 仪表盘 + 能力并集**全部读 DB 这份单一事实源**；管理员经「角色权限」页改格子即时生效，不改代码不发版。

### 8.2 §3 矩阵的确认改动（逐行评审 5 个判断点）
1. 病理 `库存`/`预警` 保留 **R**（不变）。
2. `BOM` → 主任、技术员 **W**（原仅 admin W）。
3. `检测项目` → 主任、技术员、病理、财务 **W**；采购 **—**；仓管 **—**。
4. `物料成本分析` → 采购 **R**（完整读，非"仅价格"）。
5. `设备管理` → 主任、技术员、财务 **W**（财务由"仅折旧"升为完整 W）；仓管 R，病理/采购 —。
- 一致性确认：检测项目对 病理/技术 开放 W，但 **ABC看板/单片成本/盈利仍对病理与技术为 —**（成本不下放诊断/技术线，符合"病理去成本"初衷）。

### 8.3 背景约束（用户原话）
- 决策基于**当前实验室现状（小公司、第三方）**；要求**为后续角色权限配置预留灵活性** → 故选数据驱动而非硬编码（见 8.1.3）。
