## 〇、结论先行（TL;DR）

> **✅ PM 已拍（2026-07-03）**：
> - **① 方向 = 配置类保留 + 报表类收敛**（「ABC 前端全废」不采纳；接受分治：10 配置页保留、9 报表页收敛）。
> - **② 报表类去向 = 新建统一报表平台**（把报表类 9 页收编成该平台的 Tab/维度，而非逐个塞进 hospital-pnl）。
> - **由此，6 个「待定」报表页的落点已明确 = 统一报表平台（待建）**；3 个「合并候选」中 `model-validation`/`trend` 也并入该平台，`supplier-costs` 仍归 `cost-analysis` 供应商 Tab（采购视角，不属 ABC 平台）。**下一步 = 另立"统一报表平台设计 + 孤儿配置页补导航 + 2 缺陷处置"实现项**（见 §五）。本清单仍不动任何代码。

**19 个页面：`保留 10` · `合并候选 3` · `待定 6` · `可直接删 0`。**

- **没有一个页面可以直接删。** 每一条要么是某类参数的唯一录入入口（丢了方法论没处落参数），要么是某个数据切面的唯一视图（丢了就少一块能力），要么带着一个必须先决策的口径/接线问题。全部列为**候选**，等 PM 拍。
- **「ABC 前端全废」不成立的关键漏洞**：19 页里有 **10 页是「配置类」**——它们是作业中心、成本动因、成本池、BOM↔收费映射、间接成本分摊、预算、质量成本、季度调整这些 ABC 方法论参数**在系统里唯一的录入界面**。把方法论落地成可算的数，恰恰要靠这些页面录参数。删掉它们，"方法论"就退回到"只能改数据库/直接调 API"。
- **真正与既有功能重叠的是「报表类」9 页**：它们把 ABC 引擎已经算好的成本/利润数据，换个图表/维度展示；这类才与 `hospital-pnl`（医院盈利看板）、`逐抗体成本`、`cost-analysis`（物料成本分析）等重叠，是废弃/合并候选。
- **可发现性才是眼下最尖锐的问题**：18 个 `/abc/*` 路由里 **14 个是「孤儿路由」**——在 `App.tsx` 注册了，但侧栏没有入口、也不在任何角色的可访问路径表里。**普通用户在正常操作中根本点不到它们**（只能手敲 URL）。其中还包含好几个**有独有录入能力的配置页**（fee-mappings / cost-drivers / cost-pools / budgets / quality-costs / quarterly-adjustment）。所以对这批页，第一决策往往不是"删不删"，而是"方法论要不要用它——要用就得补导航，不用才谈下线"。
- **不是空壳**：19 页里 **18 页有真实后端支撑**（真 SQL、真表、非 mock）。所以重叠讨论的是"用途重复"，不是"假页面"。唯一例外见下方缺陷①。

**两个顺带查出的实现缺陷（不在本次处置范围，建议另立修复项）**：
- **缺陷① `personnel-efficiency`（人员效率）调了个"幽灵接口"**：它请求的 `GET /api/v1/reports/personnel-efficiency` 在后端**根本不存在**，页面运行时必然 404、四块区域从未显示过真实数据。是全清单里唯一"连空壳都算不上"的页。
- **缺陷② `variance`（成本差异）的"标准成本"是假的**：后端把"标准成本"直接等于"材料实际成本"（复制一份），人工/设备/间接的标准值硬编码为 0。页面标题宣称的"BOM 标准 vs 实际"名不副实；真正的"理论 vs 实际"口径其实已由 `消耗对账` 页真实交付（物量口径）。

## 一、判据说明（怎么分的组）

| 维度 | 含义 | 为什么重要 |
|---|---|---|
| **分类 config / report** | config=录入/维护参数的表单（增删改）；report=只读分析/图表 | **config 是方法论的落参数入口，删了没替代**；report 是展示层，可能与别处重复 |
| **数据来源 / 是否真后端** | 调哪些接口、是真 SQL 还是 mock/占位 | 判断是"真功能重叠"还是"假页面" |
| **独有能力（删了丢什么）** | 若删除，系统会失去什么别处拿不到的东西 | **对抗判据核心**——防止把"唯一入口"当"重复项"误删 |
| **重叠对象** | 哪个现有功能覆盖了同一地盘 | 决定"合并去向" |
| **可达性** | 侧栏有入口 ✅ / 孤儿路由 ⚠️（只能敲 URL） | 决定"该补导航"还是"该下线" |

**四个"重叠对象"的真实边界**（供判断合并去向；均已读码核实、真实数据）：
- **`hospital-pnl` 医院盈利看板**：按医院上卷「财务实收→实验室收入→ABC 核算成本→院级毛利」的 P&L 视图 + 月度趋势 + 负毛利病例下钻。**是成本结果的消费/归因端**，不碰成本参数配置。
- **`account-reconcile` 账实核对**：收入侧对账引擎（账单片数 vs LIS 物理片数），**只算收入准确性，不算成本、不出利润**。
- **`逐抗体成本` antibody-cost**：**纯后端引擎**（逐片一抗材料成本 + 抗体名解析），**目前没有前端消费页**；工时/设备两参数还是"G2 估·待校准"。
- **`cost-analysis` 物料成本分析 + `reconciliation` 消耗对账**：出库料成本的**简单汇总**（SUM/GROUP BY，无 ABC 多维分摊）+ BOM 理论用量 vs 实际出库的物量对账。**是 ABC 的上游（料成本/用量），不是全成本分摊**。

## 四、两个已披露的实现缺陷（不在本次处置范围，建议另立修复项）

> 审计顺带查出，如实记录。**本次不修**（本任务纯审计），供 PM 决定是否另开修复项。

- **缺陷①（personnel-efficiency 幽灵接口）**：前端 `reportsApi.getPersonnelEfficiency()` → `GET /api/v1/reports/personnel-efficiency`，但后端 `reports-v1.1.ts` 只有 4 条路由（cost-by-project/material/supplier/trend），全仓 grep `personnel` 零命中。**该端点不存在**，页面必 404。要么补后端端点（需先设计"标准工时/效率"计算口径），要么页面+路由+api 三件套一并清理。
- **缺陷②（variance 假标准成本）**：后端 `abc-v1.1.ts` 里 `totalStandard += materialActual`（把材料实际成本当"标准"），labor/equipment/indirect 的 Standard 硬编码 0。页面标题宣称的"BOM 标准工时/用量计划值"后端根本没产出。真正的"理论 vs 实际"其实已由 `消耗对账` 页真实交付（BOM 理论用量×病例数 vs 实际出库量，物量口径 + 差异分级 + SoD 双人复核）。

**另记两处相邻观察（更次要）**：
- **披露A（budgets 文案超前）**：预算页弹窗声称"确认后将接住成本看板/成本预警"，但目前**无任何下游模块消费 `abc_budgets` 表**，属文案层夸大（不影响"唯一录入入口"的判断）。
