> 用途：给产品决策用。本文不写代码，只给**问题 → 为什么阻断采纳 → 方案 → 影响/工作量 → 关联技术债**。

## 0. 一句话判断

**现在已经证明「规整逐项明细」能端到端跑通（和睦家黄金 ¥13,152），但离财务每月真正用起来，差一个「月结产品」闭环。**
核心缺口：① 一批真实模板（类别汇总 / 共建 / 纯外送）**只能预览、不能入库**，对应医院会从院级盈亏里消失；
② 宽表「待人工归类」诚实但**不可操作**；③ OUT（转销/远程/共建）**没形成财务可用闭环**；④ 缺**月结工作台 / 历史重算 / 信任锚产品化**。

## 2. 阻断采纳级（必须先解决，否则财务用不起来）

### Epic A — 月结事实层：让所有真实模板都能入库进盈亏
- **问题**：现实现只有 `line_item`（逐项明细）能 `/commit` 落库。`category_summary`（东安，合计 **¥121,016.90**，类别小计里混 IN[常规/IHC/EBER/TCT] 与 OUT[HPV/基因/FISH]）、`joint_venture`（石门，科室共建利润表）只返回 `note`；`outsourced_detail`（赣州，纯外送 **¥40,219.20**，**无病理号**）走到「无可逐 case 落库」→ 失败。
- **为什么阻断**：这批医院（东安/石门/赣州…）会**从院级盈亏里消失**，财务必须线下另维护一套表 → 系统当不成月结主流程。
- **方案**：增加「**月度对账事实层**」，不强求每张账单逐 case：
  - `category_summary`：类别 → 业务线(IN/OUT) 映射，写 **partner-month 聚合**（不逐 case）；
  - 纯 OUT 账单：允许入库为 `labRevenue=0 / outSettle=全额 / 无 case 成本`（用于月结完整性 + 公司级转销交接）；
  - `joint_venture`：专用共建分成事实表，进看板时明确「不参与实验室工序毛利」或单列。
- **真实素材**：`out_category_summary__dongan`、`out_joint_venture__shimen`、`out_outsourced_detail__ganzhou`。
- **关联技术债（04）**：会重写 commit/落库路径 → 04 的「跨院串账」「NGS 缺成本」等大概率要在这层一起重新设计。

### Epic B — 宽表按列归类：让「待人工归类」可操作
- **问题**：平泉（`consult_remote`）业务语义在列头（`远程会诊结算`/`免组结算金额`/`结算合计`），行内项目名为空、病理号纯数字；宁波（`diagnostic_fee`）同理（`基础诊断费用`/`免疫组化结算`）。现测试台待归类卡只给「（无项目名）/病理号/金额」，**不带来源列、诊断方式、候选业务线**。
- **为什么阻断**：财务既不能把「远程会诊结算」一键归 OUT，也不能判「免组结算」是否 IN → 最后只能 `confirm:true` 入库为 0 实验室收入，或乱加前缀规则。
- **方案**：解析宽表时把每个金额来源列展开为带 `sourceColumn/sourceLabel` 的明细；测试台**按「列」归类**（非按「行」）；给候选规则（「远程会诊结算→远程诊断 OUT」「免组结算金额→线下会诊/IHC IN」）由财务确认写回该院配置。**保守不自动入库是对的**，但必须给足上下文。
- **真实素材**：`out_consult_remote__pingquan`、`out_diagnostic_fee__ningbo`。
- **关联**：这正是上一轮 04 里我「宽表保守落待人工归类」的产品化升级——做了 B，那条「保守告警」就被吸收。

### Epic C — OUT 一等公民 + 台账
- **问题**：边界明确 OUT（NGS/FISH/HPV-E6E7、远程含 retainer、共建分成、科研）不进实验室工序收入，但它们**仍是财务月结事实**。现状：对账单里 OUT 行有的不能入库（赣州纯外送）；独立 NGS 后端有订单 API 但**前端无导入入口**；月度向导不提示「这张应转外购转销台账」。
- **为什么阻断**：OUT 在系统里「消失」，财务月结不完整、无法和公司级运营系统交接。
- **方案**：月度工作台分三栏「实验室收入 / 外购转销·远程·共建（移出）/ 未归属」；外购转销有独立导入入口、缺成本治理、与公司级系统交接状态；实验室 P&L 排除 OUT 但月结不丢 OUT。
- **关联技术债（04）**：吸收「NGS 缺成本/售价硬阻断」「NGS 读路由 RBAC 收窄」「NGS 完整度评分」——都在 C 里统一设计，**先不单独修**。

## 3. 重要缺口（决定能不能规模化、能不能持续用）

### Epic D — 月结工作台（19 院 60 文件撑不起单院单文件流程）
- 现状是「选一家院 → 传一个文件 → 预览 → 入库」。财务每月需要知道：哪些院已收单/已预览/有未匹配/已确认入库/成本未接通/可关账。
- **方案**：`YYYY-MM` 月结工作台，医院按状态分组（未收单/待归类/待确认/已入库/成本待核/已关账），支持拖入多文件自动识别院+模板，批量预览逐院确认，输出月结差异报表。

### Epic E — 规则治理 + 历史重算（改规则敢不敢用）
- 逐院配置会持续纠错。财务改了「HPV-E6E7 属 OUT」或把某宽表列改成远程 OUT 后，最关心「哪些已入库月份受影响、差额多少、要不要重算」。现有版本/回滚/基线 + `case_revenue.config_version` 已是地基，但缺**影响预览、重算入口、基线生效期、关账月调整策略**。
- **方案**：保存配置前后按历史 import_batch dry-run，列受影响月份/院/金额差；未关账月一键重算，已关账月生成调整单；基线绑定「从哪个账期起生效」。
- **关联技术债（04）**：若 E 重构配置读写，则 04 的「normalizeConfig 只在保存路径、读/回滚不归一」一并在此治理（读路径归一 + 回滚校验）。

### Epic F — 样表驱动建档向导 + 信任锚产品化
- **建档**：配置页 6 tab + 大量字段，但财务最自然的是「上传该院样表 → 系统识别模板/列映射/业务线候选/扣率 → 逐步确认」。现测试台与配置页是两页。**方案**：「新医院建档向导」：上传样表→识别→IN/OUT 候选→设扣率/特殊结算→跑黄金/体检卡→设首个基线；配置页降为高级维护。
- **信任锚**：体检卡「黄金值」是财务敢信的关键，但前端**没有录入/维护 `goldenExpected` 的位置**，黄金锚停留在测试代码里。**方案**：测试台/向导加「财务期望实收/账单声明实验室收入」字段（可从文件合计自动带入），按院+账期保存，体检卡显示差异/阈值/确认人。
- **关联技术债（04）**：「新医院首次内联归类 v0 死锁」会被建档向导流程重写吸收——**先不单独修**。

### Epic G — 默认服务目录升级（少让财务手工补 OUT 词）
- 温州样本里 HPV-E6E7 多行，产品说明也明确 HPV-E6E7 属 OUT，但默认配置只有 NGS/FISH/远程/共建，**没有 HPV/E6E7 关键词** → 上线初期每家院都要手工补显然常见的 OUT 词，财务觉得「系统不懂业务」。
- **方案**：默认业务线从「8 条示例线」升级为可维护**服务目录**（NGS/FISH/HPV/STR/远程/共建预置同义词+前缀，逐院启停/覆盖）；测试台从未匹配项推荐「加入全局目录」或「仅该院」。**工作量小、见效快**。

### Epic H — 病例匹配可恢复（LIS 晚到/错院/别名）
- 真实 LIS ~1454 case/22 院，可能晚于对账单、无登记月、医院名别名不同。现导入页只显示命中/缺失计数，财务看不到缺哪些 case、去哪补 LIS、是不是「本月 LIS 未导入」。
- **方案**：病例匹配做成可恢复状态（无本期 LIS / 部分缺失 / 错院别名疑似 分开）+ 缺失 case 列表导出 + 跳转 LIS 导入 + 别名匹配建议；工作台允许「先按收入入库、标记病例/成本待补」。

## 4. 体验问题 + 增强建议（采纳后提升信任与效率）

**体验**：① `confirm:true` 高风险动作需展示未匹配/差额/无合计行原因 + 选原因(待补规则/纯外送/暂按0) + 入库后保留「待复核」状态（审计）；② 诚实信号（估算 N 例/未接通）要**可 drill-down + 有责任人**（点开 case 清单/跳成本回填/跳测试台批次/跳 NGS 补成本）；③ 类别汇总/共建即使不入库也要展示**可读专用预览表**（类别/科室、收费、结算、建议 IN/OUT、声明合计、差额、不可入库原因）。

**增强**：① 模板库+医院样本库（识别「匹配温州服务费混合，置信度 92%，上次成功 2026-04」）；② 月结对账差异包（Excel/PDF：各院合计/实验室收入/移出/未匹配/成本接通率/毛利/确认人/待办）；③ 配置与确认入库纳入 **maker-checker**（大额规则变更/未识别确认入库/OUT 改 IN 需复核）；④ OUT 台账 API/导出（给公司级运营系统/财务总账）；⑤ 外购转销成本完整度评分（订单号/售价/外包成本/月份/院匹配/缺成本确认）。

## 5. 架构决策项 X — 跨院同号口径（CRITICAL，独立于产品但影响数据可信）

- **问题（04 复审已核实）**：上一轮只把 `case_revenue` 唯一键改成 `(partner_id, case_no, service_month)`，但 `lis_cases.case_no` 仍**全局 UNIQUE**、LIS 导入 `ON CONFLICT(case_no)`、P&L `JOIN lis_cases ON lc.case_no=cr.case_no`、ABC 回填与成本 rollup 都按 `case_no` 单键 → 不同医院同本地病理号会**串 LIS 数量/串成本/相互覆盖**。
- **两个方案**：
  - **(轻) 产品上禁止跨院同号**：LIS/收入导入时检测到跨院同号即报冲突阻断。成本低，但限制了「不同医院本就可能用相同本地号」的现实。
  - **(重) 全链路升级 `(partner_id, case_no)` 口径**：`lis_cases` 自然键、LIS 导入 upsert、人工覆盖、ABC 回填、P&L join、case 成本 rollup 全改；配套回归测试覆盖 `buildPartnerPnl`/`loadCasePnlsWithCost`/`backfillAbcPartnerIds`。彻底但动到 LIS/ABC 子系统。
- **与产品的关系**：Epic A（月结事实层，重写落库）和 H（LIS 病例匹配）都会触及 case 键口径 → **建议在 A/H 实现时一并定口径**，避免重复改。**需你拍**：轻方案 or 重方案。

## 7. 建议执行顺序（MVP：让财务能真正跑一次月结）

1. **先做 G（默认目录升级，小而快）** + **B（宽表按列归类）** + **A（月结事实层）** —— 这三件让「所有真实模板都能被处理、不漏院」，是月结能成立的最小集。期间一并定 **X 口径**。
2. 再做 **C（OUT 台账）** + **F（建档向导 + 黄金值产品化）** —— 让 OUT 闭环、首用顺、财务敢信。
3. 然后 **D（月结工作台）** + **E（规则治理/重算）** —— 规模化与可持续。
4. **H + 体验/增强档** 穿插补强。

> 每个 Epic 落地后，回到 §6 复评对应技术债是否还存在、是否已被新实现吸收。
