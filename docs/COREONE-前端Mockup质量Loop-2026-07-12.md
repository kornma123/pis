# COREONE 前端 Mockup 质量 Loop

> **状态**：v1.0 草稿（2026-07-12）· 按「产出必过质疑关」规矩，PM 质疑/拍板后生效
> **从属关系**：本文是《COREONE-前端标准-流程质量设计文案UX-2026-06-27》中「mockup 先行、未定稿不写真代码」红线在 mockup 阶段的**执行流程细化**；与前端标准冲突时，一律以标准为准。文中清单为速查版，全量口径以标准原文为准。
> **适用**：所有新建或大改前端页面/流程的任务。已定稿页面的小修、纯文案微调不走本 loop。

## 0. 在前端工作流里的位置

```
需求讨论循环（清晰了、没新项）→ 【Mockup 质量 Loop（本文）】→ PM 定稿 → 写真代码 → 真跑验收
```

Mockup 阶段的唯一使命：**在写任何真代码之前，把「AI 理解的」和「PM 想要的」之间的偏差磨到零。**
返工发生在改草稿（几分钟），而不是改代码（几小时）。

## 1. Loop 全景

```
┌─> ① 产出 / 修订 mockup
│   ② AI 自检（三张清单，贴勾选结果）
│   ③ 摊假设（列出所有替 PM 做的决定，标「待拍板」）
│   ④ 自我质疑（先攻击自己的稿子，能修的当轮修掉）
│   ⑤ 交付 PM（固定五件套）
│   ⑥ PM 反馈归类：拍板 / 要改 / 新讨论
└── 有「要改」或「新讨论」→ 回到 ①
    全部拍板 + 退出条件满足 → 定稿归档 → 才允许开写真代码
```

每一轮都完整走 ①–⑥，不跳步。只交 mockup 不摊假设 = 无效轮。

## 2. 每步做什么

### ① 产出 / 修订 mockup

- 用**真实业务样例数据**（真项目名、真金额量级、真病理号格式），禁 lorem ipsum、禁「测试1」「abc」——假数据会掩盖布局和口径问题。
- 一页 mockup 至少覆盖三种状态：**正常有数据 / 空态（第一次进来什么都没有）/ 出错**。涉及角色权限差异的补「低权限用户看到什么」；加载态用全站统一样式的可不画。
- 修订轮开头一句话列出「这轮改了什么」，PM 不用逐像素找茬。

### ② AI 自检——三张清单逐项打勾（勾选结果贴进交付）

**清单 A · 视觉规范（速查版）**
- [ ] 字体 Inter，无其他字体
- [ ] 圆角分级：按钮/输入框 6px、卡片 8px、弹窗 12px、状态标签全圆 pill
- [ ] 主色 #3b82f6，hover #2563eb
- [ ] 文字：主 gray-900 / 次 gray-500~600 / 禁用 gray-400；**无纯黑**
- [ ] 边框统一 gray-200；阴影 sm/md（弹窗可 lg），不用 xl
- [ ] 按钮高 40px；focus 有蓝色 3px 光圈
- [ ] 无 0 圆角、无大面积刺眼饱和色
- [ ] HTML/组件式 mockup 的输入框**显式白底 + 深色文字 + 浅灰 placeholder**（防宿主深色模式把输入框刷黑、输入不可见）

**清单 B · 文案说人话**
- [ ] 黑名单零命中（上卷 / 下卷 / IN / OUT / ABC / P&L / 服务项结算之和 等行话）
- [ ] 按钮是动词短语（「保存修改」，不是满屏「确定」）
- [ ] 错误提示告诉用户**下一步怎么办**，不只报「失败」
- [ ] 空态有引导（告诉用户第一步做什么），不是一片空白

**清单 C · 流程完整性**
- [ ] 用户从哪进这页（导航/别的页的哪个入口）
- [ ] 每个操作完成后去哪、看到什么反馈
- [ ] 危险操作（删除/覆盖/提交）有确认或可撤销
- [ ] 不同角色分别看到什么（以**运行库真实权限**为准，别按源码种子矩阵想当然）
- [ ] 同类交互与现有页面一致（同一操作不能这页一个样、那页一个样）

### ③ 摊假设

列一张「本轮我替你做的决定」清单：字段取舍、默认值、排序规则、入口位置、名词叫法……每条标 `待拍板`。

> 这是整个 loop 的核心动作：**AI 自由发挥的地方 = 将来返工的地方**，全部摊出来给 PM 拍。

### ④ 自我质疑（质疑关的 mockup 版）

至少从这几个角度攻击自己的稿子，能修的当轮修掉，修不掉的写进讨论项：
- 第一次用的新手看得懂吗？
- 数据极端时（0 条 / 999+ 条 / 超长名称 / 大金额）会不会破版？
- 手滑点错，最坏后果是什么？
- 和系统里已有的同类页面打不打架？

### ⑤ 交付 PM（固定五件套）

1. mockup 本体（可看，最好可点）
2. 「这轮改了什么」一句话清单
3. 假设清单（含待拍板项）
4. 讨论项（修不掉的、拿不准的）
5. 结尾大白话：这页给谁用、解决什么问题、用户会怎么用

### ⑥ PM 反馈归类

- **拍板**：记进假设清单，状态改「已拍板」，之后不许再改（要改 = 作为新讨论项重新拍）
- **要改**：进下一轮修订
- **新讨论**：讨论是循环不是一问一答——新冒出来的问题继续循环，不许假装没看见

## 3. 退出条件（全部满足才算「定稿」）

1. PM 明确说了「定稿」或等价表述——**沉默不算同意**
2. 假设清单零「待拍板」残留
3. 三张清单全绿（最后一轮交付里贴勾选结果）
4. 必备状态（正常 / 空态 / 出错，含权限视角如适用）都有稿
5. 最近一整轮 PM 没有提出任何「要改」和「新讨论」

**定稿动作**：把最终稿归档（HTML/截图 + 拍板记录），写明日期和版本。**这份定稿 = 写码阶段的唯一验收基准。**

## 4. Loop 护栏（防跑偏、防空转、防烂尾）

| 护栏 | 规则 |
|---|---|
| 红线 | loop 期间禁止写任何真组件/真路由/真接口对接，「顺手先搭个架子」也不行 |
| 防空转 | 每轮必须有可见差异；连续 2 轮没实质变化 = loop 卡住，直接升级问 PM |
| 防打转 | 同一个问题改到第 3 轮还没过 → 停止再猜，把几个候选方案**并排画出来**让 PM 选 |
| 防偷改 | 已拍板的决定不许在后续轮里悄悄改回去 |
| 防烂尾 | 每轮交付必须完整走 ①–⑥；缺假设清单或缺自检勾选 = 无效轮，重交 |
| 写码衔接 | 写码阶段发现 mockup 不合理 → 回来改 mockup 重新拍板，不许在代码里自由发挥 |

## 5. 碰钱 / 碰口径页面的加固

对账、成本、毛利、预算类页面，定稿前在本 loop 之外**加一道异构复核**（第二双眼睛）：

- `codex exec -i <定稿截图> "对着这张图审口径与文案"`（视觉审），或
- Claude 对抗面板（Workflow 多 agent）

复核意见按 ⑥ 的三分类处理，处理完才算定稿。

## 6. 会话注入块（开工时贴给 AI）

新开 mockup 任务时，把下面这段贴进会话即可激活本 loop：

```
【Mockup 质量 Loop 已激活】本任务处于 mockup 阶段，按 docs/COREONE-前端Mockup质量Loop-2026-07-12.md 执行：
1. 未定稿前禁止写真代码（组件/路由/接口一律不碰）。
2. 每轮交付固定五件套：mockup、本轮改动清单、假设清单（待拍板）、讨论项、结尾大白话。
3. 交付前先过三张自检清单（视觉/文案/流程）并贴勾选结果。
4. 我说「定稿」之前循环不结束；同一问题改 3 轮未过就并排出候选让我选。
```

## 7. 本文档自身的待拍板项

| # | 事项 | 建议 |
|---|---|---|
| 1 | 定稿 mockup 归档位置 | 建议 `docs/mockups/<功能名>/`，或延续 V1.1设计稿 目录，PM 定 |
| 2 | 状态覆盖最低要求 | 现定「正常/空态/出错」三态必备，是否够用 PM 拍 |
| 3 | 碰钱/口径页面的异构复核 | 现写为强制，是否降为建议 PM 拍 |
| 4 | 生效后是否在前端标准文档里加一行指回本文 | 建议加（保持标准是唯一入口），PM 拍 |

## 8. 给 PM 的大白话

这份文档定了一个规矩：以后做前端，先用草稿图循环打磨。AI 每轮都要自己过三张检查清单、把它替你做的决定全部摊出来让你拍板、自己先挑自己的毛病，然后才交给你看。你没说「定稿」之前，AI 不准写正式代码；同一个地方改三轮还不满意，AI 就得把几个方案并排画给你选，不许一直瞎猜。这样返工都发生在改草稿上（几分钟的事），最后写出来的代码一定长成你拍过板的样子。
