## 一、背景与定位

### 1.1 这套系统是什么

- **改造对象**：现有 LIS 内的"收费系统"。现状是**手动输入旧收费代码**计费（沟通记录 §1.1）。
- **目标**：基于新的 **81 项收费代码**，让系统**自动生成计费**——把手动选码改为按 LIS 已有数据字段自动映射收费码、自动计算金额。
- **性质**：这是**正向、交易型**系统，产生的是**真实账单**（会进对账单、影响实收）。不是估算、不是报表。

### 1.2 关键设计取向（已在沟通中定稿，本文不挑战）

- **计费与诊断流程完全剥离**：不在签发时自动计费，改为用户在计费页面**手动点击"更新计费信息"**触发（沟通记录 §2.1）。原因：自动计费时间点 / 数据刷新周期不好设，且需要逐个对账人工确认。
- **新旧码差异**：旧体系按检查项目整体计价（如免疫组化 140 元/只）；新体系**按切片数量阶梯计价**（如免疫组化 1–3 片 205 元/片、4–12 片 210 元/片、13 片起 105 元/片）（沟通记录 §1.2）。
- **一期自动 + 不常用手动补**：后端自动判断 IHC 常规 / 多重染色 / 特染 / FISH；增强染色、免疫荧光、其他化学染色、切片复制等手动补计费（沟通记录 §2.10、§2.11、§五）。

### 1.3 真实业务量（决定优化优先级的关键背景）

- **约 1454 case/月**。这是后面 FLOW-1 被列为最高优先级的根本原因：方案的核心交互（逐个"更新计费"+逐个"确认"）会被这个量级直接放大成采纳失败点。
- 本文优先级排序遵循 **adoption-first（采纳优先）** 原则：技术正确但因操作负担没人用 = 产品失败。

## 二、优化建议（按严重度 CRITICAL → LOW）

> 每条结构：**现状 / 问题 / 建议（落到改文档哪一节、加什么字段、补什么算例）/ 证据**。

---

### CRITICAL

#### FLOW-1　逐个"更新计费"+逐个"确认"是最大失败点

- **现状**：方案设计为用户在计费页面对每个病例手动点"更新计费信息"生成待确认项，再手动点"确认"逐个落账（沟通记录 §2.1、§2.14）。
- **问题**：按 **1454 case/月** 的真实量，逐个操作两次（更新 + 确认）= 每月数千次人工点击。这不是"略繁琐"，而是**直接决定系统会不会被用起来**——再正确的自动计费，如果日常操作负担过重，最终会被绕开或弃用（adoption-first）。这是整个方案能否落地的第一风险。
- **建议**（请在 v5.2 文档新增一节"§批量更新与批量确认"）：
  1. **批量更新**：支持按对账周期 / 登记日期 / 医院 / 登记类型筛选后**一键批量"更新计费"**，不强制逐个。
  2. **分流确认**：批量更新后系统把结果分两类——
     - **无差异 / 系统命中明确**的 → 支持**一键批量确认**；
     - **有差异（与原记录金额不一致）/ 未命中（缺输入字段、码无法判定）**的 → 才进入**逐个人工**复核队列。
  3. **默认收敛人工到少数**：让 80%+ 的常规 case 走批量直通，人工只处理异常队列。
  - **文档落点**：新增"§批量更新与批量确认"，并在 §2.14"计费记录状态说明"中补充"批量确认"的状态流转；UI 上给出"待人工复核数 / 可批量确认数"计数。
- **证据**：沟通记录 §2.1（手动逐个对账）、§2.14（逐个确认）；真实量 1454 case/月。

### HIGH

#### FLOW-3　医院"是否启用新码"的配置机制完全缺失（含生效起始月）

- **现状**：v5.2 前置检查第 1 步是"检查该医院是否启用新收费代码"（沟通记录 §2.12、§4.1 弹窗），但**如何配置"启用"这个状态，文档完全没定义**——这正是 §七待确认问题 1。
- **问题**：没有配置载体，前置检查就是悬空的判断；且现实中医院是**某个月开始切换**新码的，需要明确**生效起始月**——它决定历史 case 用旧码还是新码计费，否则切换月前后会算错账。
- **建议**（在 v5.2 文档新增"§医院新码启用配置"一节）：
  1. **医院档案新增字段**：`新码启用开关（布尔）` + `新码生效起始月（年月）`。
  2. **生效月语义**：登记月 ≥ 生效起始月的 case 走新码自动计费；早于生效月的历史 case 不走新码（沿用旧流程或不自动计费）。请在文档补**一个跨生效月的算例**（同一医院、生效月前 1 个 case + 生效月后 1 个 case，展示分别走哪条路径）。
  3. **维护角色**：财务 / 管理员（请在文档明确谁有权改这两个字段，并纳入审计——见 FLOW-4）。
- **证据**：沟通记录 §2.12（前置检查步骤 1）、§4.1（"当前医院未启用新收费代码"弹窗）、§七问题 1。
- **结论**：**一期必做**（前置检查的地基）。

#### FLOW-4　审计留痕未定义（合规风险）

- **现状**：多个动作都会改动金额——"确认"落账、冲突时"删旧增新"、"划横线模拟删除"（沟通记录 §2.13、§2.14）——但**文档没有定义任何审计字段**：谁操作的、什么时间、原值→新值是什么，都没记录。
- **问题**：账单是真金白银。改金额却无留痕，无法追责、无法对账复盘、过不了财务/合规审计。其中"划横线模拟删除"目前只是前端视觉效果，**本质应是一次有据可查的删除操作**。
- **建议**（在 v5.2 文档新增"§计费记录审计字段"一节）：
  1. **每条计费记录补审计字段**：`创建人/创建时间`、`最后修改人/修改时间`、`操作类型（更新/确认/删除/手动补项）`、`原值→新值`（金额、数量、收费码变更前后）。
  2. **"划横线模拟删除"落为软删除**：保留行 + `is_deleted` + `删除人 / 删除时间 / 删除原因`，而非纯前端划线。
  3. 配置类变更（FLOW-3 的启用开关 / 生效月）同样纳入审计。
- **证据**：沟通记录 §2.13（踢出对账单→作废→新增）、§2.14（不可删除、可划横线模拟删除）。
- **结论**：**一期必做**（合规底线）。

### MEDIUM

#### DSGN-1　多重染色"抗体数"与计价单位"每片"未对齐

- **现状**：多重染色判断依据是"**单条免疫组化记录的抗体数 ≥ 2**"（v5.3 修正，沟通记录 §2.9），但多重染色费（012100000120001）计价单位是"**每片**"（410/420/210 元/片）。
- **问题**：判断用的是"抗体数"，计价用的是"切片数（qty）"，两者口径不同。双染通常是**1 张切片上 2 个抗体**——若直接把"抗体数=2"当成 qty=2 计价，会**翻倍计费**。
  - *补充说明（避免夸大）*：方案的控制流已保证"一条记录只出一个收费码"，因此**不存在同一记录既计常规 IHC 又计多重染色的重复计费**；本条要纠正的是**数量取值口径**，不是重复计费。
- **建议**（在沟通记录对应 §2.9 / §3.2 染色费判断处补充"取数口径"）：
  1. 明确 **qty = 该记录的切片数**（双染通常 = 1 片），**不等于抗体数**。
  2. 给出"抗体数 → 切片数"的取数字段说明（哪个字段是切片数、哪个是抗体数）。
  3. 补**一个双染算例**：1 条 IHC 记录、2 抗体、1 张切片 → 多重染色费按 1 片计 = 410 元（而非 2 片 820 元）。
- **证据**：沟通记录 §2.8（多重染色定义：同一张切片上 ≥2 抗体）、§2.9（判断逻辑）；v5.2 附录多重染色 410/片。

#### DSGN-2　限时处理有判断公式但附录无对应收费码（死逻辑）

- **现状**：方案定义了限时处理 = (签发时间 − 登记时间) ≤ 24h，系统自动判断（沟通记录 §2.5）；但 **v5.2 附录价目表中找不到任何"限时/加急"对应的收费码**。
- **问题**：一个会自动跑、但永远落不到任何收费码上的判断 = 死逻辑。要么它本该加收却漏了码，要么它根本不该存在。
- **建议**（二选一，请在沟通记录 §2.5 处明确）：
  1. 若限时处理确实要加收 → **补上对应加急收费码**（该码是否存在 / 单价多少，**需回原始《20260605病理类项目收费代码-YZ.xlsx》核对**，附录未注明，本文不臆造）；或
  2. 若不加收 → **删除该判断逻辑**，避免误导开发实现一段无落账出口的代码。
- **证据**：沟通记录 §2.5（限时处理判断公式）；v5.2 附录（无加急码）。

#### FLOW-2　"增项"与"改值"被一刀切阻断（纯新增项被误挡）

- **现状**：反审核重新签发 / 补充报告的处理是"更新原记录数值，**或新增收费项**"（沟通记录 §2.6）；但前置检查第 2 步对"已确认且已生成对账单"的 case **一律弹窗阻断，不执行更新**（沟通记录 §2.12、§2.13、§4.2）。
- **问题**：阻断逻辑没区分两种动作——
  - **改值**（修改已落账金额）：确实该走"踢出对账单 → 重做"的严格流程；
  - **增项**（在原账单基础上**纯追加**一个新收费项，不动已落账金额）：被同一道阻断误挡了，而这恰恰是补充报告的常见正常场景。
- **建议**（在沟通记录 §2.13 冲突处理表里把单维度升级为"动作 × 对账单状态"矩阵）：

  | 动作 \ 对账单状态 | 未生成对账单 | 已生成对账单 |
  |---|---|---|
  | **改已有项的值** | 删旧增新 | 踢出对账单 → 重做（严格流程） |
  | **纯新增收费项（增项）** | 直接追加 | **允许追加**（不动已落账项，新增项进入待确认；可单独并入下一次对账） |

  - **文档落点**：替换 §2.13 现有 4 行表为上面的二维矩阵，并在 §4.2 弹窗逻辑说明"仅当涉及改值时弹阻断"。
- **证据**：沟通记录 §2.6（更新数值或新增收费项）、§2.12（前置检查步骤 2）、§2.13（冲突处理）、§4.2（阻断弹窗）。

#### FLOW-5　再次"更新计费"会静默覆盖人工已修正项

- **现状**：用户可重复点"更新计费信息"重算；冲突处理对"待确认 / 已确认未出对账单"的做法是"**删旧增新**"（沟通记录 §2.13）。
- **问题**：如果人工已经手动修正过某个待确认项（如改了数量），再次"更新计费"会按"删旧增新"**静默覆盖人工修正**，且用户毫无察觉——人工对账的成果被悄悄抹掉。
- **建议**（在沟通记录 §2.13 / §2.14 补充"更新前 diff"机制）：
  1. 再次更新前先做 **diff 比对**（系统重算值 vs 当前记录值）。
  2. 对**被人工改动过**的项（系统值 ≠ 当前值），**不静默覆盖**，列出 diff 让用户**二次确认**是否覆盖。
  3. 给记录加"是否人工修改过"标记，作为 diff 判定依据。
- **证据**：沟通记录 §2.13（删旧增新）、§2.14（待确认状态可被人工处理）。

#### DSGN-5　数量为 0 / 缺失时的处理口径未声明

- **现状**：阶梯计价全程假设有有效的切片数 / 蜡块数，但**没有声明 qty ≤ 0 或字段缺失时怎么办**。
- **问题**：缺字段 / 0 值会导致算出 0 元项、报错、或静默漏算——没有统一口径，开发各自实现会不一致。
- **建议**（在沟通记录 §三"自动判断逻辑"开头加一段**全局口径**）：
  1. `qty ≤ 0 或 该项数量字段缺失` → **不生成该收费项**（而非生成 0 元项）。
  2. `整列数量缺失`（如整个 case 取不到任何切片/蜡块数）→ **提示"无可计费数量"**，纳入人工复核队列（呼应 FLOW-1 异常分流），**不静默判 OK**。
- **证据**：沟通记录 §三（自动判断逻辑全程默认有效数量，未声明边界）；v5.2 附录（所有阶梯计价依赖数量）。

#### DSGN-4　复杂标本处理费"封顶"在附录未注明（需核对）

- **现状**：v5.2 附录中，复杂标本处理费（12100000040000）= 72 元/块，>5 块每增 1 块 +14，但**附录未注明加收封顶值**（对比常规处理费明确写了封顶 72）。
- **问题**：阶梯加收若无封顶，理论上可无限增长；与常规处理费有明确封顶的设计不一致，疑似漏写。
- **建议**：
  1. **复杂标本处理费是否有加收封顶、封顶值多少，需回原始《20260605病理类项目收费代码-YZ.xlsx》核对**（本文不臆造）。
  2. 核对结果**补进 v5.2 附录该行的"计费逻辑"列**，与常规处理费的封顶写法对齐。
- **证据**：v5.2 附录（复杂处理费未注明封顶 vs 常规处理费注明封顶 72）；沟通记录 §2.7（最高加收说明仅举常规处理费为例）。

#### OPEN-2　"踢出对账单"流程缺失（§七问题 2 的最小闭环）

- **现状**：冲突处理 / 阻断弹窗都指向"把病例**踢出对账单**再重做"（沟通记录 §2.13、§4.2），但**踢出对账单这一步本身的流程完全没定义**——这是 §七待确认问题 2。
- **问题**：阻断弹窗让用户"去单据页把病例移出单据"，但没有这个操作的权限、状态、金额、留痕定义，整条冲突处理链是断的，用户照弹窗指引也无路可走。
- **建议**（在 v5.2 文档新增"§踢出对账单流程"一节，先定义**最小闭环**即可，不求一步到位）：
  1. **权限**：谁能踢出（建议财务 / 管理员）。
  2. **状态回退**：被踢出的 case 从"已确认/已出对账单"回退到什么状态（可重新更新计费）。
  3. **金额重算**：踢出后对账单总额、该 case 计费项如何处理。
  4. **留痕**：踢出操作记录谁 / 何时 / 原因（接 FLOW-4 审计）。
- **证据**：沟通记录 §2.13（踢出对账单→作废→新增）、§4.2（弹窗指引"将该病例移出单据"）、§七问题 2。

### LOW

#### DSGN-3　诊断费阶梯计价在伪代码中未体现（文档完整度）

- **现状**：v5.2 附录里诊断费是阶梯的——病理诊断费 10 张以内 105/次，每增 10 张加收 42/次，最高加收 84；但 §3.3 诊断费判断伪代码只写了"外院会诊→远程诊断 / 其余→基础诊断"，**没体现张数阶梯加收**。
- **问题**：纯文档完整度问题（附录价目是对的），但伪代码不完整会让开发漏实现阶梯加收。
- **建议**：在 §3.3 伪代码补上**张数阶梯逻辑 + 一个算例**（如 25 张：105 + 42×2，并对照最高加收 84 验证封顶）。
- **证据**：v5.2 附录（诊断费阶梯 105 / +42 / cap84）；沟通记录 §3.3（伪代码未含阶梯）。

## 三、§七待确认问题处理建议

> §七列了 3 个待确认问题，逐一给处理建议与优先级。

### 问题 1：医院收费代码配置 —— **升为一期必做**

- **结论**：不能停在"待确认"，**一期必做**。
- **理由**：它是 v5.2 前置检查第 1 步的地基（FLOW-3）。没有配置载体，"医院是否启用新码"的判断悬空、§4.1 弹窗无依据。
- **做法**：见 FLOW-3——医院档案加`新码启用开关`+`新码生效起始月`，生效月决定历史 case 走旧/新码，维护角色财务/管理员并纳入审计。

### 问题 2：踢出对账单操作流程 —— **一期做最小闭环**

- **结论**：**一期必做最小闭环**（不求完整功能，但必须打通）。
- **理由**：它是整条冲突处理链的出口（OPEN-2）。阻断弹窗已经把用户指向这一步，却无流程，用户会卡死。
- **做法**：见 OPEN-2——先定义 权限 / 状态回退 / 金额重算 / 留痕 四要素的最小闭环。

### 问题 3：同类抗体批量配置收费规则模板 —— **可保留为后续**

- **结论**：**可保留为后续**，一期不做。
- **理由（重要纠正）**：一期的**染色判断不依赖"抗体→类别"主数据**。判断走的是**记录上的特染/FISH 标记 + 该记录的抗体数**（沟通记录 §2.9、§3.2）——即"单条 IHC 记录抗体数 ≥2 → 多重染色；记录上有 FISH 标记 → 分子病理；有特染标记 → 特染；否则 IHC 常规"。这条链**不经过"抗体批量配置成某类别"的主数据**。
- **因此**：批量配置模板**不是 §3.2 染色判断的前置条件**，把它推迟到后续不会阻断一期。它的价值是减少前端逐抗体配置基础信息的工作量，属于体验优化，可在一期跑起来后再做。
- **证据**：沟通记录 §2.9（判断针对单条记录抗体数）、§3.2（染色费判断用标记+抗体数）、§2.10/§2.11（前端只配基础信息、规则后端判断）。
