## 0. 一句话目标

把「实验室收入 = Σ(IN 业务线服务项的**结算额**)，结算 = 开单 × 扣率，逐院配置规则驱动分类」做成可信的后端引擎 + 导入器（测试台 + 月度向导）+ 三页真前端，替换掉现在「实收 × 技术占比」的旧口径。**产品未上线，质量优先于速度**（adoption-first）。

## 1. 锁定的产品决策（不要重新讨论）

- **系统边界（IN/OUT）**：IN = 组织学 / 细胞·宫颈TCT / 院内冰冻 / 线下外院会诊(H 开头，含共建院送来的)。OUT = 外送分子(NGS/FISH/HPV-E6E7，病理号 M/Q/F) / 远程诊断(含每月固定保底费 retainer) / 共建分成净额 / 科研。共建院的「线下 H 会诊」留 IN，只有「共建分成结算」才 OUT。
- **收入口径**：实验室收入 = Σ(IN 服务项结算额)。结算(实收) = 开单 × 扣率，扣率 ∈ 0–1，逐院（默认 / 按线 / 按项三级，细盖粗）。看板一律未税。账单未到 → 退化「估算」(回款 × 经验占比) 并显式标注。三态：已对账(实际) / 估算 / 已修正。
- **导入器 PM 4 决策**：① 先 mockup 定稿（已完成）→ 写真码；② 解析=**混合**（通用列映射+逐行分类覆盖逐项明细类；类别汇总+明细子表、共建利润分成走**专用解析器**）；③ 规则归属=**有配置权限的财务可直接改规则、立即生效**（无审批队列）；④ v1 范围=**一次覆盖全部 7 模板 / 19 家**。
- **测试台定位**：**保留独立页，但读每家医院自己的配置规则、改规则写回该院 config（单一事实源）**。测试台 + 配置页 + 月度向导共享同一份逐院配置。

## 4. 架构总览

```
[前端] 读 xlsx → 2D 网格 ──POST──┐
                                  ▼
[后端] 解析层(P1) 模板识别 + 通用/专用解析器 → 规范化行[{病理号,项目名,备注,开单,结算?,院区?}] + declaredTotal(独立合计)
          ▼
       分类引擎(P2) 逐院 config.lines → classify(前缀>关键词/备注, IN/OUT, 歧义/未匹配)
          ▼
       收入(P2) 结算=开单×扣率(逐院三级) → 实验室收入=Σ(IN结算) + 逐线拆分
          ▼
       评分/完整度(P3) 识别率 / 对账闭合(Σ开单 vs declaredTotal) / 病例匹配双向(行↔LIS) / 黄金值
          ▼
   /preview(干跑,不落库) ──→ 测试台 & 月度向导预览
   /commit(落库 case_revenue + 回填 partner_id + 重算 partner-pnl)
          ▼
[看板] partner-pnl(P5) 已对账=Σ(IN结算) / 无账单=估算fallback / 完整度双向标注
```

**逐院配置(P0)** 是贯穿一切的单一事实源：解析(模板/列映射)、分类(lines/规则/scope)、扣率、特殊结算、变更记录/版本。配置页(P6a)、测试台(P6c)、月度向导(P6d) 都读它、可改它。

### P0 — 逐院配置数据模型（单一事实源）｜地基
**目标**：每家医院一份**版本化**配置（lines+规则+scope、三级扣率、模板/列映射、计税、特殊结算、变更记录），可回滚、可按版本追溯重算。

- **存储方案（推荐 B：版本化 JSON blob + 变更表）**——与 mockup 配置对象 1:1、回滚=恢复某版本 JSON、契合「按版本追溯重算」（仿本分支 `bom_versions` 范式）：
  - `partner_configs(partner_id, version, config_json, is_current, is_baseline, created_at, created_by)` —— `config_json` = mockup 的配置对象（basic/amount/parse/lines/discount/special）。
  - `partner_config_changes(partner_id, version, changed_at, changed_by, tab, diffs_json, snapshot_json)` —— 逐字段「调整前→调整后」+ 快照（回滚源）。
  - 备选 A（规范化多表 partner_lines / partner_line_rules / partner_discount_rules / partner_settings）：若后续有跨院规则统计需求再拆；v1 不需要。
- **迁移/seed**：既有 `partners` 无配置 → 首次访问按默认模板生成（8 条默认线 = 组织学/细胞·宫颈TCT/院内冰冻/线下外院会诊[in] + 外送基因NGS/荧光原位FISH/远程诊断/共建分成净额[out]，默认扣率占位）。沿用 DatabaseManager `CREATE TABLE IF NOT EXISTS` + `ensureColumn` 兼容旧库。
- **case_revenue 加 `config_version` 列**：每期导入记下用的配置版本 → 改规则后可判影响面 + 追溯重算。
- **文件**：`database/DatabaseManager.ts`（建表）、`utils/partner-config.ts`（load/save/diff/rollback/seed 默认，纯函数 + DB loader，复用 config mockup 的 deepDiff/fLabel/classify 思路）。
- **测试**：`tests/partner-config.test.ts`（默认 seed / 保存生成版本+变更 / 回滚 / 版本隔离逐院）。
- **验收**：逐院 config CRUD + 版本化跑通；既有 golden 零回归。
- **粗估**：1–2 天。

### P1 — 解析层（混合，7 模板）
**目标**：网格 → 规范化行 + 独立声明合计(declaredTotal) + 模板识别。

- **输入**：前端 `xlsx`(SheetJS, 前端已装) `sheet_to_json {header:1}` → 2D 网格 POST 给后端（避免后端加 xlsx 依赖；结构性模板的解析逻辑仍在服务端权威）。
- **解析器注册表** `utils/statement-parser/`：
  - `detectTemplate(grid)` → 7 模板之一（按表头/结构特征）。
  - 通用 `lineItemParser(grid, colMap)` —— 复用/重构 `billing-revenue.ts` 逻辑（detail-row 检测、monthOf、扣率列、code-agnostic），覆盖：逐项明细 / 会诊远程 / 诊断服务费 / 服务费混合 / 外送明细（5/7）。
  - 专用 `categorySummaryParser`（类别汇总+明细子表：明细子表求和 vs 类别汇总行，declaredTotal=类别汇总）。
  - 专用 `jointVentureParser`（共建利润分成：抽出「分成净额」单列为 OUT 科目 + 线下 H 会诊行为 IN）。
  - 每解析器产出 `{ rows:[{no,item,remark,bill,settle?,campus?}], declaredTotal, template }`。
- **declaredTotal** 必须来自对账单**独立声明的合计/小计**（不是逐行求和），用于真对账闭合。
- **fixtures**：康湾 19 院 60 文件（`scratchpad/duizhang` 解压；用 venv openpyxl 转网格 JSON 存入 `tests/fixtures/statements/`）。每模板至少 1 真实样本。
- **文件**：`utils/statement-parser/index.ts` + 各 parser；`tests/statement-parser.test.ts`。
- **验收**：7 模板各 1 真实文件解析出正确 rows + declaredTotal；通用解析与现 aggregateBilling 行为对齐（回归）。
- **粗估**：2–3 天（真实文件适配是主要不确定性）。

### P2 — 分类引擎（重写 revenue-attribution seam）
**目标**：逐院 config.lines → 每行分类 IN/OUT；结算=开单×扣率；实验室收入=Σ(IN结算)。

- `utils/classifier.ts`：`classify(lines, {no,item,remark})` → `{line, scope, by}` | `{amb:[...]}` | `{none}`。**前缀优先 → 关键词/备注；NFKC 归一；空值跳过；歧义=命中多条**。直接移植 config/测试台 mockup 的 classify（已对抗审过）。**这替换 `revenue-attribution.ts` 的 A/B/C 模型**（旧文件可标注废弃或重构为薄封装）。
- `utils/statement-revenue.ts`：`computeStatementRevenue(rows, config)` → 逐行分类 + `settle = round(bill × rate(line/item/default))` + `labRevenue = Σ(IN settle)` + outSum + 逐线拆分 + unmatched/ambiguous 计数。**扣率三级**（按项 > 按线 > 默认）复用 config。
- **黄金锚**：`tests/golden/statement-revenue.golden.test.ts` —— 和睦家真实样本：实验室收入(实收) = **¥13,152**（= Σ(IN 开单 × 扣率)；康湾对账单「结算金额 = 医院收费 × 扣率 = 实收」）。逐行手算交叉核。
- **测试**：classifier 单测（7 模板代表行 + 边界：全角 H、冰前缀、共建分成净额 OUT、H 会诊 IN）；statement-revenue 黄金 + 三态。
- **验收**：和睦家黄金绿；既有 golden 零回归。
- **粗估**：1–2 天。

### P3 — 评分 / 完整度（体检卡）
**目标**：把测试台 v2 的四项体检做成后端可信判定。

- `utils/import-score.ts`：`scoreStatement(rows, config, ctx)` →
  - **识别率** = (行数 − 未匹配 − 歧义) / 行数。
  - **对账闭合** = |Σ开单 − declaredTotal| < 阈值（**独立合计**，抓漏读行）。
  - **病例匹配（双向）**：正向 = IN 行 ↔ `lis_cases`(按病理号) 命中率；反向 = 本期该院 LIS 计入病例中**对账单未覆盖**的数（= 看板「估算/待对账」口径）。
  - **黄金值** = 算出实验室收入 vs 财务录入的期望实收（外部值；不符=纠错信号）。
  - **status**：todo（任一未过）/ review（全过待人工核对）/ ready（已人工核对设为基线）。
- **LIS 双向 join**：正向 join lis_cases by (partner, 病理号)；反向 = lis_cases 该期该院 IN 病例集合 − 已覆盖。
- **文件**：`utils/import-score.ts`；`tests/import-score.test.ts`（构造对账不平 / 黄金不符 / 病例缺口三失败场景，对齐 v2 mockup 的演示样本）。
- **验收**：三失败场景如期 todo；全过场景 review；零回归。
- **粗估**：1 天。

### P4 — 后端 API（config + import）
**目标**：把 P0–P3 接成路由，供三页前端调用。RBAC：config 写操作限 finance/admin（配置权限）。

- `routes/partner-config-v1.1.ts`：`GET /:id`（含默认 seed）、`PUT /:id`（保存生成版本+变更）、`GET /:id/changes`、`POST /:id/rollback`、`POST /:id/baseline`（设为月度导入基线）。`requirePermission` 门控。
- `routes/statement-import-v1.1.ts`：
  - `POST /preview`（dry-run）：网格 → 解析 → 分类 → 评分 → 返回 preview（**不落库**）。测试台 & 月度向导共用。
  - `POST /commit`：解析 → 分类 → 写 `case_revenue`/`case_revenue_lines`（复用/扩展 aggregateBilling，带 config_version + period）→ 回填 partner_id → 触发 partner-pnl 重算。幂等（按 caseNo+serviceMonth）。
  - `POST /classify-rule`：把某行归类写回**该院** config.lines（加识别词/新建线含 scope）+ 记一条变更 → 立即生效。供测试台内联归类。
- 挂 `app.ts`，保持 `/api/v1/` 前缀 + 统一 `success/data/error`。
- **测试**：`tests/partner-config-routes.test.ts` + `tests/statement-import-routes.test.ts`（preview 干跑不落库 / commit 幂等 / classify-rule 只动该院 / RBAC finance 写 OK·pathologist 403）。
- **验收**：路由全绿；零回归。
- **粗估**：2 天。

### P5 — partner-pnl 收入侧改造
**目标**：看板数字改为配置驱动；保留估算 fallback + 完整度双向。

- `partner-pnl.ts`：`computeCasePnl` 拆两路：① **已对账**（有对账单 case）→ 实验室收入 = 该 case 的 Σ(IN 结算)（来自 statement-revenue）；② **估算**（LIS-only 无账单）→ 沿用现「实收 × 占比」逻辑（明确标 `estimated`）；③ **已修正**（人工覆盖留痕）。完整度：双向（账单缺 / LIS 缺）显式标注。

- **测试**：`tests/partner-pnl-statement.test.ts`（三态 + 双向完整度 + 院级上卷）。
- **验收**：和睦家端到端（配置→导入→看板）一致；golden 零回归。
