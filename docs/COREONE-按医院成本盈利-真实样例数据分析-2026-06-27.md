## B. 财务对账单 / 收费单据（W4 输入）——「金额真值」层【用户已重传准确文件】

> ⚠️ 第一份 `对账单详情列表.xls`(单行) 用户确认不准；**准确文件 = `单据2026062607544902号收费单据.xls`**。

- **17 列逐明细行 + 每 case 小计 + 末尾总计**：`序号|病理号|送检医院|登记类型|姓名|性别|标本名称|收费项目|收费代码|单价|数量|单位|计费金额|扣率|(开单)单价|开单金额|计费时间`。
- **本单据 = 上海和睦家医院 · 组织病理 · 25 case · 服务月 2026-06**；计费 ¥15840 → **实收(开单) ¥13152**，整体扣率 **0.83**。
- **关键发现**：
  1. **`开单金额 = 计费金额 × 扣率`**（80%/100%）→ **开单金额 = 折后实收**，即拆分用的金额真值。逐 case 实收 = Σ 明细行开单金额 = 小计行开单金额。
  2. **收费代码是旧码/院定制码**（`270300002b`内镜活检·`270900099-2`外籍人士中英文报告·`270300005a`手术标本小标本·`270500002a/b`免疫组化前八项/超八项·`270500001-1`特殊染色）——**非新 81 码目录** → 印证锁定模型「比例永远用新目录引擎+LIS数量算（含旧码/私立）」。**v1 code-agnostic：只取 case 实收总额，不解析这些码**；原始明细行原样存(`case_revenue_lines`)备 v2 逐码交叉核对。
  3. **病理号 100% 命中 LIS 导出**（25/25 在 LIS 找到）→ 按 `病理号` 匹配可行；服务月取 `计费时间`。
  4. **扣率 per 行可见**（和睦家：常规项 80%、外籍报告 100%）→ 是真实的「量价/扣率」数据，v1 存下(`discount_rate`)，扣率模拟分析 v2。
- **⚠️ 重要软边界（引擎拆分的局限）**：和睦家多为常规活检，LIS **HE切片数=0、IHC=0**（仅蜡块数>0）→ 新目录引擎对这些 case 只算出组织处理费、技术占比≈1.0、**诊断/技术分解退化**（真实账单里 ¥150×16 报告费=诊断，引擎却归不到诊断）。**结论**：① 院级**收入总额**(=Σ实收)可靠；② 但「技术 vs 诊断」**组分分解**在 LIS 数量稀疏时不可信 → 决策用院级毛利(收入−成本)，组分分解标「未校正」。③ `with_diagnosis` 医院(如和睦家，本中心出报告)实验室收入=全额实收（拆分自然 collapse 到全额，数对）；`technical_only` 医院才真正依赖技术占比拆分。
- **登记类型** = 组织病理（本单据全部）；其他类型(细胞病理等)的单据结构预计一致。

## H. 增量纠错架构（用户确认 2026-06-27——勿当临时补丁）

> 真实常态＝**数据不完整 + 业务周期长（case 登记→加做 IHC→签发→月度开单跨数周/月）+ 判断逻辑需持续纠正**。按下列架构处理（记忆 `coreone-incremental-correction-architecture`）：

1. **三层分离**：原始事实层(不可变·幂等 upsert) / 派生推断层(纯函数·随时全量重算) / 呈现层(带完整度状态)。逻辑改了只**重跑**，原始不动。
2. **推断＝数据驱动 + 逐 case 可覆盖 + 留痕**：`specimen_type` 等落字段标 `source=auto|manual`，人工改 manual 永远赢且写对账留痕；模板/目录入库可改不发版。
3. **真实 case 黄金回归集**：用户确认一个真实 case 期望产出就锁进 golden（仿 abc-golden-accuracy），改逻辑先跑 golden 防回归。**首锚＝和睦家 2026-06 实收 ¥13152/25 case**（`tests/golden/partner-revenue-golden.test.ts`）。
4. **显式生命周期/完整度**：进行中/已签发/已开单/已对账；月结前=标准估、月结=实际核实；缺口绝不静默当 0。
5. **版本化 + 追溯重算**：沿用 `bom_versions` + 对账 `retroactive`（future-only 或重算关账月转调整单）。
6. **对账差异＝自动纠错信号**：引擎标准 vs 账单实收之差 + 匹配不上 case → 自动进"待复核清单"，系统自指判断逻辑该修哪。
