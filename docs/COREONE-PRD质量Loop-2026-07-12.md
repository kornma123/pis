# COREONE PRD 质量 Loop（产品需求整理）

> **状态**：v1.0 草稿（2026-07-12）· 按「产出必过质疑关」规矩，PM 质疑/拍板后生效
> **从属关系**：本文从属于工作模型（方法论主线）与共用契约，是其在**需求整理 / PRD 产出阶段**的执行流程细化，冲突时以工作模型与契约为准。与《COREONE-前端Mockup质量Loop-2026-07-12.md》配套：**PRD 定稿 = Mockup Loop 的输入**。
> **适用**：所有新功能、大改动的需求整理。已定稿需求的小补充、纯执行任务不走本 loop。

## 0. 在工作流里的位置

```
想法/痛点 → 【PRD 质量 Loop（本文）】→ PRD 定稿 → Mockup 质量 Loop → mockup 定稿 → 写真代码 → 真跑验收
```

PRD 阶段的唯一使命：**在画任何界面、写任何代码之前，把「解决什么问题、为谁解决、做到什么样算成」磨到 PM 拍过板。**
专治两个老毛病：AI 误读需求后「和 PM 自信地一起错」；PM 靶子没说清就开工、做完才发现不是那个意思。

## 1. Loop 全景

```
┌─> ① 产出 / 修订 PRD 草稿
│   ② AI 自检（三张清单，贴勾选结果）
│   ③ 摊假设（含「解读型假设」：你说的 X，我理解为 Y）
│   ④ 自我质疑（先攻击自己的稿子，能修的当轮修掉）
│   ⑤ 交付 PM（固定五件套）
│   ⑥ PM 反馈归类：拍板 / 要改 / 新讨论
└── 有「要改」或「新讨论」→ 回到 ①
    全部拍板 + 退出条件满足 → 定稿归档 → 才交棒给 Mockup Loop
```

每一轮都完整走 ①–⑥，不跳步。只交 PRD 不摊假设 = 无效轮。

## 2. 每步做什么

### ① 产出 / 修订 PRD 草稿

固定骨架（轻量起步，一页写得下就一页）：

1. **问题**：用一个真实发生过的例子描述（哪家医院/哪个月/什么场景卡住了），不用抽象概括开头
2. **用户与场景**：谁会用、多久用一次、不做这个功能他现在的土办法是什么
3. **方案范围**：做什么 + **明确不做什么**（负面清单）
4. **关键口径**（碰钱/碰数必写）：每个数字怎么算，标 ①有真值照算 / ②约定值不下沉 / ③算不出绕开
5. **验收标准**：每个功能点配至少一个**真数据例子**（真实输入 → 期望看到什么），能让第三方判 yes/no
6. **依赖与风险**：需要的数据现在有没有、动到哪些现有口径
7. **开放问题**：没想清楚的全列出来，不许假装已解决

硬要求：
- 用真实业务例子说话（真医院名、真项目、真金额量级），禁「某医院」「某项目」
- 禁空话：「提升效率」「优化体验」这类词必须落地成可观察的行为变化，落不了地就删
- 修订轮开头一句话列出「这轮改了什么」

### ② AI 自检——三张清单逐项打勾（勾选结果贴进交付）

**清单 A · 需求完整性**
- [ ] 问题有真实证据（谁说的 / 哪个例子 / 哪来的数据），不是推测出来的痛点
- [ ] 消费者明确：谁会用、什么时候用、不用会怎样——回答不了就是「功能先于消费者被建」
- [ ] 「不做什么」写了，且 PM 看得到（范围没说死 = 将来扯皮）
- [ ] 每个功能点有对应验收例，没有「只有描述没有例子」的条目
- [ ] 开放问题全列了，零隐藏

**清单 B · 口径与数据（碰钱/碰数必查）**
- [ ] 每个数字口径标了 ①/②/③ 档，没有含糊的「系统自动计算」
- [ ] 与现有已拍板口径（黄金锚 / ADR）比对过，冲突处显式列出让 PM 拍，不静默覆盖
- [ ] 边界有答案：数据缺失显示什么、算不出是显示 0 还是「无法计算」（诚实降级，禁编数）
- [ ] 需要的数据现在系统里有没有；没有的话，补数据本身算不算这个需求的一部分

**清单 C · 文案与可判断性**
- [ ] 功能名/字段名/状态名说人话，黑名单零命中（上卷 / 下卷 / IN / OUT / ABC / P&L 等行话）
- [ ] 每条验收标准可判断（第三方读了能判 yes/no），无「体验良好」「基本可用」这类判不了的
- [ ] 复述测试：用 PM 的语言把需求核心重述一遍，附在 PRD 开头，等 PM 确认「就是这个意思」

### ③ 摊假设

列「本轮我替你做的决定」清单，每条标 `待拍板`。PRD 阶段特设一类**解读型假设**：

> 你说「对账要方便一点」，我理解为「导入后自动匹配、只把匹配不上的留给人工」——是这个意思吗？

PM 的每句模糊表述都要有对应的解读条目。**AI 自由解读的地方 = 将来「做出来不是我要的」的地方**，全部摊出来。

### ④ 自我质疑（质疑关的 PRD 版）

至少从这几个角度攻击自己的稿子，能修的当轮修掉，修不掉的进讨论项：
- 这问题是真的吗？证据够硬吗？（防伪需求）
- 做出来谁会用？他为什么不继续用现在的土办法？（采纳优先于技术正确）
- 能不能更小？哪些砍掉 v1 也成立？（别造复杂度，塞不下就拆期）
- 是不是已有功能改改就行？（先查现状再立新项，别重建现成的）
- 真实数据撑得起这个口径吗？（纸面成立、真数据一算就翻车的先例已有）

### ⑤ 交付 PM（固定五件套）

1. PRD 草稿本体（含开头的复述段）
2. 「这轮改了什么」一句话清单
3. 假设清单（含解读型假设，标待拍板）
4. 讨论项（修不掉的、拿不准的）
5. 结尾大白话：这功能给谁用、解决什么问题、做完后用户的一天有什么不同

### ⑥ PM 反馈归类

- **拍板**：记进「已拍板决定」章节，之后不许再改（要改 = 新讨论项重新拍）；碰钱/口径的重要决定沉淀成 ADR
- **要改**：进下一轮修订
- **新讨论**：讨论是循环不是一问一答——回答里长出来的新问题继续循环，不许假装没看见

## 3. 退出条件（全部满足才算「定稿」）

1. PM 明确说了「定稿」或等价表述——**沉默不算同意**
2. 假设清单（含解读型）零「待拍板」残留
3. 三张清单全绿（最后一轮交付里贴勾选结果）
4. 每个功能点都有真数据验收例
5. 开放问题清零，或逐条标「留到实现阶段再定」且 PM 逐条点头
6. 最近一整轮 PM 没有提出任何「要改」和「新讨论」

**定稿动作**：归档（PRD 终稿 + 已拍板决定清单，写明日期版本）。**PRD 定稿 = Mockup 阶段的唯一输入基准**；下游发现需求有问题 → 回来改 PRD 重新拍板，不许在 mockup 或代码里自由发挥。

## 4. Loop 护栏（防跑偏、防空转、防烂尾）

| 护栏 | 规则 |
|---|---|
| 红线 | PRD 定稿前不进正式 Mockup Loop、不写代码；讨论中可以画示意草图帮 PM 看懂选项，但草图只是讨论道具，不是交付物 |
| 防空转 | 每轮必须有可见差异；连续 2 轮没实质变化 = loop 卡住，直接升级问 PM |
| 防打转 | 同一议题改到第 3 轮还没收敛 → 停止再猜，把 2–3 个候选方案（各带真数据例子和利弊）**并排摆出来**让 PM 选 |
| 防偷改 | 已拍板决定不许在后续轮里悄悄改回去 |
| 防烂尾 | 每轮交付必须完整走 ①–⑥；缺假设清单或缺自检勾选 = 无效轮，重交 |
| 防大而全 | 一份 PRD 只服务一个可独立验收的目标；塞不下就拆期，每期单独走 loop |

## 5. 碰钱 / 碰口径需求的加固

对账、成本、毛利、预算类需求，定稿前在本 loop 之外加三道：

1. **真数据试拍**：口径值和规则先拉真实数据算一遍再拍板，别拍纸面值（绝对值在真数据面前翻车的先例已有）
2. **异构复核**：codex（`codex exec -s read-only`）或 Claude 对抗面板，专攻「与现有黄金锚冲突、边界漏洞、被高估的数据可得性」
3. **ADR 沉淀**：重要口径决定单独成 ADR 编号归档，供后续所有阶段引用

三道都过完才算定稿。

## 6. 会话注入块（开工时贴给 AI）

新开需求整理任务时，把下面这段贴进会话即可激活本 loop：

```
【PRD 质量 Loop 已激活】本任务处于需求整理阶段，按 docs/COREONE-PRD质量Loop-2026-07-12.md 执行：
1. PRD 未定稿前不进 mockup、不写代码；讨论中可画示意草图，但只是讨论道具。
2. 每轮交付固定五件套：PRD 草稿（开头带复述段）、本轮改动清单、假设清单（含解读型）、讨论项、结尾大白话。
3. 交付前过三张自检清单（需求完整性/口径数据/文案可判断性）并贴勾选结果。
4. 我的每句模糊表述，先写「你说的X，我理解为Y」让我确认，再落进 PRD。
5. 验收标准必须带真数据例子，能判 yes/no；写不出例子的功能点先别写进范围。
6. 我说「定稿」之前循环不结束；同一问题 3 轮没收敛就并排出候选方案让我选。
```

## 7. 本文档自身的待拍板项

| # | 事项 | 建议 |
|---|---|---|
| 1 | PRD 七段骨架是否固定 | 建议固定为默认、允许按需减段（减段要说明为什么），PM 拍 |
| 2 | 「开放问题留到实现阶段」的口子 | 现写为需 PM 逐条点头才能留，是否太严 PM 拍 |
| 3 | 碰钱/口径需求的三道加固 | 现写为强制，是否降为建议 PM 拍 |
| 4 | PRD 归档位置 | 建议 `docs/prd/<功能名>.md`，或延续现有 docs 平铺，PM 定 |
| 5 | 与 Mockup Loop 文档互相指回 | 建议两文互加一行链接（生效时一起改），PM 拍 |

## 8. 给 PM 的大白话

这份文档给「整理需求、写 PRD」定了和画草稿图同一套规矩：AI 每轮都要自己过三张检查清单，把它对你原话的理解逐句摊开（「你说的X，我理解为Y，对吗？」）让你确认，每个功能都必须配一个真实数据的例子来说明「做到什么样算成」，写不出例子的就不许写进范围。你没说「定稿」之前不许开始画界面、更不许写代码；涉及钱和算法口径的需求，拍板前还要先拿真实数据算一遍、再找第二个 AI 独立挑刺。这样最容易出大返工的「做出来才发现理解错了」会被消灭在写需求这一步——那时候改的只是几段文字。
