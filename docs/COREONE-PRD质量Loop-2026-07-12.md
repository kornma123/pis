# COREONE PRD 质量 Loop（薄入口 · 需求整理）

> **状态**：v1.0（2026-07-12）· 待 PM 定稿拍板
> **从属**：本文是《`docs/COREONE-质量Loop契约-2026-07-12.md`》在 **PRD/需求整理阶段**的薄入口——只写本阶段增量。**六步骨架、五件套、假设两级、异步节奏、证据分级 A/B/C、复核堆栈、通用定稿退出模板、护栏全部在契约，本文一律不复制。** 冲突时以契约与其上游（工作模型/共用契约）为准。
> **适用**：新功能/大改动的需求整理。已定稿需求的小补充、纯执行任务不走本 loop。

## 0. 位置

想法/痛点 →【PRD（本文）】→ PRD 定稿 → Mockup → 写码 → 真跑验收。
使命：画界面/写代码前，把「解决什么、为谁、做到什么算成」磨到 PM 拍过板。专治：AI 误读需求「自信地一起错」；PM 靶子没说清就开工、做完才发现不是那意思。

## 1. 本阶段增量（契约之外，PRD 特有）

**① 产出骨架（七段，一页起）**：① 问题（真实例子：哪家医院/哪月/什么场景卡住，禁「某医院」）② 用户与场景（谁用/多久用一次/现在的土办法）③ 方案范围（做什么 + **明确不做什么**）④ **关键口径**（碰钱/碰数必写）：按契约 §5 给每个数字标等级 + 源（规则以 §5 为准、本文不复述）⑤ 验收标准（每功能点配**真数据例子**：真实输入 → 期望看到什么，第三方能判 yes/no）⑥ 依赖与风险 ⑦ 开放问题（全列，不装已解决）。用真业务例子、禁空话（「提升效率」落不成可观察行为就删）、修订轮开头一句「这轮改了什么」。

**② 解读型假设（PRD 特设，是契约 §3「摊假设」的一类）**：PM 每句模糊表述都写「你说 **X**，我理解为 **Y**——对吗？」，等确认再落进 PRD。AI 自由解读处 = 将来「做出来不是我要的」处，全摊出来。

**③ 自检清单（过完贴勾选结果，清单指回权威源不复制）**：
- 需求完整性：问题有真实证据 / 消费者明确（谁用·何时·不用会怎样，答不了 = 功能先于消费者被建）/ 写了「不做什么」/ 每功能点有验收例 / 开放问题零隐藏。
- 口径与数据：每数字标 A/B/C + 源（契约 §5）/ 与黄金锚·ADR 比对，冲突显式列不静默覆盖 / 边界诚实降级（缺数据显示 0 还是「无法计算」，禁编数）/ 需要的数据现在系统里有没有。
- 文案可判断性：**过前端标准 §4.2 术语表（说人话唯一源，本文不枚举黑话词）** / 每条验收可判断 / 开头附「复述段」等 PM 确认「就是这个意思」。

**④ 自我质疑角度（PRD 版，是契约 §1 ④ 的本阶段落地）**：这问题真的吗、证据够硬吗（防伪需求）/ 做出来谁用、为什么不继续用土办法（采纳优先）/ 能不能更小、砍掉 v1 也成立吗 / 是不是已有功能改改就行（先查现状）/ 真实数据撑得起这口径吗（纸面成立、真数据翻车有先例）。

> **PRD 范围纪律（防大而全）**：一份 PRD 只服务一个可独立验收的目标；塞不下就拆期、每期单独走 loop。

## 2. PRD 特有退出追加项（叠加契约「通用定稿退出模板」，不重写那四条）

- 每功能点都有真数据验收例；
- 碰钱/口径数字的 grade-C 退出与「PM 接受签字 → 自动入队」规则**见契约 §5、本文不复述**（本阶段只须保证台账里每个决策数字都带等级 + 源）；
- 开放问题清零，或逐条 PM 点头「留到实现阶段再定」。

**定稿动作**：归档（PRD 终稿 + 已拍板决定，写日期版本）。**PRD 定稿 = Mockup 阶段唯一输入基准**；下游发现需求有问题 → 回来改 PRD 重拍，不在 mockup/代码里自由发挥。

## 3. 碰钱/口径需求加固（叠加契约）

定稿前三道：① **真数据试拍**（口径值/规则先拉真实数据算一遍再拍，别拍纸面值——试拍结果即口径台账的等级 + 源）② **L1 定向对抗复核**（按契约 §6，镜头/产物规则以 §6 为准；PRD 关注点 = 与黄金锚冲突、边界漏洞、被高估的数据可得性）③ **ADR 沉淀**（重要口径决定单独编号归档）。

## 4. 会话注入块（开工贴给 AI）

```
【PRD 质量 Loop 已激活】按《质量Loop契约》+ docs/COREONE-PRD质量Loop-2026-07-12.md（薄入口）执行：
1. PRD 未定稿前不进 mockup、不写代码；讨论中可画示意草图（只是道具）。
2. 轮次自主：每轮 产出→自检→摊假设(细节带标前进/方向禁止→出候选)→自我质疑→写台账；不逐轮找我，到闸点才清算(契约§4)。
3. 我的每句模糊表述先写「你说X，我理解为Y」等我确认。
4. 碰钱/碰数的每个数字带等级+源；A=对外部真实数据核过(自洽/测试绿都不算)；支撑决策的数字仍B/C时不许标定稿，除非我明写「接受grade C」(契约§5)；碰钱口径定稿前过L1定向对抗留定式产物(契约§6)。
5. 验收标准带真数据例子(能判yes/no)；文案过前端标准§4.2术语表。
6. 我说「定稿」前循环不结束；同一问题3轮不收敛并排出候选让我选。
```

## 5. 给 PM 的大白话

整理需求这一段：AI 每轮自己过检查清单、把它对你原话的理解逐句摊开让你确认、每个功能配一个真实数据例子说明「做到什么算成」；涉及钱的数字必须标明「是真数据验过的、还是听来的、还是猜的」并写出处，猜的想定稿你得亲手签字接受。你没说「定稿」前不许画界面、更不许写代码。这样最容易出大返工的「做出来才发现理解错了」，被消灭在只改几段文字的阶段。
