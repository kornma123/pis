# COREONE 「按医院(客户)成本/盈利」—— 调研结论 + 数据模型设计 + gap

> **生成**: 2026-06-27 · **方法**: deep-research `wf_8908e509-227`（5 路/25 源/102 claim → 25 条 3 票对抗验证 → 24 确认 1 证伪 / 107 子代理）+ 当前系统代码实测映射（Explore）
> **目的**: 让系统可信回答「每个合作医院的成本、收入、利润各是多少」。聚焦"模型该怎么造"。
> **边界**: 本文档只沉淀设计 + gap，**不动代码**。改不改/怎么改由用户拍板。配套现状底图见本文 §2。

## 0. TL;DR

1. **价格必须 key 在「客户 × 检测」上，不是只 key 在检测上**（同一检测对不同医院价不同＝行业常态，非异常）。当前系统 `fee_standards`/`bom_fee_mappings` 全局单一口径 → 缺「客户」维度。
2. **检测项目身份（identity）与收费代码+价格（billing/price）要分两层**：身份用国家标准码/项目（LOINC↔CPT 单向映射证实二者分离；中国 15 位码只表身份不含价），**per-client 的收费代码与价格是叠加在身份之上的可配置层**——正合你"前端可配置 per-hospital 收费代码"的要求。
3. **分工：LIS 给事实，本系统配定价**（HL7 订单段不带金额，金额在独立 billing 系统）——正合你"医院信息可经 LIS 导入、收费规则在本系统配"的设想。
4. **盈利用 ABC 上卷 + 多级贡献毛利**：单检测/单 case 成本（已可信）→ 按客户分组上卷成客户级 P&L；CM I=收入−直接成本，CM II=−客户专属成本，CM III=−分摊共享间接。**单检测成本随量变 → 量价阶梯重要**。
5. **落点**：医院＝case 的属性（哪家医院送检，LIS 导入）→ 挂 `lis_cases.partner_id`，流到 `outbound_abc_details.partner_id`；**per-hospital 定价＝独立配置层 `partner_fee_schedules`**（覆盖默认 `fee_standards`）；盈利分析加 `dimension=partner`。

### 1.2 ❌ 已证伪
- "ABC 是医疗界**公认**(accepted)成本追溯法"——措辞过宽被否（1-2）。**用 ABC 作引擎，但客户级分配规则须用自有数据验证**，别当成铁律。

### 1.3 ⚠️ 关键 caveat（写进决策）
- **所有 fee schedule 来源都是"付款方/政府参考价"，没有一条是"独立实验室→医院 B2B 合同价"的直接文献** → **per-client 合同定价是"有力且被佐证的行业推断"，非单一引用规范**。落地须结合你们真实合同规则。
- **客户级上卷部分属推断**（ABC 客户盈利理论 Kaplan&Cooper 佐证，非逐字）→ 客户级分配规则用自有数据验证。
- **57/23/14 成本结构是临床化学专属**；IHC/特染试剂占比更高，**勿写死比例**；唯一稳的是"间接/共享成本永远是必须分摊的大头"。
- 中国 '11'/'12' 区划码＝省级**临时**项目（非泛指"地方变体"）。

### 1.4 研究未答（= 你的业务规则待定，见 §5）
- 独立实验室 per-client 合同价**具体结构**（生效日版本化价目表、量价阶梯阈值、打包/按例）——无直接文献。
- per-client 定价/收费代码**维护责任与 SoD、复核周期**——**无任何直接证据**（须你定）。
- 中国**加收码 × per-client 定价**如何叠加。
- 客户级 P&L **报表版式 + 共享成本"客户专属 vs 公司级"拆分约定**。

## 4. 提议数据模型（设计提案）

> 原则：**身份层不动**（`projects`/`fee_standards` 留作国家标准目录/默认价）；**新增"客户"维度 + "per-client 定价"叠加层**；**LIS 给事实、本系统配价**。

### 4.1 新增表
1. **`partners`（合作医院/客户主数据）**：`id, code(外部/LIS 编码,唯一), name, short_name, contact, contract_no, status, created/updated`。LIS 导入按 `code` 做外部 ID 映射（upsert）。
2. **`partner_fee_schedules`（per-client 价目表＝核心）**：`id, partner_id, project_id(或 fee_standard_id/test_code), charge_code(per-client 收费代码), price/fee_per_slide, tier_rules(JSON,量价阶梯), cap_amount, effective_from, effective_to, version, status, created/updated`。**唯一键** (partner_id, project_id, effective_from)。**前端可配置**（财务/商务编辑）。解析时：(partner×test) 命中→用之；未命中→回退 `bom_fee_mappings`/`fee_standards` 默认价。
   - （可选拆分）`charge_codes` 国家标准码目录表 + 上表只存 per-client 覆盖——MVP 可先把 charge_code 内联进价目表。

### 4.2 既有表加列（仅 ADD + ensureColumn，幂等）
3. `lis_cases.partner_id`（哪家医院送检；LIS 导入带入）。
4. `outbound_abc_details.partner_id`（从 case 冗余，供按客户上卷）。
5. （可选）`outbound_records.partner_id`（出库即捕获，历史可追溯）。

### 4.3 收入计算改动（cost-calculator.ts）
- `getFeeMappings`/费用解析改 **partner-aware**：先查 `partner_fee_schedules`(partner×test×生效日) → 命中用 per-client 价(+阶梯/加收)；未命中回退现有 BOM→fee_standards。`profit=feeAmount−totalCost` 不变。
- 写 `outbound_abc_details` 时落 `partner_id` + 用的 `partner_fee_schedule` 版本快照（审计/可复现，复用 source_snapshot 模式）。

### 4.4 按客户盈利（abc-v1.1.ts）
- `/abc/profitability` 加 `dimension=partner` + `partnerId` 过滤 → `GROUP BY partner_id`：收入/成本/利润/毛利率 per 医院。
- `/abc/dashboard` 加"按医院 top N 盈利/亏损"板块。
- **多级 CM 分期**：MVP＝CM I（收入−直接 ABC 成本＝客户毛利）；后续＝CM II（−客户专属固定）/CM III（−分摊共享间接，按活动动因分摊）。

### 4.5 RBAC（数据驱动，加种子即可）
- 新模块 `partners`(客户管理) + `partner_pricing`(定价/收费代码配置)。种子：finance/lab_director/admin = W（定价归财务/商务）；其它读或无。**SoD：定价维护与成本核算分离**（定价 finance、成本 technical）。因 RBAC 已数据驱动，加模块到矩阵 + 角色权限页可调即可。

## 5. 关键设计决策（待用户拍板——正好是研究的"开放问题"=你的业务规则）

1. **定价粒度**：per-hospital 价 key 在「医院×检测项目」逐项？还是「医院×统一折扣率」（在国家/默认价上打折）？还是两者都要（逐项覆盖 + 兜底折扣）？
2. **打包/分成**：有没有打包价(bundled，多检测一口价)/按例(per-case)/分成(与医院分润)？还是纯逐项？
3. **量价阶梯**：要不要量价阶梯（月量越大单价越低）？阈值怎么定？
4. **收费代码**：per-hospital 要"不同收费代码"还是"同代码不同价"？是否需要国家标准码目录（中国 15 位/CPT）作为身份锚？
5. **价目版本/生效日**：合同价是否生效日版本化（改价留历史、按 case 日期取当时价）？（建议是，复用 BOM 版本化模式）
6. **LIS 字段**：LIS 能否提供"送检医院"字段？字段名/编码？（决定 `lis_cases.partner_id` 怎么填）
7. **盈利口径**：客户级先到 CM I（收入−直接成本＝毛利）即可，还是一步到位含共享间接分摊（CM III）？
8. **治理/SoD**：per-client 定价谁维护（财务/商务）、多久复核、是否要审批卡口（像对账核准链那样）？

## 5.1 ✅ 用户锁定决策（2026-06-27）

1. **定价粒度＝逐项价目表**（每医院 × 每检测项目定价）。
2. **计价模式＝逐项固定价 + 与医院分成(分润)**（无打包/按例）。→ `partner_fee_schedules` 须支持两种 `pricing_mode`：
   - `fixed`：实验室每检测收固定价 `price/fee_per_slide`（可带 `tier_rules` 量价阶梯）。
   - `revenue_share`：实验室收 = `share_ratio` × 基准价(`base_ref`：国家/默认 `fee_standards` 价 或 约定基准)。即 `feeAmount = share_ratio × base_price × qty`。
   - 同一医院可逐项目选模式（某些项目固定价、某些项目分成）。
3. **盈利口径＝先 CM I 客户毛利**（收入 − 直接 ABC 成本）。共享间接分摊（CM III）留后续。
4. **定价治理＝财务/商务直接改 + 审计留痕**（who/when/old/new），**无前置审批卡口**（非对账那种核准链）。

> 据此细化 §4.1：`partner_fee_schedules` 增 `pricing_mode('fixed'|'revenue_share')`、`share_ratio`、`base_ref`；收入解析 §4.3 按 mode 分支算 feeAmount。审计：定价改动写 `operation_logs`（已有），不建审批表。

## 5.2 ⭐ 重大转向（2026-06-27，基于用户实际情况）—— 收入从"规则计算"改为"实收导入 + 拆分"

> 用户披露：① 公立目录**新旧并存**(per-hospital 用不同版) + 同检测**自费/医保双码双价**(per-case)，自动映射解决不了；② **财务有每 case 实收数据，可提供匹配**；③ 老收费规则把**实验室(技术)+诊断合并计费**，需拆分；④ 中心承担范围**看医院而定**(有的仅技术、有的含诊断)。

**新模型主线（收入侧）**：
- **收入＝导入"财务每 case 实收"为地面真相**（自费/医保、目录版本已隐含在实收里），按 `case_no` 匹配 → 不再靠我复现全套定价规则。**定价规则(`partner_fee_schedules`)降为兜底**（无实收数据的 case 才用规则估）。
- **收入拆分(decomposition)＝一等公民**：把（可能合并的）实收金额拆成 **技术(实验室) vs 诊断** 组分；**拆分规则 per-hospital**（因承担范围看医院而定）；中心收入＝它对该医院实际承担的组分之和。拆分口径**与 ABC 成本的作业中心/项目对齐** → 可算每组分利润。可对应中国码「项目分解编码 + 价格构成」(C4)。
- **利润 per case ＝ 中心承担组分收入 − ABC 成本(已可信)** → 按 **医院 × 自费/医保 × 期间** 上卷（CM I 客户毛利）。

**新增/调整数据模型**：
- `case_actual_charges`（导入的每 case 实收，新表）：`case_no, partner_id, charge_code, payment_type(自费/医保), catalog_version, item_name, amount, charge_date, import_batch, raw_snapshot`。一个 case 可多行（多收费项）。
- `charge_codes`（收费代码目录，人工维护多维）：见 §1 C5；主要用于**拆分映射**与兜底定价，不做自动派生。
- `charge_decomposition_rules`（拆分规则，per-hospital）：把 charge_code/合并项 → {技术占比/诊断占比 或 分项单价 或 关联作业中心}。
- `partners.service_scope`（该医院承担范围：tech_only / tech_plus_diagnosis）驱动"取哪些组分收入"。
- `outbound_abc_details` 加 `partner_id` + `actual_revenue`(拆分后归该中心的实收) + `payment_type`；profit = actual_revenue − total_cost。

**待用户提供（决定字段与拆分逻辑，建模前必需）**：
- **(A) 每 case 实收样例/字段**（case号/医院/收费码/自费医保/项目名/金额/日期/新旧版本）。
- **(B) 拆分规则**（合并收费→技术 vs 诊断：固定比例 / 分项单价 / 按作业中心对应；老新是否不同；per-hospital 差异）。

**不依赖 A/B、可先开工**：`partners` 主数据 + RBAC 模块 + `lis_cases.partner_id`/`outbound_abc_details.partner_id` + 按医院成本上卷脚手架（成本侧）。收入导入+拆分待 A/B 到位再做。

## 6. 分期建议

- **MVP（能回答"每个医院成本/收入/利润"）**：`partners` 主数据 + `partner_fee_schedules`(逐项价，复用 tier_rules) + `lis_cases.partner_id`/`outbound_abc_details.partner_id` + 收入解析 partner-aware + `/abc/profitability?dimension=partner`（CM I 客户毛利）+ RBAC `partners`/定价模块。前端：客户管理页 + per-hospital 价目配置页 + 盈利按医院筛选/分组。
- **后续**：多级 CM（客户专属/共享间接分摊）+ 价目版本化/生效日 + 打包/分成 + 收费代码国家标准目录 + 定价审批卡口（SoD）。

## 7. 与既有工作的衔接

- **正好接上「按医院之前先定模型」的顺序决策**：本模型定了成本/盈利/收费/看板页的信息架构 → 这些页的视觉重设计应在此之后做（一次成型）。纯 PSI 页重设计可并行。
- **复用已建基础设施**：BOM 版本化（→价目版本化）、数据驱动 RBAC（→加 partners/定价模块）、对账核准链模式（→定价审批 SoD，若需要）、`fee_standards.tier_rules`（→per-client 阶梯）、`lis_cases`+导入端点（→加 partner）。
- **不破坏 ABC 可信成果**：成本侧不动（单片成本引擎已矫正可信）；本特性只加"客户维度 + 收入侧 per-client 定价 + 按客户上卷"。
