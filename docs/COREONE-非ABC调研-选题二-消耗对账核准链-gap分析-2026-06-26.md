## 0. TL;DR（给 PM 的三句话）

1. **最重要的发现（且与代码实测吻合）**：当前对账的 BOM 回写是**即时、无审批卡口、无版本快照的"直接覆盖"**——任何有权角色在对账页点一下就**直接改写线上 `bom_items.usage_per_sample`**(`reconciliation-v1.1.ts:429-435`,前端 `useReconciliationPage.ts:253` 已接线)。这恰好是内控领域**强制要求"二级授权"的库存调整动作**,系统当前**完全缺这个卡口**。
2. **第二严重**：对账写权限开放给 `admin + pathologist + finance + technician` 四角色——**录入消耗的人 = 审批回写的人**(同一个 POST 既改 BOM 又记日志),**违反职责分离(SoD)四功能框架**的核心("永不审批/审计自己");且**病理医师(诊断线)本不该有成本/BOM 权限**(与 RBAC 调研结论冲突)。
3. **一个被自身架构打脸的内部矛盾**：M2 已建好 `bom_versions` 版本表 + `outbound_abc_details.bom_version_id`(为成本可复现),但**对账回写绕过了整套版本基础设施**直接覆盖 `bom_items`——破坏了引擎本要保证的"历史成本可复现"。**这是已有基础设施未被复用的明确 gap,修复成本相对低。**

> ⚠️ 重要纠偏:上一会话的 Explore 概览曾记"BOM 不可写回(immutable)"——**经代码实测,事实相反:BOM 可被对账即时覆盖回写**。本文档以代码实测为准。

### 1.3 ⚠️ 证据空白（无直接来源,仅内控/成本会计原则推断）

- **核准链归属(谁审批)主体上无实验室专属法规强制**——源自内控(WA State Auditor)+成本会计(管理会计教材)成熟原则,属"基于内控原则的合理推断"。
- **特殊染色(special stains)废片率基准无任何来源**;IHC 数据外推到特染需谨慎。
- **试剂层面(非切片层面)正常浪费率基准无来源**(抗体稀释残留/开封报废/孔板边角损耗)——用于 BOM 正常损耗津贴量化校准的数据缺。
- **回写治理的具体触发(多久复核一次 BOM、需谁签核、连续 N 周期超阈才回写)无权威来源**。

### 1.4 关键 caveat（写进任何决策）

- 核准链/SoD/二级授权阈值/回写卡口**主体是内控与成本会计原则,非实验室专属法规强制**;**仅 ISO 15189 的 lot/接收/有效期/验收 + 同时记录 + 修订审计留痕是直接标准约束**,且 ISO"同时记录"范围是"影响检验质量的活动",库存/消耗记录是否在内属推断。
- 10% 阈值为**示例性,非强制**;具体阈值须 per-lab/per-test 设定。
- IHC 2% / 0.8%/9% 为**单中心单窗口量级示例,非普适常量**。

## 2. 当前 COREONE 对账设计（代码实测,gap 对照基线）

> 源:`后端代码/server/src/routes/reconciliation-v1.1.ts` + `前端代码/src/pages/reconciliation/*`（master 线 / audit-p0 worktree 实读）。

- **差异口径**:`diff = actualQty − theoryQty`,`theoryQty = caseCount × usage_per_sample`(LIS 病例数 × BOM 标准用量);仅计入挂项目的出库(口径与理论一致)。
- **阈值(硬编码,不可配)**:`warn` 当 `diff > theory×0.2`(+20%)或 `diff < −theory×0.2`(−20%);`danger` 当 `diff > theory×0.5`(+50%)。**超耗有 danger、省耗无 danger 下限**(line 174-177 / 257-259)。
- **核准链:无**。`POST /logs`(line 423-447)单次请求同时完成 ①(可选)**直接覆盖** `bom_items.usage_per_sample`(line 429-435)+ ② 插入 `reconciliation_logs`。**前端已接线**(`useReconciliationPage.ts:253` 发 `newUsage`)。grep 全模块**零** approve/核准/审批/sign-off。
- **写权限角色**:`app.ts:92` 注册 `requireRole('admin','pathologist','finance','technician')` → **四角色均可改写 BOM 标准用量**。
- **回写副作用缺失**:覆盖 `bom_items` 时**不创建 `bom_versions` 快照、不设 effective_scope(future/retroactive)、不重算已计本的 `outbound_abc_details`**——而 M2 已为成本可复现专门建了 `bom_versions` 表 + `outbound_abc_details.bom_version_id`。
- **审计留痕:有**。`reconciliation_logs` 记 who(`req.user.username`)/when/field/old/new/reason——**事后留痕尚可,但非审批前置卡口**。

## 4. 改进建议（**设计提案,非实施指令**;按"采纳优先 + 风险/收益"排序）

> 遵循"采纳优先于技术正确":先以**低摩擦**补上最危险的卡口,避免一上来就重流程吓退用户。

### 4.1 🔴 高优先（堵住无防护的 BOM 覆盖——最危险且证据最强）
- **H1+H2-a 解耦"记录差异"与"回写 BOM"**:把 `POST /logs` 拆成两步——**①提议(record/propose)**:任何技术线角色可记录"实测差异 + 建议新用量 + 原因";**②审批回写(approve & apply)**:由**独立角色**(非提议人)审批后才真正写 `bom_items`。即落地 **H-R1"录入≠审批" + H-R3"二级授权"**。最小可行版:**至少禁止"同一人一键改 BOM"**,改成"提议→他人确认"。
- **H2-b 回写复用 `bom_versions`(已有基础设施!)**:回写**不再直接覆盖** `bom_items`,而是**创建新 `bom_version` 快照 + 设 effective_scope(默认 future_only)**,线上标准切到新版本。这样**不破坏 M2 的成本可复现**(已计本 `outbound_abc_details.bom_version_id` 仍指向旧版),且天然满足 ISO 8.4.2"修订可追溯到原始版本"。**收益高、复用已有表、风险可控。**
- **H5-a 对账写权限对齐新 RBAC**:**移除 pathologist 的对账写权限**(诊断线无成本/BOM 权);用量差异提议归**技术线**,BOM 标准回写审批归**财务/lab_director**(成本 owner)。**与 RBAC 矩阵落地一并做**(见 RBAC 文档),消除越权 + 落地 SoD。

### 4.2 🟡 中优先（差异口径精细化）
- **H3-a 阈值可配 + 联动审批**:把硬编码 ±20%/+50% 改为**可配置**(全局或 per-test),并**联动**:超容差的差异**才需要审批回写**(例外管理),容差内的视为正常损耗免审。
- **H4-a 标准内置正常损耗津贴**:BOM 标准用量增"正常损耗率"概念(可达成标准),使**只有异常超耗才触发对账核准**;损耗率**按检测类型分层**(IHC 诊断类 vs 预测类差异大,见 H-R7)。可与选题一 G3/G4(用量结构化)合并设计。
- **H6-a BOM 回写版本可追溯**:即 4.1 的 H2-b 落地后自然满足;补充确保对账与试剂 **lot 关联**可追(ISO 6.6.7)——核实当前入库批次/lot 是否贯通到消耗端。

### 4.3 🔵 低优先（周期与流程）
- **H7-a 定义对账期结节奏**:引入"月度对账期结"概念,与盘点模块(P1-04)/周期盘点联动;期结后差异冻结、回写按期生效。非紧急,按运营需要再做。

## 5. 与「单片成本可信」及 RBAC 的交叉影响

- **H2(回写绕过 bom_versions)直接威胁 ABC 可信度**:M3/M4 花大力气做的"黄金用例 ¥120 + 成本可复现",会被对账页一次无版本覆盖**悄悄改掉历史成本基线**。**这是 ABC 矫正成果的一个隐藏回归口**,应优先堵(4.1 H2-b)。
- **H1/H5 与 RBAC 强耦合**:对账写权限(含病理越权)本就是 RBAC 待办 #2 的一部分。**建议把"对账核准链 + 写权限"并入 RBAC 矩阵落地**,一次对齐 nav/requireRole/ROLE_PERMISSIONS/对账卡口,避免分两次改。
- **H3/H4 与选题一交叉**:正常损耗津贴(H4)+ 用量结构化(选题一 G3/G4)是同一件事的两面——**"标准用量该多精确、差异该多宽容"**应合并设计。

## 6. 开放问题（留待进一步调研/与用户确认）

3. CLSI / NordiQC 等**实验室专属指南**是否对对账周期/盘点频率有特定建议(本轮未覆盖)。

5. 当前**入库批次/lot 是否贯通到消耗/对账端**(ISO 6.6.7 lot 追溯)——需代码核实。
