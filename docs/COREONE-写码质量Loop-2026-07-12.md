# COREONE 写码质量 Loop（实现阶段）

> **状态**：v1.0 草稿（2026-07-12）· 按「产出必过质疑关」规矩，PM 质疑/拍板后生效
> **从属关系**：本文从属于共用契约（agent-operating-contract）、CLAUDE.md 安全与编码规范、pr-governance。**那些文件里已有的规则本文一律不复制**（防两处文本各改各的、镜像不同步），本文只定一件事：实现阶段**一轮怎么转、按什么顺序过哪些关**。
> **输入**：功能实现 = 定稿 PRD + 定稿 mockup（缺一样就回上游，不进本 loop）；bug 修复 = 可复现路径 + 期望行为。
> **适用**：功能实现、bug 修复。纯文档改动不走本 loop。

## 0. 在工作流里的位置

```
mockup 定稿 → 【写码质量 Loop（本文）】→ PR + 独立复核 → PM 批准合并 → 真跑验收 Loop
```

使命：让「测试绿、复核过、和定稿一致」在**每一轮**都成立，而不是最后一天补。

## 0.5 先分档（决定走哪条道）

| 档 | 判定 | 走法 |
|---|---|---|
| 微改 | 纯文案/纯样式，不碰逻辑、不碰钱、不碰权限 | 轻量道：只走 ③机器关 + diff 自查 |
| 常规 | 一般功能实现 / bug 修复 | 全 loop ①–⑥ |
| 碰钱/口径/权限 | 对账、成本、毛利、预算、RBAC | 全 loop + 第 5 节加固 |

拿不准算哪档 → 按高的算。

## 1. Loop 全景

```
┌─> ① 开工核实（preflight + 现状核实，别按记忆/清单开工）
│   ② 写码（一轮一个可运行、可验证的小目标）
│   ③ 机器关（相关测试/黄金锚、构建纪律闸、diff 自查）
│   ④ 变异证有牙（新测试先证明抓得住坏实现）
│   ⑤ 摊偏离 + 自我质疑（与定稿的所有偏离，标待拍板）
│   ⑥ 异构复核（对抗面板 / codex）→ 意见归类：修 / 讨论项 / 误报（要证据）
└── 有要修的 → 回 ②
    全绿 + 偏离零未拍 + 复核意见清零 → 开 PR 交 PM（合并授权在 PM）
```

## 2. 每步做什么

### ① 开工核实
- 跑 preflight（develop 模式）、fetch 现场 master；分支/PR 状态现查 `gh`，不信记忆和旧文档。
- **现状核实**：要修的 bug 现在还在吗？要建的东西是不是已有现成的？待办清单常失真，别照单开工。
- bug 修复加一条铁律：**先复现再动手**——在提交态数据上复现（用仓库里的 dev 库，别先 seed 把现场洗掉）；复现不了就把「复现不了」本身作为发现交出去，不硬修。

### ② 写码
- 一轮一个小目标：每轮结束必须有可运行、可验证的增量；禁「一把梭到最后才第一次跑」。
- 写的过程中发现定稿 PRD / mockup 不合理 → **停，记进偏离清单**，不许码上自由发挥（上游两个 loop 的出口规矩在这里兑现）。

### ③ 机器关
- 相关测试 + 黄金锚必须绿；红了如实说红，不许「应该没影响」。
- 构建纪律闸 / lint 过掉。
- `git diff` 自查：无越界文件、没碰别人的 worktree、dev 库已复原、没混进临时日志或调试代码。

### ④ 变异证有牙
- 新写的每个关键测试，先人为改坏实现 → 测试必须变红 → 改回 → 绿。**抓不住坏实现的测试等于没写。**
- bug 修复：修完用**同一条复现路径**证明不再复现。

### ⑤ 摊偏离 + 自我质疑
- 偏离清单：实现与定稿 PRD/mockup 不一致的每一处（有意的和被迫的都算），标「待拍板」。
- 自我质疑：边界（空 / 超大 / 并发 / 权限外角色）、最坏误操作后果、对现有口径的波及面。

### ⑥ 异构复核
- 常规档至少一道：对抗面板或 codex review。
- 复核意见三分类：**修**（回 ②）/ **讨论项**（交 PM）/ **误报**（必须给出反证证据，不许一句「不影响」带过）。

### 交付（PR 五件套）
1. PR（body 按 handoff 模板：owned files、依赖、验证证据）
2. 改动清单（这个 PR 到底动了什么）
3. 偏离清单（全部已拍板，或明确列为待拍板）
4. 测试与复核证据（跑了什么、结果如何，红的如实写）
5. 结尾大白话

## 3. 退出条件（全部满足才算「实现完成」）

1. 机器关全绿（相关测试、黄金锚、纪律闸）
2. 关键测试做过变异验证；bug 修复有「同路径不再复现」证据
3. 偏离清单零未拍残留
4. 异构复核意见全部三分类处理完
5. PR 已开、五件套齐
6. **合并 = PM 批准后执行**，本 loop 不含自动合并

## 4. Loop 护栏

| 护栏 | 规则 |
|---|---|
| 提交纪律 | 只显式 add 自己改的源码；禁 `git add -A`；dev 库提交前复原（细则以 CLAUDE.md 为准，此处不复制） |
| 所有权 | 一个文件一个实现 owner；不碰别人的 worktree（细则以 pr-governance 为准） |
| 防一把梭 | 连续多轮没有可验证增量 = 任务拆得不对，停下重拆 |
| 防打转 | 同一个红修 3 轮没过 → 停止再猜，把根因分析 + 候选修法并排摆给 PM / 复核方 |
| 诚实交付 | 没跑的说没跑、红的说红；没拿到结论前不许写「已完成 / 已兜底」 |
| e2e 幻觉 | 别把 CI e2e 当回归网（PR 门只跑极小子集）；改关键流程本地真跑相关 spec |

## 5. 碰钱 / 口径 / 权限档的加固

1. **黄金锚零漂移是硬门**：相关黄金数字必须逐位一致；漂了先解释清楚再谈合并。
2. **异构复核升格**：多镜头对抗面板（不同视角各攻一面），不止一次普通 review。
3. **数值边界专项**：写库前的 Infinity / 超界 / 空串等边界单独过一遍——这类坑单测常漏，必须列专项自查。
4. **权限对照运行库**：权限改动以运行库真实权限验证，不按源码种子矩阵想当然。

```
4. bug 修复先复现再动手，修完用同一路径证明不再复现。
5. 交付 PR 五件套；合并等 PM 批准，不自动合并。
```

## 8. 给 PM 的大白话

这份文档管「真正写代码」这一段：AI 每写一小块就要自己测一轮，还要故意把代码改坏一次、证明测试真的抓得住错，抓不住的测试等于白写。写的过程中如果发现你拍过板的需求或草稿图有问题，它不许自作主张改，必须记下来回头找你重拍。最后要经过第二个 AI 独立挑刺、所有检查真绿，才能提交给你批准合并——它自己不许合。改个错别字这类小事走简化流程，不会事事兴师动众；碰钱碰权限的改动则层层加码。
