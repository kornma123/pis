> **状态**：v1.0（2026-07-12）· 待 PM 定稿拍板
> **从属**：本页只是**路由**——从属工作模型（方法论主线）与共用契约；规则的唯一源是《质量 Loop 契约》，**本页不含任何规则、也不下结论**。「完成」仍受工作模型状态台账约束：**PM 验收之外的发布/上线是独立状态，不由本家族收口**（工作模型「PR 合并 ≠ 发布」高于本家族）。

## 主链

想法/痛点 → PRD → Mockup → 写码 → 真跑验收 → PM 验收拍板（= 消费者被服务；但发布/上线是独立状态）。
横切：报告/结论（任何调研/审计/评估/复盘，随时插入）。

## 路由表

| 你要做的事 | 走哪个入口 |
|---|---|
| 有个想法/痛点，要变成需求 | PRD |
| 需求定了，要画界面 | Mockup |
| 界面定了，要写代码（含 bug 修复） | 写码 |
| 代码合了，要确认真的能用 | 真跑验收 |
| 要一份调研/审计/评估/复盘结论 | 报告/结论 |
| 纯执行小事（改错别字、调颜色） | 契约的微改档 / 各入口轻量道 |

## 有意不建的 Loop（摊出来，防止被当成遗漏）

| 候选 | 不建理由 |
|---|---|
| 讨论循环 | 已内嵌每个 loop 的「新讨论 → 继续循环」 |
| **真数据校准 loop** | 不建独立 loop：校准 = 报告 loop 定靶（「口径校准任务」靶型）+ 写码 loop 落值 的组合，建第 7 个 loop 必然重抄报告 loop 的分级机器（Q2 刚杀的镜像病）；且 loop 能强制「现形」不能强制「数据存在」，建 loop = 功能先于消费者被建；它是间歇的、数据驱动的**队列**工作，用 Issue 队列建模才对（见契约 §5「接受 grade C 自动入队」） |
| DB 迁移 / E2E 编写 | 已有 `/db-migration`、`/e2e-testing` 技能 |
| 事故 / 操作安全 | 属操作铁律层（契约 / guardrails），是红线不是循环形态 |

## 给 PM 的大白话

这一页是全家的地图：从一个想法到功能真能用，中间过四道循环（写需求 → 画草稿 → 写代码 → 真跑验收），外加一道管所有「给结论的报告」。所有规矩只写在一份《质量 Loop 契约》里，这五个入口和这页都只指过去、不重抄（免得各改各的对不上）。哪些环节故意没建循环、为什么，也列在上面了。这页只管指路。
