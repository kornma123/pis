# COREONE 按医院(客户)成本与盈利 —— 实施计划（2026-06-27）

> 配套设计/调研：[COREONE-按医院客户成本盈利-调研+数据模型设计-2026-06-27.md](COREONE-按医院客户成本盈利-调研+数据模型设计-2026-06-27.md)。
>
> **⭐ 模型锁定（2026-06-27，用户最终确认，覆盖下方旧 §1/§5/P2）**：
> - **收入 = 财务实收 × 引擎技术占比**（引擎按真实收费目录规则算每 case 的诊断/技术/取材占比；实收为金额真值）。利润 = 实验室收入 − ABC 成本；按医院×期间上卷（CM I）。
> - **v1 砍掉定价配置（fixed/revenue_share `partner_fee_schedules`）**——收入全用财务实收，无 per-hospital 配价。（→ v2 才考虑）
> - **收入独立计算、不经 `cost-calculator.ts`**（比原 P3 更安全）：成本引擎零改动，golden 天然全程绿。
> - 引擎"应收总额"产出留作 **收费稽核(应收vs实收)** 与无实收兜底 —— **稽核 v1 先不做**（用户：先不定）。
> - 适用旧码/私立：占比只需 LIS 数量，与医院用什么收费码无关。
> **红线：不动可信 ABC 成本引擎**；`abc-golden-accuracy.test.ts` 全程绿为合并闸。

## 1. 费用解析注入设计（partnerId 穿线不破黄金成本路径）
- 链：`calculateSlideCostWithFee(492)` → `calculateFeeBreakdown(270, 注入点 278)` → `calculateFeeAmountFromStandard(76)`；`profit=feeAmount−totalCost(555)`＝CM I 天然成立。
- `getFeeMappings` **保持原样**（既有单测直接断言）。新增 `resolveFeeMappings(db,bom,partnerId?)`：
  - `partnerId` 空 → 逐字回退 `getFeeMappings`（黄金/单测路径 0 变化）。
  - `partnerId` 有 → 查 `partner_fee_schedules`：`fixed` 模式构造同形 mapping(走 tier/cap)；`revenue_share` 模式 base_price=`share_ratio×基准价`；未命中 schedule → 回退 getFeeMappings。
- 成本不变：注入点在收入侧之前；黄金/单测不传 partnerId → 等价路径。**P3 断言「partner 与无 partner 的 totalCost 完全相等」**。
