## 1. 测试概述

### 1.1 测试目标

- 验证 v1.1 版本所有功能符合 PRD 需求定义
- 确保核心业务流（入库→出库→成本归集→报表）数据一致性
- 验证系统性能和稳定性满足非功能需求
- 发现潜在缺陷，降低上线后故障率

### 1.2 测试范围

**包含测试**:
- 功能测试：8个业务模块，20个页面，25个弹窗
- 集成测试：模块间数据流转和接口调用
- 性能测试：核心API响应时间和报表生成
- 安全测试：认证授权、输入验证、SQL注入
- 兼容性测试：主流浏览器适配
- 数据一致性测试：库存变动、成本归集、批次管理

**不包含测试** (v1.2+):
- 移动端APP测试
- 第三方系统集成测试
- 压力测试（超过50并发）
- 灾难恢复测试

### 1.3 测试环境

| 环境 | 配置 | 用途 |
|------|------|------|
| 本地开发环境 | Docker Compose | 开发自测 |
| 测试环境 | 独立服务器 4核8G | 功能测试、集成测试 |
| 预发布环境 | 独立服务器 8核16G | UAT验收、性能测试 |

## 4. 集成测试

### 4.1 核心业务流程测试

#### 流程1: 完整入库→出库→成本核算流程

```
步骤1: 创建物料 → 验证物料创建成功
步骤2: 入库登记（10盒，单价100元） → 验证库存增加10
步骤3: 再次入库（20盒，单价110元） → 验证库存增加20，总库存30
步骤4: 出库领用（15盒，关联项目A） → 验证库存减少15，总库存15
步骤5: 验证出库成本 = 10×100 + 5×110 = 1550元（FIFO）
步骤6: 查看项目A成本 → 验证成本=1550元
步骤7: 查看物料成本报表 → 验证消耗15盒，成本1550元
步骤8: 查看库存 → 验证当前库存15盒
```

#### 流程2: BOM配置→一键出库→成本分析流程

```
步骤1: 配置检测项目BOM（物料A:0.5ml, 物料B:0.3ml）
步骤2: 绑定BOM到检测项目
步骤3: 选择BOM一键出库，样本数10
步骤4: 验证出库数量 = 物料A:5ml, 物料B:3ml
步骤5: 验证成本按实际批次单价计算
步骤6: 查看项目成本包含BOM物料成本
```

#### 流程3: 库存盘点→差异调整→库存校正流程

```
步骤1: 创建盘点任务
步骤2: 系统显示账面库存
步骤3: 录入实盘数量（与账面不一致）
步骤4: 生成差异报告
步骤5: 审批并调整库存
步骤6: 验证库存更新为实盘数量
步骤7: 验证库存流水记录调整记录
```

## 5. 数据一致性测试

### 5.1 核心数据公式验证

| 公式 | 验证场景 | 测试方法 |
|------|---------|----------|
| 当前库存 = ∑入库 - ∑出库 - ∑报废 | 任意时间点查询 | 随机抽查+边界值 |
| 项目成本 = ∑(该项目出库数量 × 对应批次入库单价) | 月度成本报表 | 与手工计算对比 |
| 物料成本 = ∑(该物料出库数量 × 对应批次入库单价) | 物料成本分析 | 与手工计算对比 |
| 批次剩余 = 入库数量 - 该批次的出库数量 | 批次查询 | 逐批次核对 |

### 5.2 验收测试数据

使用专用验收测试数据验证数据一致性：

```
验收测试试剂盒 (MAT-TEST-001)
├── 供应商: 云深生物 (SUP-TEST-001)
├── 入库1: 2026-04-22, VAL-20260422, 10盒, 单价500元
├── 入库2: 2026-04-23, VAL-20260423, 20盒, 单价520元
├── 出库: 2026-04-23, EGFR基因突变检测, 15盒
│   └── 出库成本 = 10×500 + 5×520 = 7600元 (FIFO)
├── 库存: 30 - 15 = 15盒
└── 成本报表: 消耗15盒, 总成本7600元
```

**验证清单**:
- [ ] 入库记录显示2条入库，共30盒
- [ ] 出库记录显示1条出库，15盒
- [ ] 库存列表显示当前库存15盒
- [ ] 物料成本报表显示消耗15盒，成本7600元
- [ ] 项目成本报表显示EGFR项目包含7600元成本

## 9. 缺陷管理

### 9.1 缺陷等级定义

| 等级 | 定义 | 响应时间 | 修复时限 | 是否阻塞上线 |
|------|------|----------|----------|-------------|
| P0 - 致命 | 系统崩溃、数据丢失、安全漏洞 | 立即 | 4小时 | 是 |
| P1 - 严重 | 核心功能不可用、主要流程阻塞 | 2小时 | 24小时 | 是 |
| P2 - 一般 | 非核心功能异常、有 workaround | 1天 | 3天 | 否 |
| P3 - 轻微 | UI问题、文案错误 | 3天 | Sprint内 | 否 |

### 9.2 缺陷跟踪流程

```
发现缺陷 → 提交Bug → 开发确认 → 修复 → 验证 → 关闭
              ↓           ↓         ↓        ↓

```
