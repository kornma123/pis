# COREONE v1.0 功能补全清单

> **生成日期**: 2026-05-22  
> **基于**: 实际代码扫描 + [交互规范](interaction-specs/inbound.md) + [PRODUCT-REQUIREMENT-MAPPING](PRODUCT-REQUIREMENT-MAPPING.md)  
> **目标**: 将已实现但含演示数据/空壳逻辑的功能补全为可用状态，同时补齐前端缺失页面  
> **原则**: 按优先级分批推进，每批可独立验证

---

## 补全策略

```
Phase 1: 修复现有页面的"假功能"（P0）
    └─ 让已有页面真正可用，消除演示数据和空壳逻辑

Phase 2: 补齐后端有API但前端无页面的功能（P0）
    └─ 退库、报废、调拨管理页面

Phase 3: 体验优化（P1）
    └─ 分页、校验、提示等细节优化

Phase 4: 采购订单独立管理（P2）
    └─ 采购订单视角的入库管理
```

---

## Phase 1：修复现有页面的"假功能"

### 1.1 入库页面（Inbound）

| ID | 问题 | 影响 | 优先级 | 前后端 | 工作量 | 修复方案 |

| IN-P0-02 | 选择采购订单后，数量输入无 `remainingQty` 上限校验 | 用户可输入超量，后端会返回 400，但前端无即时反馈 | P0 | 前端 | 2h | 选择采购订单后，数量框加 `max={remainingQty}`，超量时红色提示 |
| IN-P0-03 | 确认入库弹窗（confirm modal）未绑定 state，`handleConfirmInbound` 仅 toast 未调 API | "确认入库"按钮完全不可用 | P1 | 前端 | 4h | 绑定 quantity state → 调用 inboundApi.create + purchaseOrderApi.receive → 成功后刷新列表 |
| IN-P0-04 | 恢复入库弹窗数据硬编码（`quantity + 400`） | 恢复后库存显示为假数据 | P1 | 前端 | 1h | 从后端获取当前库存，计算恢复后真实库存 |
| IN-P0-05 | 扫码入库为模拟（固定返回"DNA提取试剂盒"） | 无法真正扫码 | P1 | 前端 | 4h | 接入摄像头/扫码枪 API，或提供手动输入条码后查询耗材的 fallback |

| IN-P0-07 | 打印预览操作人固定为"张医生" | 打印单据信息不准确 | P2 | 前端 | 1h | 从 JWT token 或 localStorage 获取当前操作用户名 |
| IN-P0-08 | 查询按钮为空函数 | 功能冗余但无害 | P2 | 前端 | 0.5h | 移除查询按钮，筛选已实时生效 |
| IN-P0-09 | 表格 pending 状态判断为 `quantity > 1000` | 状态显示不正确 | P2 | 前后端 | 2h | 后端入库记录增加 `pending` 真实状态，或前端去掉演示逻辑 |

### 1.2 出库页面（Outbound）

> **已核实（2026-07-02）**：出库页整体为**真实功能**，无入库页那类"演示数据/空壳逻辑"。已扫描
> [Outbound.tsx](前端代码/src/pages/outbound/Outbound.tsx) 及其 `components/` 子组件、
> [outbound-v1.1.ts](后端代码/server/src/routes/outbound-v1.1.ts)，逐项确认如下。

**已确认为真的能力**（无需补全）：

| 能力 | 实现 |
|:---|:---|
| 列表 / 分页 | 后端真分页（`usePagination` → `GET /outbound`，非前端 `slice`）；筛选条件（关键词/物料/类型/状态/日期）随 URL 同步 |
| 统计卡片 | 从 `GET /outbound/stats` 取真实计数（总数/已完成/待出库/已取消/累计金额），无硬编码 fallback |
| 出库登记 / 编辑 / 删除 | 真实调 `POST/PUT/DELETE /outbound`；删除走软删 + 库存/批次回退 + `stock_logs` 留痕；带幂等键防双击/重发重复出库 |
| 批量导出 / 打印 | 真实 xlsx 导出与单据打印（选中项或全部） |
| **表头排序** | ✅ **本次补齐**（原 UX-P1-03）：后端白名单列 `sortField`(金额/数量/时间)+`sortOrder`，前端表头点击切换升/降序；跨全部分页的**后端真排序**（非仅当前页） |

**残留低优先项**（真实但非阻断，改动落其他线/后续）：

| ID | 问题 | 影响 | 优先级 | 前后端 | 修复方案 |
|:---|:---|:---|:---:|:---:|:---|
| OUT-P2-01 | 取消原因未落库：`OutboundCancelModal` 收集的取消原因/备注在 `handleCancel` 中被丢弃（实际调 `delete`，不传 reason） | 取消动作有效（库存回退），但取消原因不留痕 | P2 | 前后端 | 后端删除/取消接口接收 `reason` 并写 `stock_logs.remark`，前端 `handleCancel` 传入 |
| OUT-P2-02 | `pending`/`cancelled` 状态形同虚设：后端出库恒以 `status='completed'` 落库，无置为 `pending` 的流程；故"待出库"筛选、仅 `pending` 才显示的"取消出库"按钮实际永不触发 | 无害的冗余 UI；统计中 pending/cancelled 恒为 0 | P2 | 前后端 | 若确需审批式出库再引入 pending 状态机；否则移除相关 UI 分支 |

## Phase 3：体验优化

| ID | 问题 | 影响 | 优先级 | 前后端 | 工作量 |
|:---|:---|:---|:---:|:---:|:---|
| UX-P1-01 | 分页为前端分页（`filteredData.slice`） | 大数据量时性能差 | P1 | 前端 | 4h | 改为后端分页，筛选条件传给后端 |
| UX-P1-02 | 筛选条件变化时无 URL 同步 | 刷新页面后筛选丢失 | P1 | 前端 | 3h | 使用 URL query params 保存筛选状态 |
| UX-P1-03 | 表格列无排序 | 用户无法按金额/数量排序 | P1 | 前后端 | 4h | ✅ **出库页已落地（2026-07-02）**：后端白名单列 `sortField`(金额/数量/时间)+`sortOrder`（防注入）、前端表头点击排序（后端真排序，跨全部分页）。库存页现为当前页客户端排序，入库页待沿用同范式跟进 |
| UX-P2-01 | 入库记录删除无二次确认详情 | 用户不知道删除会有什么后果 | P2 | 前端 | 2h | 删除前调 `checkDeletable` API，展示具体原因 |
| UX-P2-02 | 统计卡片数字为硬编码（如 `|| 156`） | 无数据时显示假数字 | P2 | 前端 | 1h | 去掉硬编码 fallback，无数据时显示 0 或 `-` |
| UX-P2-03 | 日期范围筛选结束日期不含当天结束 | 可能漏掉当天数据 | P2 | 后端 | 1h | 后端已拼接 `T23:59:59`，确认前端行为正确 |

## 验收标准

每个 Phase 完成后应能验证：

**Phase 1 验收**:
- [ ] 采购订单可多次分批入库，每次入库后订单状态正确更新
- [ ] 数量超过剩余待入量时前端即时提示
- [ ] 确认入库按钮点击后真正创建入库记录
- [ ] 恢复入库显示真实库存变化
- [ ] 扫码/导入/打印为真实功能或可明确标注为"待实现"

**Phase 2 验收**:
- [ ] 退库页面可访问，能创建退库记录
- [ ] 报废页面可访问，能创建报废记录
- [ ] 调拨页面可访问，能创建调拨记录
- [ ] 采购订单页面可访问（或通过入库页面完整操作）
