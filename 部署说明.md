# COREONE 部署指南

> 当前仓库只提供本地商业发布候选合同。生产、R3、真实数据库、真实凭据、DNS、证书与流量切换都必须停在 operator 授权门；本文不表示已经部署或正式上线。

## 本地商业发布候选合同（R3 operator 授权门）

`scripts/release/**` 是唯一发布入口。候选验收把批准的 40 位 commit SHA、不可变镜像 ID、external SQLite volume、仓库外 secret 文件和单跳代理网段绑定在一起。任何工作树漂移、可变 tag、inline secret、缺失数据库或不受信代理都会 fail-closed。

### 固定源码、镜像与运行边界

- `COREONE_RELEASE_SHA` 必须是批准的 40 位小写 commit SHA；`build-local-images.mjs` 要求 `HEAD` 等于它且工作树干净。
- 后端基础镜像固定为带 digest 的 Node `22.23.1`。builder 安装开发依赖并编译业务与 provisioning；runtime 只执行 `npm ci --omit=dev`，不携带 `tsx`、TypeScript 或 esbuild。
- 前后端构建 context 都由固定 commit 的显式 allowlist 生成。后端 `.dockerignore` 再次限制 Docker 可见范围；新增文件默认不会进入镜像。
- `COREONE_BACKEND_IMAGE` 与 `COREONE_FRONTEND_IMAGE` 必须是 build receipt 中的 `sha256:` image ID。Compose 不含 `build:`，且 `pull_policy: never`。
- `COREONE_DATA_VOLUME_NAME` 必须指向 operator 已确认的 external volume。常驻 backend 固定 `COREONE_ALLOW_DATABASE_CREATE=0`，不会把缺失数据库静默变成空系统。
- `COREONE_JWT_SECRET_FILE` 只能指向仓库外受控只读文件。常驻 backend 不接收 inline JWT，也不挂载一次性 admin secret。
- backend 以 `1000:1000`、frontend 以 `101:101` 运行；根文件系统只读、capabilities 全部移除，只允许受控 volume 与 tmpfs 写入。
- 正式拓扑固定为“浏览器 → 单一 Nginx → backend”。Compose 固定 `TRUST_PROXY_HOPS=1`，并把 operator 的同一个 `COREONE_INTERNAL_SUBNET` 注入 `TRUST_PROXY_CIDRS` 和 internal network IPAM。不得改名、拆分填写或增加第二跳。
- 公网、生产、R3 和真实数据操作全部需要独立 operator 授权；本地合同通过不等于运行时或上线许可。

### 本地候选与构建计划

```bash
# 22 项或当前提交更新后的准确计数；不启动 Docker daemon 或服务。
node --experimental-sqlite scripts/release/release-contract.selftest.mjs

# backend 已完成 npm run build 后才运行；只用显式临时数据库和随机一次性输入。
node --experimental-sqlite scripts/release/release-contract.selftest.mjs --launcher-smoke

# 默认只验证 clean 固定 SHA 并输出计划，不创建镜像。
node scripts/release/build-local-images.mjs \
  --release '<approved-40-character-commit-sha>' --json
```

只有在 Docker Engine 可用、固定基础镜像已经预载、目标 SHA 已提交并获得相应授权时，才可增加 `--execute`：

```bash
node scripts/release/build-local-images.mjs \
  --release '<approved-40-character-commit-sha>' \
  --output '<absolute-outside-repository-build-receipt.json>' \
  --execute --json
```

build receipt 位于仓库外且不可覆盖。Docker daemon 不可用时，真实 image/container/network/volume 验收保持 NO_GO；静态合同、宿主编译或 Compose 解析不能替代这些证据。

### Compose 解析与启动前门禁

```bash
COREONE_RELEASE_SHA='<approved-40-character-commit-sha>' \
COREONE_BACKEND_IMAGE='sha256:<64-hex-from-build-receipt>' \
COREONE_FRONTEND_IMAGE='sha256:<64-hex-from-build-receipt>' \
COREONE_DATA_VOLUME_NAME='<operator-approved-external-volume>' \
COREONE_JWT_SECRET_FILE='<absolute-path-to-operator-provided-secret-file>' \
COREONE_INTERNAL_SUBNET='<operator-approved-rfc1918-ipv4-cidr>' \
node --experimental-sqlite scripts/release/release-contract.selftest.mjs --compose-config
```

`COREONE_INTERNAL_SUBNET` 必须是网络地址对齐、至少保留 6 个可用地址的 RFC1918 IPv4 CIDR；公网段、`0.0.0.0/0`、host address 和过小网段都会被拒绝。解析不会读取 secret 正文，也不会创建容器、网络或 volume。

## 首次安装与旧卷迁移

普通 `docker compose up` 不会创建数据库，也不会激活 operator profile。

全新 external volume 只允许一次性 `operator-r3-first-install`：

```bash
# R3 未授权时禁止执行；这里只固定入口，不表示已操作任何目标 volume。
docker compose --profile operator-r3-first-install run --rm --no-deps \
  database-init
```

运行前必须让 `COREONE_ADMIN_INITIAL_PASSWORD_FILE` 指向仓库外只读 secret。初始化器拒绝已有数据库及残留的 `-wal`、`-shm`、`-journal` sidecar。成功后立即删除源一次性 secret，再在数据库保持离线时按下节导入当前批准 manifest；bootstrap admin 不是长期账号清单。

旧卷权限迁移只允许在备份、校验和隔离恢复演练通过后执行 `operator-r3-volume-migration`。operator 必须提供同一次已验证备份绑定的确认、SHA-256、basename、备份与 manifest 的仓库外绝对路径；迁移时常驻 backend 停止、目标库离线。自由填写的确认文本或散列外观不是备份证据。

## 批准账号 provisioning

### 输入合同

批准 manifest 是非秘密 JSON，只允许：

- 根字段 `schemaVersion: 1`、非空 `approvalReference`、非空 `accounts`；
- 每个账号字段 `username`、`realName`、`roles`、`primaryRole`，以及可选 `department`；
- `primaryRole` 必须属于该账号的 `roles`，账号名大小写归一后不得重复，账号数与角色数受实现上限约束；
- 清单不得包含 password、credential、secret、token、hash 或任何未声明字段，也不得自行扩大批准账号或权限。

凭据 envelope 使用 `schemaVersion: 1` 和与 manifest 用户名精确对应的 `credentials` 字符串映射。它只能由批准的 secret injector 在内存中生成并通过 stdin 输入；不得写入 manifest、文件、argv、环境变量、shell 历史、日志或版本控制。每个账号必须有独立凭据，并通过当前口令策略与已泄露默认口令检查。

`PROVISIONING_MANIFEST_PATH`、`DATABASE_PATH` 必须是已存在文件的绝对路径；`PROVISIONING_MANIFEST_SHA256` 必须是 operator 对批准 manifest 计算并复核的 64 位 SHA-256。CLI 不提供 `--dry-run`，也不接受包括 `--help` 在内的额外 argv；任何额外参数都会 fail-closed。

### RED 前的只读验证

在没有真实 DB 授权时，只允许：

```bash
cd 后端代码/server

# 静态语法/类型和临时 mock SQLite 自测；测试自行创建并清理临时库。
npm run build
npm run test:node -- tests/approved-account-provisioning.test.ts
```

已构建镜像内还应对 `provisioning-dist/scripts/reset-passwords.js` 执行 `node --check`。这些检查不连接真实数据库，不是正式 provisioning 的 dry-run，也不证明 HTTP 登录或旧 session 已失效。

### 正式执行入口

执行前必须停止所有会写目标 SQLite 的服务，完成下节备份、验证和隔离恢复演练，复核批准 manifest/散列/账号与角色，并由 secret injector 独占连接进程 stdin。以下命令不含任何真实路径或凭据，当前仓库任务没有执行它：

```bash
cd 后端代码/server

# fd 0 必须已由批准的 secret injector 管道连接；不得在终端粘贴或落盘凭据正文。
PROVISIONING_MANIFEST_PATH='<absolute-approved-manifest.json>' \
PROVISIONING_MANIFEST_SHA256='<verified-64-hex-sha256>' \
DATABASE_PATH='<absolute-existing-offline-coreone.db>' \
npm run reset-passwords
```

源码工作区入口使用现有 `tsx` 开发依赖；正式 backend image 保持 production-only，并由只接受固定 `scripts/reset-passwords.ts` 目标的 runtime shim 把同一 `npm run reset-passwords` 转交 builder 产出的编译后 JS。shim 不接受其他脚本目标，额外 argv 仍由 CLI fail-closed；两者执行同一个批准实现。

### 幂等、失败和审计边界

- 全部账号在一个 SQLite 事务中创建或对齐；stdout 每账号只输出 `apply=created|updated|unchanged`、凭据状态和默认凭据状态，不输出秘密、散列或数据库路径。
- 同一 manifest 和同一凭据重复执行应全部为 `unchanged` 且不写库。账号数量不固定为历史清单；唯一权威是当前批准 manifest。
- 事务提交前的校验、schema、凭据、角色或写入失败会整体回滚并以退出码 1 失败，不允许半量提交。
- SQLite 锁或繁忙导致 `PROVISIONING_CONFLICT` 时保证零写入；operator 必须先重新确认服务仍停止、数据库状态和 manifest 未漂移，才可显式重试。
- 若并发写入已改变被批准状态，`PROVISIONING_CONCURRENT_STATE_CONFLICT` 会拒绝覆盖；不得自动重试或把冲突当成功。
- 如果事务已提交但 stdout 证据写入失败，CLI 返回退出码 2 和 `COMMITTED_EVIDENCE_WRITE_FAILED`。此时不得盲目重跑；先按 manifest SHA、数据库审计与逐账号状态确认提交结果。
- 成功证据含 `provisioning=committed`、账号数、`manifest-sha256` 和每账号非秘密状态。operator 还必须在受控外部验收中逐账号验证新登录、已知默认凭据拒绝、旧凭据拒绝、旧 session/token 失效和权限边界；CLI 输出本身不证明这些 HTTP/runtime 结果。

### 回滚与失败恢复

事务提交前由 SQLite 原子回滚。提交后不得恢复旧口令、旧 secret 或已知风险镜像；代码/镜像可以按提交逆序回退，但数据库中的凭据状态不会随 Git 回退。若发生另行授权的整库灾难恢复，目标保持离线，并在恢复后重新执行当前批准 manifest、完成登录/session/权限复核，再允许接流量。

## SQLite 备份、校验与隔离恢复

所有路径必须是仓库外绝对路径。备份使用 SQLite online backup API，随后执行 `quick_check`、`integrity_check`、`users` schema 检查和 SHA-256 manifest；恢复只写原本不存在的隔离目录。

```bash
node --experimental-sqlite scripts/release/backup-sqlite.mjs \
  --database '<absolute-source-db>' \
  --output-dir '<absolute-private-backup-dir>' \
  --release '<approved-40-character-commit-sha>' --json

node --experimental-sqlite scripts/release/verify-backup.mjs \
  --backup '<absolute-backup-db>' \
  --manifest '<absolute-manifest-json>' \
  --release '<approved-40-character-commit-sha>' --json

node --experimental-sqlite scripts/release/restore-drill.mjs \
  --backup '<absolute-backup-db>' \
  --manifest '<absolute-manifest-json>' \
  --target-dir '<new-absolute-drill-dir>' \
  --release '<approved-40-character-commit-sha>' --json
```

篡改、manifest 漂移、SQLite 损坏、错误 release、相对路径、符号链接或已存在恢复目录全部 fail-closed。checksum 证明仓库内完整性合同，不替代目标环境的离机加密、保留期和 operator 身份控制。

## 升级与前滚

```bash
node --experimental-sqlite scripts/release/plan-forward.mjs \
  --current-release '<current-40-character-commit-sha>' \
  --target-release '<target-40-character-commit-sha>' \
  --backup '<absolute-backup-db>' \
  --manifest '<absolute-manifest-json>' \
  --data-volume '<operator-approved-external-volume>' \
  --output '<new-absolute-forward-plan-json>' --json
```

前滚入口只生成 `executionAuthorized=false` 的不可覆盖计划；它不停止服务、不改库、不构建镜像、不切流，也拒绝执行模式。数据一旦变更，失败策略是保持离线并用获批安全版本前滚。

## 当前运行架构

```text
operator-approved host
├── frontend image ID (Nginx, non-root, host port)
│   └── overwrites Forwarded/X-Forwarded-For → one trusted hop
├── internal app network (COREONE_INTERNAL_SUBNET)
│   └── backend image ID (Node 22.23.1, non-root, no host port)
└── external SQLite volume (operator-approved identity)
```

backend health 同时检查 HTTP `/api/health` 和只读 SQLite `users` schema。容器、网络、volume、secret store、证书与生产流量均由目标环境 operator 管理，不能由本地静态证据推定已存在或已验收。

## 结果解释

- PASS：对应命令在当前明确环境中完成，并有可复核输出。
- FAIL：合同或实现不满足，必须修复后重新取得 RED→GREEN 证据。
- BLOCKED：环境或权限缺失，例如真实 launcher 需要尚未具备的运行条件。
- NO_GO：不能宣称发布或上线；Docker daemon/真实镜像/容器/网络/volume、Node 22 实机、真实 Playwright、Claude 异构复核或生产授权缺一时都保持相应边界。
