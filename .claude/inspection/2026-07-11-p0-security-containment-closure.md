## #119 依赖与风险

PR #119 **不是当次**生产 JWT 轮换或现有八账号改密的硬依赖：当前 `master` 支持外部 `JWT_SECRET`，也有用户口令更新路径。

PR #119 是**持久止血**依赖：它让生产 JWT 配置缺失时拒绝启动、禁止历史/弱密钥、停止生产环境种默认账号，并提供八账号原子改密工具。

部署顺序存在明确停机风险：若先部署 #119、后改生产八账号，新的启动门禁可能因检测到历史默认凭据而拒绝启动。因此在获得部署批准后，也必须遵循“先生产改密，再部署 #119”的顺序。

## 最小用户接管步骤

当前无需用户提供尚不存在的生产信息。生产规划启动后，按上线前门禁手册执行：

1. 用占位符登记生产平台、环境、URL、数据库、实例、负责人和审批信息；文档只保存 reference，不保存秘密。
2. 在 secret store 生成 JWT 与八账号口令材料，并完成八账号安全初始化或原子改密。
3. 一次性部署全部实例，验证旧 token、默认口令、新登录及重启后账号状态。
4. 单独取得 #119 或等效安全变更的部署批准；本任务不会把该批准视为已经取得。

## 计划验收与 forward-only 回滚

- Kimi 项当前按事故负责人确认关闭；若未来重新订阅、恢复相同凭据或创建替代凭据，必须重新打开厂商凭据门禁并取得新的失效/启用证据。
- 生产改密部分失败：不恢复历史口令；使用新一组互异强口令完成前滚，并记录八账号各自结果。
- JWT rollout 失败：不恢复已泄露密钥；修复部署配置后继续使用新的强密钥完成全实例前滚。
- 旧令牌验证：`GET /api/v1/auth/me` 与 `POST /api/v1/auth/refresh` 都应返回 401，并需覆盖每个实例。
- 账号验证：八账号新口令登录应成功，历史默认口令登录应返回 401；客户端只记录用户名、实例和 HTTP 状态。
