# 会话B Handoff 机制

> **目的**: 避免长对话导致的上下文漂移和回答质量下降  
> **参考**: 会话A（Roo）的批次化工作模式  

---

## 何时触发 Handoff

以下情况出现时，当前会话应主动建议开启新对话：

| # | 触发条件 | 当前状态 | 建议动作 |
|:-:|:---|:---:|:---|
| 1 | **已完成一个完整批次** | ✅ 批次2完成 | **立即 handoff** |
| 2 | **修改文件超过 15 个** | 当前累计 ~12 个 | 下批次开始时 handoff |
| 3 | **对话轮次超过 40 轮** | 当前 ~35 轮 | 接近阈值，准备 handoff |
| 4 | **涉及完全不同的功能域** | 当前：前端修复 | 若切换至"后端API开发"或"E2E测试"则 handoff |
| 5 | **出现上下文遗忘迹象** | 暂无 | 若我开始重复提问或遗漏已确认的需求，立即 handoff |
| 6 | **计划工期超过 2 天** | 当前批次半天 | 大任务拆分为多批次，每批次一个对话 |

## Handoff 流程

### 1. 当前会话结束前（由我自己执行）

```
□ 更新 SESSION-B-WORKLOG.md（记录本批次所有变更）
□ 更新 AGENTS.md 中的状态快照
□ 运行 TypeScript 编译确认无错误
□ 创建 .claude/handoffs/ 目录下的 handoff 摘要文件
□ 向用户报告：本批次完成 + 下批次建议 + handoff 建议
```

### 2. Handoff 摘要文件模板

文件路径: `.claude/handoffs/YYYY-MM-DD-批次名.md`

内容结构：
```markdown
# Handoff: [批次名称]（[日期]）

## 已完成
- [ ] 具体修改1
- [ ] 具体修改2

## 代码状态
- 最后修改文件：`文件路径:行号`
- TypeScript编译：通过/失败
- E2E影响：无/需关注

## 待办交接
- [ ] 下一批次要做什么
- [ ] 已知阻塞/风险
- [ ] 会话A需同步的事项

## 关键上下文（防止遗忘）
- 重要决策：为什么这样改
- 技术约束：不能做什么
- 用户信息：用户的偏好
```

### 3. 新会话启动后（由新对话读取）

```
□ 读取 AGENTS.md 了解分工边界
□ 读取 SESSION-B-WORKLOG.md 了解历史变更
□ 读取最新的 handoff 摘要文件
□ 读取 待办清单.md 了解剩余任务
□ 向用户确认：是否继续上批次的计划，还是调整优先级
```

## 为什么现在建议 Handoff

1. **批次边界清晰**：批次1和批次2已完成，批次3（跨页面改造）是全新的工作域
2. **对话长度适中**：~35轮，质量尚未明显下降，但继续会进入风险区
3. **上下文切换大**：从"单页面修复"切换到"17个页面批量改造"，需要全新的思维框架
4. **会话A可能也在推进**：新对话可以重新同步双方的最新状态
